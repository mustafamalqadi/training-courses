00 · مُقَدِّمة الدَّورة | Course Introduction

كَـ vSphere Administrator، تَحتاج إِلى فَهم عَميق لِتَقنيّات التَّخزين لِإِدارة بيئة VCF بِكَفاءة. هَذِه الدَّورة تُغَطّي كُلّ شَيء مِن vSAN إِلى VMFS إِلى NFS إِلى NVMe-oF.

أَهلاً بِكُم في VCF Storage Fundamentals

هَذِه الدَّورة تُزَوِّدك بِالمَعرِفة الأَساسيّة لِإِدارة Datastores في بيئة VMware Cloud Foundation. سَتَفهَم تَقنيّات التَّخزين المُختَلِفة وَبِنية vSAN وَسياسات الحِماية وَالمُرونة.

🧠 أَهداف الدَّورة | Learner Objectives

  • تَحديد أَنواع Datastores المُختَلِفة في بيئات VCF وَمَتى تَختار كُلّاً مِنها.
  • وَصف ميزات vSAN وَأَنواع الـ Clusters (Standard، Stretched، 2-Node).
  • التَّمييز بَين vSAN OSA وَ vSAN ESA — اِنعِكاسات الأَداء وَالمِساحة.
  • تَصميم Storage Policies (SPBM) لِـ workloads مُختَلِفة (Mission-Critical، Production، Dev/Test).
  • مُقارَنة أَنواع الـ Objects في vSAN ESA وَفَهم Performance/Capacity Leg.
  • تَشخيص حالات صِحّة Components وَObjects، وَتَفسير سُلوك Resync.
  • تَنفيذ Runbooks عَمَليّة: host onboarding، disk replacement، patching، capacity recovery.
  • تَجَنُّب Anti-Patterns الشّائِعة (snapshot hoarding، MTU mismatch، force-provisioning abuse).
  • تَطبيق المَعرِفة في 5 سيناريوهات حَقيقيّة: Greenfield، Migration، OSA→ESA، Capacity Crunch، DR Drill.

لِماذا هَذا القِسم؟

قَبل أَن تَغوص في vSAN وَVMFS وَNFS، تَحتاج أَن تَفهَم لِماذا يَتَصَرَّف التَّخزين بِهَذِه الطَّريقة. هَذا القِسم يُؤَسِّس الفَهم المِعماري الَّذي يَفصِل المُهَندِس المُحتَرِف عَن مُجَرَّد مُنَفِّذ الأَوامِر.

0.5.1 خَريطة الأَدوار — مَن يَفعَل ماذا؟ | Role Map

Storage Admin — يُدير الـ Datastores وَالسِّياسات وَالأَداء. يَتَعامَل يَوميّاً مَع vSAN Health وَCapacity Planning.

0.5.2 لِماذا وُجِد التَّخزين المُوَزَّع؟ | Why Distributed Storage Exists

F1 مِن SAN المَركَزي إِلى vSAN المُوَزَّع — التَّحَوُّل
🎯 المَفهوم الأَساسي

التَّخزين المَركَزي (FC SAN) كانَ المِعيار لِعُقود: أَقراص بَعيدة، شَبَكة مُخَصَّصة، وَإِدارة مُنفَصِلة. vSAN قَلَبَ هَذا النَّموذج بِجَعل كُلّ host يُساهِم بِأَقراصِه المَحَلّيّة في pool مُشتَرَك.

🏗 العَوامِل الدافِعة
  • التَّكلُفة: FC SAN يَحتاج HBAs + Switches + Array licenses. vSAN يَستَخدِم أَقراص مَحَلّيّة فَقَط.
  • التَّعقيد: Zoning + LUN Masking + Multipathing مُقابِل سِياسة واحِدة في vSAN.
  • المُرونة: إِضافة سِعة = إِضافة host جَديد. لا حاجة لِشِراء shelf جَديد.
  • الأَداء: NVMe مَحَلّي أَسرَع مِن FC 32Gbps في مُعظَم السِّيناريوهات.
⚙ مَسار التَّطَوُّر
  1. 2005–2012: FC SAN هُوَ الخِيار الوَحيد لِلـ Enterprise. كُلّ VM تَعتَمِد عَلى LUN مَركَزي.
  2. 2013: vSAN 1.0 — أَوَّل HCI مُدمَج في ESX. بَسيط لَكِن مَحدود.
  3. 2016–2020: vSAN OSA يَنضُج — Disk Groups، Dedup، Encryption. لَكِن Cache Tier يُعَقِّد التَّصميم.
  4. 2022+: vSAN ESA — تَخطيط جَديد بِالكامِل. Storage Pool واحِد، لا Cache/Capacity tiers، NVMe-only.
  5. VCF 9.0: vSAN ESA هُوَ الخِيار الاِفتِراضي وَالمُوصى بِه. OSA لِلتَّوافُق فَقَط.
⚠ أَخطاء شائِعة في الفَهم
  • «vSAN بَديل كامِل لِلـ SAN» — لَيسَ دائِماً. بَعض Workloads (Oracle RAC, mainframe) ما زالَت تَحتاج FC.
  • «NVMe = سَريع دائِماً» — الشَّبَكة هِيَ الـ bottleneck غالِباً. 25GbE minimum لِـ ESA.

0.5.3 أَنماط التَّخزين الثَّلاثة | The Three Storage Patterns

✅ Block Storage (FC/iSCSI/NVMe-oF)
  • أَعلى أَداء خامّ (Raw IOPS)
  • مِثالي لِـ Databases وَ Transactional workloads
  • يَحتاج بُنية تَحتيّة مُنفَصِلة
  • في VCF: Supplemental فَقَط (لَيسَ Principal)
✅ File Storage (NFS)
  • بَسيط — mount وَاِنتَهى
  • مِثالي لِـ Templates، ISOs، Shared content
  • لا يَدعَم كُلّ الميزات (no SIOC مَع v4.1)
  • في VCF: Supplemental فَقَط

Object Storage (vSAN) — النَّمَط الثالِث وَالأَهَمّ في VCF. كُلّ VM = مَجموعة Objects. كُلّ Object لَه سِياسة حِماية مُستَقِلّة. يَتَوَزَّع تِلقائيّاً عَلى الـ Cluster. هَذا هُوَ Principal Storage الاِفتِراضي.

0.5.4 قَواعِد القَرار — مَتى تَختار ماذا؟ | Decision Framework

Framework شَجَرة قَرار اختِيار نَوع التَّخزين في VCF
  1. هَل هُوَ Management Domain أَو VI WLD جَديد؟ → vSAN ESA (إِلزامي لِلـ Management، مُوصى لِلـ VI).
  2. هَل الـ Workload يَحتاج FC مُباشِر؟ (Oracle RAC, Mainframe) → أَضِف VMFS FC كَـ Supplemental.
  3. هَل تَحتاج مُشارَكة مَلَفّات بَسيطة؟ (ISOs, Templates) → أَضِف NFS v3 كَـ Supplemental.
  4. هَل تَحتاج أَكثَر مِن 400K IOPS لِـ single VM؟ → NVMe-oF Supplemental أَو vSAN ESA مَع hosts كافية.
  5. هَل لَدَيك 3+ hosts فَقَط؟ → vSAN Standard Cluster. لِ 2 hosts → vSAN Two-Node + Witness.
❌ لا تَفعَل هَذا

«أُريد NFS لِأَنَّه أَرخَص مِن vSAN» — في VCF، vSAN هُوَ الـ Principal وَلا يُمكِن إِزالَته. NFS يُضاف بِالإِضافة لا بَدَلاً مِنه.

0.5.5 المَفاهيم المِعماريّة الأَساسيّة | Core Architectural Concepts

F2 Data Locality vs Data Distribution — أَين تَعيش البَيانات؟
🎯 المَفهوم

في FC SAN، البَيانات دائِماً بَعيدة — تَعبُر الشَّبَكة لِكُلّ I/O. في vSAN، البَيانات مُوَزَّعة — بَعضُها مَحَلّي وَبَعضُها عَلى hosts أُخرى لِلحِماية. الفَهم الصَّحيح: vSAN لَيسَ «تَخزين مَحَلّي» بَل «تَخزين مُوَزَّع ذَكي».

📤 الفوائِد العَمَليّة
  • Read Locality القِراءة مِن النُّسخة الأَقرَب (Performance Leg مَحَلّي إِن أَمكَن).
  • Auto-Rebalance البَيانات تَتَوَزَّع تِلقائيّاً عِند إِضافة/إِزالة hosts.
  • Policy-Driven مُستَوى الحِماية يُحَدَّد per-VM لا per-LUN.
💡 مَثَل واقِعي

VM تَعمَل عَلى Host-A. في RAID-5 (4+1)، بَياناتها مُوَزَّعة عَلى 5 hosts. عِند القِراءة، vSAN يَقرَأ مِن أَقرَب stripe. عِند الكِتابة، تُكتَب أَوَّلاً في Performance Leg (N+1 mirroring) ثُمَّ تُنقَل لِلـ Capacity Leg كَ full stripe. هَذا يُفَسِّر لِماذا ESA أَسرَع في الكِتابة مِن OSA.

F3 Thin Provisioning وَ Over-Commitment — الحُدود الآمِنة
🎯 لِماذا مُهِمّ

كُلّ vSAN ESA Datastore يَستَخدِم Thin Provisioning افتِراضيّاً — لا يُحجَز المِساحة حَتّى تُكتَب البَيانات فِعليّاً. لَكِن هَذا يَعني أَنَّ مَجموع الـ Provisioned Space يُمكِن أَن يَتَجاوَز الـ Physical Capacity.

⚠ الخَطَر الحَقيقي
  • 80% Physical Usage — vSAN يَبدَأ Auto-Rebalance. الأَداء يَنخَفِض.
  • 92% Usage — vSAN يَرفُض كِتابات جَديدة. VMs تَتَجَمَّد.
  • لا تَنتَظِر التَّنبيه! — راقِب Skyline Health يَوميّاً. ضَع alert عِند 70%.
💡 حِساب سَريع

Cluster مِن 6 hosts × 4 NVMe × 3.2TB = 76.8TB Raw. مَع RAID-5 (4+1) overhead = ~61TB Usable. مَع 70% safe threshold = ~43TB Effective. لا تُوَعِّد أَكثَر مِن هَذا.

01 · نَظرة عامّة عَلى التَّخزين | VCF Storage Overview

1.1 أَهَمّيّة فَهم التَّخزين | Why Storage Matters

فَهم تَقنيّات التَّخزين (vSAN، VMFS، NFS، NVMe) ضَروري لِإِدارة بيئات VCF بِكَفاءة. التَّخزين يُؤَثِّر مُباشَرة عَلى الأَداء وَالتَّوافُر وَقابِليّة التَّوَسُّع.

لِماذا فَهم التَّخزين أَوَّلاً؟

كُلّ VM تَحتاج مَكاناً تُخَزِّن فيه مَلَفّاتها. فَهم أَنواع Datastores وَطُرُق الوُصول إِلَيها هُوَ الأَساس لِأَيّ عَمَليّة في VCF. هَذِه الوِحدة تُؤَسِّس لِكُلّ ما يَأتي.

🧠 Principal vs Supplemental Storage

  • Principal Storage — التَّخزين الرَّئيسي لِلـ Domain. لا يُمكِن إِزالَته بَعد النَّشر.
  • Supplemental Storage — تَخزين اختِياري. يُمكِن إِضافَته وَإِزالَته في أَيّ وَقت.
  • الحَدّ الأَدنى: 2 hosts لِتَخزين خارِجي، 3 hosts لِـ vSAN Standard Cluster.

🧠 About Datastores — ما هُوَ الـ Datastore؟

  • Datastore = وِحدة تَخزين مَنطِقيّة تَستَخدِم مِساحة مِن أَجهِزة فِعليّة.
  • يَحتَوي على: VMs، Templates، ISO Images.
  • كُلّ VM تُخَزَّن كَمَلَفّات في مُجَلَّدها الخاصّ أَو كَـ Objects.
  • VCF يَدعَم: vSAN، VMFS، NFS.

🧠 About vSAN — نَظرة عامّة

  • vSAN يَستَخدِم أَجهِزة تَخزين مَحَلّيّة (NVMe SSDs, SSDs, HDDs) لِتَكوين Datastore واحِد عَبر عِدّة ESX Hosts.
  • يَدعَم سياسات RAID-1/5/6، حَتّى 3 أَعطال مَسموحة.
  • تَشفير البَيانات أَثناء السُّكون وَأَثناء النَّقل.
  • ضَغط وَإِزالة التَّكرار.

1.3 Fibre Channel وَ iSCSI

🧠 Fibre Channel (FC) Overview

  • FC يُرسِل أَوامِر SCSI عَبر شَبَكة Fibre Channel إِلى تَخزين مَركَزي.
  • HBAs (Host Bus Adapters) تَربُط ESX بِشَبَكة FC.
  • يَدعَم ESX سُرعات حَتّى 128 Gbps وَ FCoE.
  • LUNs مِن الـ Storage Array تُصبِح مُتاحة وَتُهَيَّأ كَـ VMFS Datastores.

🧠 iSCSI Overview

  • iSCSI يُرسِل أَوامِر SCSI عَبر شَبَكة Ethernet.
  • يَستَخدِم NICs عاديّة أَو Hardware iSCSI HBAs.
  • في VCF: يُستَخدَم كَـ Supplemental Storage فَقَط.

1.4 VMFS وَ NFS

🧠 VMFS — Virtual Machine File System

  • نِظام مَلَفّات مُتَخَصِّص لِلتَّخزين القائِم عَلى Block (FC أَو iSCSI).
  • Concurrent Access — عِدّة ESX Hosts تَقرَأ وَتَكتُب في نَفس الوَقت.
  • On-Disk Locking — يَمنَع تَشغيل نَفس الـ VM عَلى أَكثَر مِن Host.
  • الحُدود:
    • حَتّى 128 host لِكُلّ Datastore.
    • حَتّى 1,024 LUN لِكُلّ ESX Host.
    • حَجم أَقصى: 64 TB لِلـ Datastore.
    • يَدعَم أَجهِزة 4K Native.
  • Space Reclamation تِلقائي (UNMAP).

🧠 NFS — Network File System

  • بُروتوكول مُشارَكة مَلَفّات عَبر Ethernet لِلوُصول إِلى NAS.
  • يَدعَم NFS v3 وَ NFS v4.1.
  • NFS v3: قَفل بِواسِطة VMware (لَيس NLM). NFS v4.1: قَفل مِن الـ Server.
  • تَحذير: لا تَخلِط v3 وَ v4.1 عَلى نَفس الـ Datastore — خَطَر فَساد البَيانات.
  • أَفضَل مُمارَسة: عَزل traffic الـ NFS في شَبَكة مُنفَصِلة.

1.5 NVMe over Fabrics

🧠 NVMe over Fabrics (NVMe-oF)

  • يُرسِل أَوامِر NVMe عَبر الشَّبَكة. أَسرَع مِن iSCSI وَ FC التَّقليدي.
  • أَقَلّ Latency، أَعلى Throughput، طَلَبات مُتَوازية، حِمل CPU أَقَلّ.
  • الأَنواع المَدعومة:
    • NVMe/TCP — عَبر TCP/IP.
    • NVMe/FC — عَبر Fibre Channel.
    • NVMe/RDMA — نَقل مُباشِر لِلـ memory بِدون TCP.
  • يُستَخدَم كَـ Supplemental Storage في VCF.

🧠 Datastore Contents — المُحتَوى

  • Block-backed (SAN — iSCSI/FC): بَيانات مُخَزَّنة كَـ blocks.
  • File-backed (NAS — NFS): بَيانات في تَسَلسُل هَرَمي (folders/files).
  • Object-based (vSAN): كُلّ VM = مَجموعة Objects (VM config + VMDKs + swap). كُلّ Object لَه بَيانات وَ metadata وَ ID فَريد.

DC1 Datastore Internals — ماذا يَحدُث داخِل الـ Datastore فِعليّاً
🎯 المَفهوم الأَساسي

Datastore هُوَ abstraction layer — يُخفي تَفاصيل التَّخزين الفِعلي (LUNs, NFS exports, vSAN objects) وَيُقَدِّم واجِهة مُوَحَّدة لِلـ VMs. لَكِن سُلوكه يَختَلِف جَذريّاً حَسَب النَّوع.

🏗 الفُروق الجَوهَريّة
  • VMFS Datastore: Block device → File system. كُلّ VM = مُجَلَّد يَحتَوي .vmx + .vmdk + .nvram + .log files.
  • NFS Datastore: Remote mount. نَفس بِنية المَلَفّات لَكِن عَبر الشَّبَكة. Locking عَلى مُستَوى Array.
  • vSAN Datastore: Object store. كُلّ VM = مَجموعة Objects (VM Home + VMDKs + Swap). لا مَلَفّات تَقليديّة — Objects مَع metadata.
⚙ ماذا يَحدُث عِند Power-On
  1. VMFS: ESX يَفتَح .vmx → يَحجِز .vswp → يَفتَح .vmdk files → I/O يَمُرّ عَبر SCSI/NVMe stack.
  2. NFS: ESX يَفتَح المَلَفّات عَبر NFS client → locking عَلى Array → I/O عَبر TCP/IP.
  3. vSAN: ESX يَقرَأ Object metadata → يَعرِف أَين كُلّ component → I/O مُباشِر لِلـ owner host أَو locally.
⚠ أَخطاء شائِعة
  • «كُلّ Datastores مُتَشابِهة» — لا! vSAN datastore لا يَظهَر في ls /vmfs/volumes/ بِنَفس الطَّريقة. لا تَبحَث عَن .vmdk files عَلى vSAN.
  • «VMFS 6 = VMFS 5» — VMFS 6 يَدعَم 4K native وَ automatic UNMAP. الفُروق مُهِمّة لِلأَداء.
DC2 Multipathing وَ Path Selection — كَيف يَصِل الـ I/O
🎯 المَفهوم

لِلتَّخزين الخارِجي (FC, iSCSI, NVMe-oF)، كُلّ host لَديه مَسارات مُتَعَدِّدة لِلوُصول لِنَفس الـ LUN. PSP (Path Selection Plugin) يُقَرِّر أَيّ مَسار يُستَخدَم لِكُلّ I/O.

🏗 المُكَوِّنات
  • NMP (Native Multipathing Plugin) — الإِطار الرَّئيسي في ESX.
  • SATP (Storage Array Type Plugin) — يَتَعَرَّف عَلى نَوع الـ Array وَيُحَدِّد السُّلوك.
  • PSP (Path Selection Plugin) — Round Robin (افتِراضي) | Fixed | MRU.
  • HPP (High Performance Plugin) — لِلـ NVMe-oF. يَستَبدِل NMP بِالكامِل.
📤 التَّأثير عَلى الأَداء
  • Round Robin يُوَزِّع I/O عَلى كُلّ المَسارات — أَفضَل throughput.
  • Fixed مَسار واحِد فَقَط — لِلـ arrays القَديمة الَّتي لا تَدعَم ALUA.
  • HPP أَقَلّ latency — مُصَمَّم لِ NVMe queues المُتَعَدِّدة.
💡 مَثَل عَمَلي

Host لَديه 2× FC HBAs، كُلّ HBA مُتَّصِل بِ 2 FC switches → 4 paths لِكُلّ LUN. مَع Round Robin وَ IOPS=1، كُلّ I/O يَذهَب عَبر path مُختَلِف → 4× throughput مُقارَنة بِ Fixed.

DC3 UNMAP وَ Space Reclamation — اِستِرجاع المِساحة
🎯 لِماذا مُهِمّ

عِندَما تَحذِف مَلَفّات داخِل VM، المِساحة لا تُعاد لِلـ Datastore تِلقائيّاً — هَذا «dead space». UNMAP يُخبِر الـ Array/vSAN أَنَّ هَذِه الـ blocks لَم تَعُد مَطلوبة.

⚙ كَيف يَعمَل
  1. Guest OS يَحذِف مَلَفّ → يُرسِل TRIM/UNMAP لِلـ virtual disk.
  2. ESX يَستَلِم الأَمر → يُتَرجِمه لِـ SCSI UNMAP لِلـ Datastore.
  3. VMFS 6: Automatic UNMAP (background priority 50%). لا حاجة لِأَوامِر يَدَويّة.
  4. vSAN: UNMAP/TRIM يَعمَل تِلقائيّاً — مُدمَج في الـ object management.
  5. NFS: لا يَدعَم UNMAP. المِساحة تَبقى مَحجوزة حَتّى يُحذَف الـ .vmdk file.
⚠ أَخطاء شائِعة
  • «حَذَفت VMs لَكِن المِساحة لَم تَتَحَرَّر» — تَأَكَّد أَنَّ الـ disk هُوَ Thin وَلَيسَ Thick Eager Zero.
  • «UNMAP يُؤَثِّر عَلى الأَداء» — في VMFS 6، الـ priority مُنخَفِض (50/1000). لَن تُلاحِظ تَأثيراً.

vSAN هُوَ مَنصّة التَّخزين الأَساسيّة في VCF. بَدَلاً مِن شِراء SAN مُنفَصِل، يُجَمِّع الأَقراص المَحَلّيّة في كُلّ ESX Host وَيُنشِئ Datastore واحِد مُوَزَّع عَلى الـ Cluster بِالكامِل.

vSAN هُوَ قَلب التَّخزين في VCF

vSAN هُوَ مَنصّة التَّخزين الأَساسيّة في VCF. يُدمَج في ESX Hypervisor مُباشَرة، يُجَمِّع الأَقراص المَحَلّيّة في Datastore واحِد مُشتَرَك، وَيُطَبِّق سياسات حِماية لِكُلّ VM بِشَكل مُستَقِلّ.

🧠 vSAN Storage Overview — النَّظرة الشامِلة

  • يُطَبِّق سياسات التَّخزين عَلى مُستَوى VM / VMDK.
  • يُوَفِّر HA مِن خِلال تَحَمُّل الأَعطال المَدمَج.
  • يُدار عَبر vSphere Client وَ SDDC Manager — لا أَدَوات خارِجيّة.
  • تَكلِفة أَقَلّ بِإِلغاء الحاجة لِـ SAN hardware مُنفَصِل.

🧠 Object vs Block vs File Storage

  • vSAN = Object-based — كُلّ VMDK هُوَ Object مَع سياسة تَخزين خاصّة.
  • VMFS = Block-based — بَيانات كَـ blocks عَلى LUNs.
  • NFS = File-based — مَلَفّات في تَسَلسُل هَرَمي.

تَشبيه: vSAN Datastore = خِزانة مَلَفّات. Objects = مُجَلَّدات. سياسة التَّخزين تُحَدِّد كَيف تُخَزَّن البَيانات. RAID-1 = نُسخَتان. RAID-5 (3+1) = 3 شَرائِح بَيانات + 1 Parity.

2.3 خَصائِص vSAN Datastore

🧠 vSAN Datastore — الخَصائِص

  • Datastore واحِد مُشتَرَك لِكُلّ الـ Cluster.
  • قابِل لِلتَّوَسُّع بِإِضافة أَجهِزة أَو hosts.
  • سياسات التَّخزين تُحَدِّد مَوقِع البَيانات وَمُستَوى الحِماية.
  • يَدعَم File Services (SMB/NFS/iSCSI).
  • سياسات افتِراضيّة وَمُخَصَّصة (RAID-0/1/5/6).

🧠 vSAN Features — المَيِّزات

  • Data-at-Rest Encryption — تَشفير البَيانات عَلى الأَقراص.
  • Data-in-Transit Encryption — تَشفير البَيانات أَثناء النَّقل بَين Hosts.
  • File Shares — مُشارَكة مَلَفّات (SMB/NFS) لِلـ VMs وَالأَنظِمة الخارِجيّة.
  • iSCSI Target — تَخزين لِلـ VMs وَالأَنظِمة الفِعليّة.
  • Dedup & Compression — مَدعوم في OSA وَ ESA. الضَّغط مُفَعَّل افتِراضيّاً في ESA.

DC4 vSAN Networking — الشَّبَكة هِيَ العَمود الفَقَري
🎯 المَفهوم الأَساسي

vSAN يَعتَمِد كُلّيّاً عَلى الشَّبَكة — كُلّ I/O يَمُرّ عَبر vSAN VMkernel port. إِذا كانَت الشَّبَكة بَطيئة أَو غَير مُستَقِرّة، التَّخزين بِالكامِل يَتَأَثَّر.

🏗 المُتَطَلَّبات
  • OSA: 10GbE minimum (مُوصى 25GbE). Dedicated أَو shared مَع vMotion.
  • ESA: 25GbE minimum (مُوصى 100GbE). Dedicated لِـ vSAN traffic.
  • MTU: 9000 (Jumbo Frames) لِأَفضَل أَداء — يَجِب أَن يَكون end-to-end.
  • Multicast → Unicast: في vSAN 7.0 U1+، لا حاجة لِ multicast. Unicast فَقَط.
  • RDMA: اختِياري لِـ ESA — يُقَلِّل latency بِشَكل كَبير (RoCE v2).
⚠ أَخطاء قاتِلة
  • «10GbE كافٍ لِ ESA» — لا! ESA يَكتُب Performance Leg mirrors عَلى الشَّبَكة. 10GbE = bottleneck دائِم.
  • «Jumbo Frames عَلى vSAN فَقَط» — يَجِب تَفعيلها عَلى: VMkernel + vSwitch + Physical Switch + الطَّرَف الآخَر.
  • «شَبَكة واحِدة لِكُلّ شَيء» — لا تَخلِط vSAN traffic مَع VM traffic. استَخدِم NIOC أَو Physical separation.
💡 مَثَل واقِعي

Cluster مِن 8 hosts ESA. كُلّ host لَديه 2× 25GbE NICs. NIC1 = vSAN + vMotion (NIOC: vSAN 80%, vMotion 20%). NIC2 = VM traffic. Jumbo Frames مُفَعَّلة end-to-end. النَّتيجة: ~18Gbps effective لِ vSAN per host.

DC5 Witness وَ Quorum — كَيف يَتَّخِذ vSAN القَرارات
🎯 المَفهوم

كُلّ vSAN Object لَديه Witness — component صَغير (metadata فَقَط) يُصَوِّت في حالة الاِنقِسام. الأَغلَبيّة (Quorum) = أَكثَر مِن 50% مِن الأَصوات يَجِب أَن تَكون مُتاحة.

⚙ كَيف يَعمَل Quorum
  1. RAID-1 FTT=1: Object = 2 data replicas + 1 witness. كُلّ مُكَوِّن = 1 صَوت. الأَغلَبيّة = 2 مِن 3.
  2. RAID-5 (4+1): Object = 5 components. كُلّ component = 1 صَوت. الأَغلَبيّة = 3 مِن 5.
  3. Network Partition: الجانِب الَّذي يَملِك الأَغلَبيّة يَبقى Active. الآخَر = Inaccessible.
  4. Two-Node Cluster: Witness VM عَلى host ثالِث (أَو في vSphere). بِدون Witness = لا quorum عِند فَشَل host.
⚠ أَخطاء شائِعة
  • «Witness يُخَزِّن بَيانات» — لا! Witness = metadata + vote فَقَط. حَجمُه صَغير جِدّاً.
  • «3 hosts + RAID-5 = آمِن» — RAID-5 (2+1) عَلى 3 hosts: فَقَدان host واحِد = لا quorum لِبَعض objects!
DC6 OSA vs ESA — ما الَّذي تَغَيَّر حَقّاً وَلِماذا
🎯 المَفهوم

ESA لَيسَ مُجَرَّد «OSA بِدون Cache Tier». هُوَ إِعادة كِتابة كامِلة لِلـ I/O path. الفُروق جَوهَريّة في كَيف يَتَعامَل مَع الكِتابة وَالقِراءة.

🏗 الفُروق الهَندَسيّة
  • OSA — Log-Structured: كُلّ كِتابة → Cache SSD (log) → destage إِلى Capacity. Cache = bottleneck وَ single point of wear.
  • ESA — Direct Write: كُلّ كِتابة → Performance Leg (N+1 mirrors) → Capacity Leg (full stripe). لا log، لا destage overhead.
  • OSA — Disk Groups: 1 Cache + 1-7 Capacity per group. فَشَل Cache = فَقَدان كُلّ Disk Group!
  • ESA — Storage Pool: كُلّ الأَقراص في pool واحِد. فَشَل قُرص واحِد = فَقَدان ذَلِك القُرص فَقَط.
  • OSA — Dedup/Compress: Capacity Tier فَقَط. لا يَعمَل عَلى Cache.
  • ESA — Compression: مُفَعَّل دائِماً عَلى كُلّ الطَّبَقات. لا Dedup مُنفَصِل — الضَّغط يَكفي.
📤 النَّتيجة العَمَليّة
  • 4.5 ops vs 6 ops ESA يَحتاج عَمَليّات أَقَلّ لِكُلّ كِتابة (توفير 25% I/O).
  • No Cache Bottleneck لا يوجَد Cache SSD يَتآكَل أَو يَمتَلِئ.
  • Better Failure Domain فَشَل قُرص = فَقَدان 1 disk، لَيسَ disk group كامِل.
  • Higher HW Bar ESA يَحتاج NVMe-only + 128GB RAM + 25GbE. لا HDD إِطلاقاً.

03 · vSAN ESA — سياسات التَّخزين | Storage Policies

3.1 بِنية vSAN ESA | ESA Architecture

vSAN ESA يُعيد تَصميم كَيف تُخَزَّن البَيانات. بَدَلاً مِن Disk Groups ثُنائيّة الطَّبَقة (OSA)، يَستَخدِم Storage Pool أُحاديّ الطَّبَقة مَع فَصل ذَكي بَين Performance Leg وَ Capacity Leg.

vSAN ESA — البِنية المُتَقَدِّمة

vSAN ESA يُقَدِّم بِنية جَديدة لِلتَّخزين: Performance Leg وَ Capacity Leg. هَذِه البِنية تُحَقِّق أَداء RAID-1 مَع كَفاءة مِساحة RAID-5/6. فَهم هَذِه المَفاهيم ضَروري لِتَصميم سياسات تَخزين فَعّالة.

🧠 Failures to Tolerate (FTT)

🧠 Failure Tolerance Method (FTM)

  • No Data Redundancy — لا حِماية. لا تَنجو مِن أَيّ عُطل.
  • RAID-1 (Mirroring) — نُسَخ مُتَعَدِّدة. يَدعَم حَتّى FTT 3.
  • RAID-5/6 (Erasure Coding) — شَرائِح بَيانات + Parity. يَدعَم حَتّى FTT 2.

Parity يَستَخدِم حِسابات رِياضيّة لِإِعادة بِناء البَيانات المَفقودة.

🧠 Performance Leg وَ Capacity Leg

  • كُلّ Object لَه Performance Leg (perf leg) وَ Capacity Leg (cap leg).
  • كُلّ كِتابات جُزئيّة تَذهَب أَوَّلاً إِلى Perf Leg.
  • عِندَما يَتَراكَم Full Stripe (128 KB) → يُلقى إِلى Cap Leg.
  • هَذا يُلغي الكِتابات الجُزئيّة عَلى التَّخزين النِّهائي → أَداء كِتابة أَفضَل.
  • الـ VM تَتَلَقّى تَأكيد الكِتابة عِند وُصول البَيانات لِلـ Perf Leg (لَيس Cap Leg).
  • النَّتيجة: RAID-5/6 يُحَقِّق أَداء مُكافِئ لِـ RAID-1 مَع كَفاءة مِساحة أَعلى.

🧠 Capacity Leg — Over-Provisioning

    • 2× لِـ RAID-5/6 (مَثَل: 2 TB addressable لِـ 1 TB VMDK)
  • الـ Objects تُنشَأ بِـ Thin Provisioning افتِراضيّاً — لا استِهلاك فِعلي مُسبَق.

3.3 تَخطيطات RAID

🧠 No Data Redundancy (RAID-0/FTT 0)

  • 1 Perf Leg component + 3 Cap Leg components (لِلتَّوَسُّع).
  • لا Mirrored Set — عُطل واحِد = فَقد البَيانات.
  • 128 KB يَتَراكَم في Perf Leg قَبل الإِلقاء لِلـ Cap Leg.

🧠 RAID-1/FTT 1 Storage Layout

  • 2 Mirrored Perf Leg components.
  • 2 Mirror Sets مَع 3 Child components لِكُلّ مَجموعة في Cap Leg.
  • 256 KB يَتَراكَم قَبل الإِلقاء لِلـ Cap Leg.
  • استِهلاك: 2× المِساحة (100 GB VMDK → 200 GB).

🧠 RAID-5/FTT 1 — التَّخطيط الديناميكي

  • 3 hosts → 2+1 (1.5× استِهلاك مِساحة).
  • 5+ hosts → 4+1 (1.25× استِهلاك مِساحة).

🧠 RAID-6/FTT 2 Storage Layout

  • 3 Mirrored Perf Leg components.
  • 6 Stripes في تَخطيط 4+2 مَع 2 RAID-0 Child components لِكُلّ Stripe.
  • لا يَتَكَيَّف ديناميكيّاً (عَكس RAID-5).
  • 512 KB يَتَراكَم في Perf Leg قَبل الإِلقاء.
  • استِهلاك: 1.5× المِساحة (100 GB VMDK → 150 GB).

3.4 تَحسين الكِتابة وَمُقارَنة المِساحة

🧠 Write Improvement — تَحسين الكِتابة

  1. Guest OS يُصدِر كِتابة لِلـ Virtual Disk.
  2. الكِتابة تُرسَل إِلى ESX Hosts بِـ Perf Leg components.
  3. ESX يَكتُب في Perf Leg component.
  4. تَأكيد يُرسَل لِلـ Guest OS.
  5. Full Stripe يَتَراكَم (128 KB لِكُلّ data bit) → يُلقى لِلـ Cap Leg.

مُقارَنة:

  • vSAN OSA RAID-6: 6 عَمَليّات (3 reads + 3 writes)

DC7 Performance Leg وَ Capacity Leg — الطَّبَقَتان في ESA
🎯 المَفهوم الأَساسي

في ESA، كُلّ كِتابة تَمُرّ بِمَرحَلَتَين: Performance Leg (سَريعة، N+1 mirroring) ثُمَّ Capacity Leg (فَعّالة مِساحيّاً، RAID-5/6). هَذا التَّصميم يَجمَع بَين سُرعة الكِتابة وَكَفاءة المِساحة.

⚙ مَسار الكِتابة خُطوة بِخُطوة
  1. VM تَكتُب 64KB: ESX يُرسِل الكِتابة لِلـ Performance Leg.
  2. Performance Leg: يَكتُب N+1 mirrors (إِذا FTT=1 → 2 نُسَخ عَلى hosts مُختَلِفة). يُؤَكِّد لِلـ VM فَوراً.
  3. تَجميع (Coalescing): Performance Leg يُجَمِّع كِتابات صَغيرة حَتّى يُكَوِّن full stripe.
  4. Destage لِلـ Capacity Leg: Full stripe يُكتَب كَـ RAID-5/6 عَلى hosts مُتَعَدِّدة.
  5. تَنظيف Performance Leg: بَعد destage ناجِح، المِساحة تُحَرَّر في Perf Leg.
📤 لِماذا هَذا أَفضَل مِن OSA
  • Write ACK فَوري VM لا تَنتَظِر كِتابة RAID stripe كامِل.
  • لا Destage Storm في OSA، Cache يَمتَلِئ → destage يُبَطِّئ كُلّ شَيء. في ESA، الـ coalescing مُستَمِرّ.
  • أَقَلّ Write Amplification كِتابة full stripe واحِدة بَدَل partial writes مُتَكَرِّرة.
⚠ أَخطاء شائِعة
  • «Performance Leg = Cache Tier» — لا! هُوَ layer مُنفَصِل مَع mirroring خاصّ بِه. Cache Tier في OSA كانَ SSD واحِد.
  • «أَحتاج أَقراص مُخَصَّصة لِلـ Performance Leg» — لا! كُلّ NVMe يُشارِك في كِلا الطَّبَقَتَين. Storage Pool واحِد.
DC8 Adaptive RAID — كَيف يَتَغَيَّر التَّخطيط تِلقائيّاً
🎯 المَفهوم

في ESA مَع RAID-5، التَّخطيط لَيسَ ثابِتاً! إِذا أَضَفت hosts، vSAN يُغَيِّر مِن 2+1 (ثَلاثة hosts) إِلى 4+1 (خَمسة hosts) تِلقائيّاً لِتَحسين كَفاءة المِساحة. وَالعَكس عِند الإِزالة.

⚙ قَواعِد التَّحَوُّل
  1. 3 hosts → RAID-5 (2+1): أَقَلّ كَفاءة (1.5× space) لَكِن يَعمَل بِ 3 hosts فَقَط.
  2. 4 hosts → RAID-5 (3+1): يَتَحَوَّل تِلقائيّاً — لا تَدَخُّل مِنك.
  3. 5+ hosts → RAID-5 (4+1): أَفضَل كَفاءة (1.25× space). التَّخطيط النِّهائي.
  4. فَترة الاِنتِظار: vSAN يَنتَظِر 24 ساعة بَعد تَغيير عَدَد الـ hosts قَبل إِعادة التَّخطيط.
  5. Scale-Down: إِزالة host → إِذا بَقي 3 hosts فَقَط → يَعود لِـ (2+1). قَد يَحتاج مِساحة إِضافيّة!
⚠ أَخطاء شائِعة
  • «أَحتاج 5 hosts لِـ RAID-5» — لا! ESA يَقبَل RAID-5 بِ 3 hosts (2+1). لَكِن الكَفاءة أَقَلّ.
  • «أُزيل host وَلا شَيء يَحدُث» — اِنتَظِر 24h. vSAN يُعيد التَّخطيط بَعد فَترة stability.
  • «Adaptive = RAID-6 أَيضاً» — لا! RAID-6 ثابِت دائِماً عَلى (4+2). Adaptive لِـ RAID-5 فَقَط.
💡 سيناريو عَمَلي

بَدَأت بِ 3 hosts (RAID-5 2+1): overhead = 1.5×. أَضَفت host رابِع → بَعد 24h: يَتَحَوَّل لِ 3+1 (overhead = 1.33×). أَضَفت خامِس → 4+1 (overhead = 1.25×). وَفَّرت ~17% مِساحة بِدون تَغيير السِّياسة!

DC9 Auto-Policy وَ Skyline Health — الأَتمَتة الذَّكيّة
🎯 المَفهوم

Auto-Policy في ESA يَختار أَفضَل RAID layout تِلقائيّاً بِناءً عَلى عَدَد الـ hosts المُتاحة. Skyline Health يُراقِب الاِمتِثال وَيُنَبِّه عِند وُجود مُشكِلة.

⚙ كَيف تَعمَل Auto-Policy
  1. 3 hosts: Auto-Policy غَير مَدعومة (يَجِب اختِيار يَدَوي).
  2. 4 hosts: Default Storage Policy (RAID-1 FTT=1).
  3. 5-6 hosts: تِلقائيّاً RAID-5 FTT=1 (أَفضَل balance).
  4. 7+ hosts: تِلقائيّاً RAID-6 FTT=2 (أَقصى حِماية).
  5. VM جَديدة بِدون سِياسة مُحَدَّدة: تَأخُذ Auto-Policy. VMs مَوجودة لا تَتَأَثَّر.
📤 Skyline Health Checks
  • Compliant VM تُطَبِّق سِياستها بِنَجاح — كُلّ components في مَكانها.
  • Out of Date السِّياسة تَغَيَّرت لَكِن لَم تُطَبَّق بَعد — يَحتاج Apply.
  • Non-Compliant لا يوجَد مَوارِد كافية لِتَحقيق السِّياسة — تَحَقَّق مِن عَدَد الـ hosts وَالمِساحة.
⚠ أَخطاء شائِعة
  • «Auto-Policy تُغَيِّر VMs المَوجودة» — لا! تُؤَثِّر فَقَط عَلى VMs الجَديدة بِدون سِياسة مُحَدَّدة.
  • «Non-Compliant = بَيانات مَفقودة» — لا! يَعني أَنَّ الحِماية أَقَلّ مِمّا طَلَبت. البَيانات مَوجودة.

3.7 Knowledge Check

Q1. What are the four vSAN ESA object types?

Correct: B

ESA objects: VM Home Namespace (required), VMDK, VM Swap, VM Memory Snapshot.

Q2. What is the purpose of the Performance Leg?

Correct: B

Perf Leg buffers partial writes. When a full stripe accumulates (128 KB), it's flushed to Cap Leg.

Q3. For RAID-5/FTT 1, what is the space consumption multiplier with 5+ hosts?

Correct: B

RAID-5 (4+1) with 5+ hosts = 1.25× space consumption.

Q4. How many hosts are recommended minimum for RAID-5 (4+1)?

Correct: C

Recommended = 6 hosts (5 for data + 1 spare fault domain).

Q5. Does RAID-6 dynamically scale like RAID-5?

Correct: B

RAID-6 does NOT dynamically scale — unlike RAID-5 which adapts between 2+1 and 4+1.

Q6. How many write operations does vSAN ESA RAID-6 use vs OSA?

Correct: B

ESA RAID-6 = 4.5 operations. OSA RAID-6 = 6 operations. ESA is faster.

Q7. With 7+ hosts and Host Rebuild Reserve enabled, Auto-Policy selects:

Correct: C

7+ hosts with Host Rebuild Reserve → Auto-Policy selects RAID-6/FTT 2.

Q8. When Auto-Policy detects a suboptimal policy, what happens?

Correct: B

Skyline sends notification. VMs already deployed with old policy need manual update.

Q9. In vSAN ESA, snapshots are stored as:

Correct: B

ESA combines snapshots with the VMDK object — no separate snapshot object, improving performance.

RAID-5 (2+1) = 1.5× → 100 GB × 1.5 = 150 GB.

🧠 Durability Behavior — 3 سِيناريوهات

السِّيناريو 1: Component يَعود خِلال المُؤَقَّت

  • Component الأَصلي يَعود → vSAN يَدمِج Durability Component مَعَه.
  • أَسرَع مِن إِعادة بِناء كامِلة.

السِّيناريو 2: Component لا يَعود

  • المُؤَقَّت يَنتَهي → vSAN يَبني components جَديدة كامِلة.
  • يَستَخدِم البَيانات مِن Durability + Components السَّليمة.

السِّيناريو 3: Active يُفقَد + Absent يَعود

  • Component نَشِط يَفشَل بَينَما Component غائِب يَعود.
  • vSAN يَستَخدِم Component الغائِب (stale) + Durability Component لِاستِعادة البَيانات.

4.6 أَمثِلة عَمَليّة | Practical Examples

🧠 Degraded — مِثال فَشَل القُرص

  • قُرص يَفشَل في Host → Components عَلى ذَلِك القُرص تُصبِح Degraded.
  • vSAN يَبدَأ إِعادة بِناء فَوريّة عَلى أَقراص سَليمة.
  • بَعد الاكتِمال → Object يَعود إِلى Healthy.

🧠 Absent — مِثال فَشَل Host

  • Host يَفشَل → كُلّ Components عَلَيه تُصبِح Absent.
  • مُؤَقَّت 60 دَقيقة يَبدَأ (Object Repair Timer).
  • إِذا عاد Host خِلال 60 دَقيقة → دَمج (merge).
  • إِذا لَم يَعُد → إِعادة بِناء كامِلة.
  • Auto-rebalance عِند 80% مِن السِّعة المُستَخدَمة.

🧠 Network Partition Examples

مِثال 1: 4-Node RAID-5 Cluster

  • 1 Host مَعزول → Objects تَبقى مُتاحة (3 hosts كافِية لِـ RAID-5).
  • كُلّ الـ Hosts مَعزولة → Objects Inaccessible (لا quorum).

مِثال 2: 6-Node RAID-6 Cluster

  • 2 Hosts مَعزولان → Objects تَبقى مُتاحة (4 hosts كافِية).
  • 3 vs 3 partition → Inaccessible (لا أَغلَبيّة لِلـ quorum).

4.7 · Deep Dive Concept Cards — ما يَجِب أَن يَعرِفَه مُهَندِس المُرونة

المُرونة لَيسَت «ميزة» — هِيَ السَّبَب الرَّئيسي لِوُجود vSAN. هَذِه البِطاقات تَشرَح ما يَحدُث فِعليّاً عِند الأَعطال.

DC10 Object Repair Timer — الـ 60 دَقيقة الحَرِجة
🎯 لِماذا 60 دَقيقة؟

عِندَما يَغيب component (Absent)، vSAN لا يَبدَأ الإِصلاح فَوراً — يَنتَظِر 60 دَقيقة. السَّبَب: إِذا عادَ الـ host خِلال هَذِه الفَترة (reboot مَثَلاً)، يَكفي merge بَسيط بَدَل full rebuild.

⚙ مَسار الأَحداث
  1. T=0: Host يَختَفي → components تَتَحَوَّل لِـ Absent. Objects = Reduced Availability.
  2. T=0 → T=60min: vSAN يَنتَظِر. Durability Components تَلتَقِط الكِتابات الجَديدة.
  3. إِذا عادَ قَبل 60min: Merge فَقَط — يُدمَج ما كُتِب أَثناء الغِياب. سَريع جِدّاً.
  4. إِذا لَم يَعُد بَعد 60min: Full Rebuild — يُعيد بِناء كُلّ الـ components عَلى hosts أُخرى.
  5. Repair Objects Immediately: يُمكِن تَجاوُز المُؤَقَّت يَدَويّاً (Maintenance Mode → Full data evacuation).
⚠ أَخطاء شائِعة
  • «60 دَقيقة = البَيانات في خَطَر» — لا! Objects ما زالَت قابِلة لِلوُصول (Reduced Availability ≠ Inaccessible). لَكِن FTT الفِعلي أَقَلّ.
  • «أُغَيِّر المُؤَقَّت لِ 5 دَقائِق» — مُمكِن لَكِن خَطير. Reboot عادي يَأخُذ 10+ دَقائِق. سَتُسَبِّب rebuilds غَير ضَروريّة.
DC11 Durability Components — الحارِس غَير المَرئي
🎯 المَفهوم

Durability Components هِيَ components مُؤَقَّتة يُنشِئها vSAN لِالتِقاط الكِتابات الجَديدة أَثناء غِياب component أَصلي. تَضمَن أَنَّ لا بَيانات تُفقَد حَتّى أَثناء العُطل.

🏗 كَيف تَختَلِف حَسَب RAID
  • RAID-5/6: Durability Components تَعمَل في كُلّ الأَعطال (Planned + Unplanned). هَذا لِأَنَّ erasure coding يَحتاج كُلّ الـ stripes لِلكِتابة.
  • RAID-1: Durability Components تَعمَل في Planned فَقَط (Maintenance Mode). لِلأَعطال غَير المُخَطَّطة، النُّسخة الثانية كافية.
⚙ دَورة الحَياة
  1. عُطل يَحدُث: vSAN يُنشِئ Durability Component عَلى host آخَر.
  2. كِتابات جَديدة: تُخَزَّن في الـ Durability Component (بِالإِضافة لِلـ components الفَعّالة).
  3. العَودة (قَبل Timer): Merge — الكِتابات مِن Durability Component تُدمَج مَع الـ component الأَصلي.
  4. عَدَم العَودة (بَعد Timer): Full Rebuild — Durability Component يُصبِح جُزءاً مِن الإِصلاح الكامِل.
  5. بَعد الاِكتِمال: Durability Component يُحذَف — لا مِساحة ضائِعة.
💡 مَثَل

VM عَلى RAID-5 (4+1). Host-C يَدخُل Maintenance Mode. vSAN يُنشِئ Durability Component عَلى Host-F. كُلّ كِتابة جَديدة تُسَجَّل هُناك. بَعد 30 دَقيقة يَعود Host-C → merge (دَقائِق). لَو لَم يَعُد → full rebuild (~1h+ حَسَب الحَجم).

DC12 Network Partition — أَسوَأ سيناريو وَكَيف تَتَعامَل
🎯 المَفهوم

Network Partition = مَجموعة مِن الـ hosts لا تَستَطيع التَّواصُل مَع مَجموعة أُخرى. كُلّ جانِب يَظُنّ أَنَّ الآخَر «مات». Quorum يُحَدِّد مَن يَفوز.

⚙ ماذا يَحدُث
  1. الاِنقِسام: Switch failure أَو misconfiguration → hosts لا تَرى بَعضها.
  2. تَقييم Quorum: كُلّ جانِب يُحسِب عَدَد components + witnesses المُتاحة لَدَيه.
  3. الأَغلَبيّة تَفوز: الجانِب الَّذي يَملِك >50% أَصوات يَبقى Active لِلـ Object.
  4. الأَقَلّيّة: Objects تَصيرHere Inaccessible. VMs عَلى هَذا الجانِب تَتَجَمَّد (I/O pause).
  5. الإِصلاح: إِعادة الشَّبَكة → merge تِلقائي → objects تَعود Healthy.
💡 مَثَل: 4-node cluster مَع RAID-5

RAID-5 (3+1) → 4 components + 1 witness = 5 أَصوات. اِنقِسام 2|2 → لا أَحَد لَديه أَغلَبيّة! الحَلّ: Witness عَلى host ثالِث أَو 5th component يُرَجِّح كَفّة. لِهَذا السَّبَب 5+ hosts مُوصى لِ RAID-5.

⚠ كَيف تَتَجَنَّب
  • تَصميم الشَّبَكة: 2× switches + LAG/LACP. لا single point of failure.
  • عَدَد فَردي مِن hosts: يَضمَن أَنَّ جانِباً واحِداً دائِماً لَديه أَغلَبيّة.
  • Stretched Cluster: لِلمَواقِع المُتَعَدِّدة، Witness في مَوقِع ثالِث.

4.8 Knowledge Check

Q1. A disk fails permanently. What is the component state?

Correct: C

Permanent failure = Degraded → immediate rebuild begins.

Q2. A host reboots unexpectedly. Component state becomes:

Correct: B

Transient outage (host reboot) = Absent → 60-minute timer starts.

Q3. How long is the Object Repair Timer?

Correct: B

Default Object Repair Timer is 60 minutes for Absent components.

Q4. RAID-1 creates Durability Components for:

Correct: B

RAID-1 creates Durability Components for planned outages only.

Q5. RAID-5/6 creates Durability Components for:

Correct: C

RAID-5/6 creates Durability Components for both planned and unplanned outages.

Q6. In Scenario 1, the absent component returns within the timer. What happens?

Correct: B

Component returns → vSAN merges Durability Component with it — faster than full rebuild.

Q7. In a 4-node RAID-5 cluster, if 1 host is network-partitioned:

Correct: B

3 remaining hosts are sufficient for RAID-5 — objects remain accessible.

Q8. In a 6-node RAID-6 cluster with a 3 vs 3 network partition:

Correct: B

3 vs 3 = no majority on either side → objects become inaccessible.

TK.1 · vSAN ESA Host Onboarding Runbook

Runbook 1 إِضافة Host جَديد إِلى vSAN ESA Cluster

Phase 1 — التَّحَقُّق المُسبَق (Pre-Flight Checks)

  1. تَأَكَّد أَنَّ الـ Host vSAN ESA ReadyNode مُعتَمَد في الـ HCL (broadcom.com/vmware/hcl).
  2. تَحَقَّق مِن المُواصَفات الدُّنيا: NVMe SSDs فَقَط، 128GB+ RAM، 25GbE+ NICs، CPU مَع AES-NI.
  3. تَأَكَّد مِن firmware versions: NVMe، NIC، BIOS — يَجِب أَن تُطابِق الـ HCL بِالضَّبط (لَيس "قَريب مِنها").
  4. اِفحَص الـ Cluster مِساحة مُتاحة ≥30% لِأَنَّ إِضافَة Host تُطلِق rebalance.
  5. تَحَقَّق مِن vSAN Skyline Health أَخضَر بالكامِل قَبل البَدء.

Phase 2 — إِعداد الشَّبَكة

  1. أَنشِئ vSAN VMkernel Port مَع MTU 9000 عَلى الـ VDS الصَّحيح.
  2. اِختَبِر connectivity: vmkping -d -s 8972 -I vmk1 <vSAN-IP-of-existing-host>. لا يَنجَح؟ مُشكِلة MTU في الـ Switch.
  3. تَحَقَّق مِن NIOC shares: vSAN ≥ 100 shares (High).
  4. اِفصِل vSAN traffic عَلى uplinks مُخَصَّصة إِن أَمكَن.

Phase 3 — الإِضافة

  1. أَضِف الـ Host لِلـ Cluster: vCenter → Cluster → Add Host (أَو SDDC Manager في VCF).
  2. Claim disks: vSAN → Disk Management → Claim Disks → Storage Pool (في ESA كُلُّ القُروص في Pool واحِد).
  3. تَأَكَّد مِن إِظهار الـ Host في Cluster Capacity view بِالحَجم الصَّحيح.

Phase 4 — بَعد الإِضافة

  1. راقِب Resync progress: vSAN → Resyncing Objects — قَد يَستَغرِق ساعات.
  2. تَحَقَّق مِن compliance: VM Storage Policies → Check Compliance.
  3. اِنتَظِر 24h لِلـ Adaptive RAID rebalance إِذا تَجاوَزت host count threshold (مَثَلاً 6→7 hosts).
  4. تَأَكَّد Skyline Health أَخضَر بالكامِل.
  5. وَثِّق الإِضافة في CMDB: Serial، Service Tag، Drive Models، MAC Addresses.

🧠 وَقت تَقديري

Pre-flight: 1-2 ساعات. Installation + Add: 2-3 ساعات. Resync كامِل: 4-24 ساعة حَسَب الحَجم. الإِجمالي: يَوم كامِل لِـ host واحِد.

TK.2 · NVMe Disk Replacement Playbook

Runbook 2 اِستِبدال قُرص NVMe فاشِل في vSAN ESA

Phase 1 — التَّشخيص

  1. تَحَقَّق مِن vSAN Health → Physical Disks → Disk Health. حَدِّد القُرص الفاشِل بِـ Serial Number.
  2. لاحِظ حالة الـ components عَلى القُرص: Degraded (فَشَل دائِم — vSAN بَدَأ الإِصلاح فَوراً) أَو Absent (مُؤَقَّت — vSAN يَنتَظِر 60 دَقيقة).
  3. اِفحَص esxcli logs: esxcli vsan storage list لِمَعرِفَة device path.
  4. تَحَقَّق مِن أَنَّ Cluster لَدَيه مِساحة كافية لِإِعادة بِناء البَيانات في مَكان آخَر.

Phase 2 — الإِزالة الآمِنة

  1. ⚠ لا تَسحَب القُرص قَبل التَّأكيد! سَحب قُرص لا يَزال مَفعَّلاً = فَقد بَيانات.
  2. اِنتَظِر اِكتِمال الإِصلاح: vSAN → Resyncing Objects = 0.
  3. في vCenter: vSAN → Disk Management → Select Disk → Remove Disk → Evacuate Data (إِذا لَم يَكُن أَوتوماتيكيّاً).
  4. اِنتَظِر evacuation (ساعات حَسَب الحَجم — رُبَّما 4-8 ساعات لِـ 3.84TB).
  5. تَأَكَّد مِن إِزالة القُرص مِن vSAN قَبل السَّحب الفِعلي.

Phase 3 — الاِستِبدال الفيزيائي

  1. اِسحَب القُرص الفاشِل (Hot-Pluggable في كُلِّ ReadyNodes الحَديثة).
  2. أَدخِل القُرص الجَديد — يَجِب أَن يَكون نَفس الـ Model أَو مِن نَفس HCL tier.
  3. تَحَقَّق مِن firmware القُرص الجَديد — حَدِّثه إِذا لَزِم قَبل المُتابَعة.

Phase 4 — إِعادة الإِدخال

  1. vSAN يَكشِف القُرص الجَديد تِلقائيّاً. تَحَقَّق: vSAN → Disk Management.
  2. Claim القُرص: Claim → Storage Pool.
  3. راقِب rebalance: vSAN يُعيد تَوزيع البَيانات تِلقائيّاً.
  4. تَأَكَّد بَعد 24 ساعة مِن Skyline Health أَخضَر.
  5. وَثِّق الاِستِبدال: Serial القَديم، الجَديد، السَّبَب (Vendor RMA reference).
⚠ Anti-Pattern: السَّحب الفَوري

"القُرص فاشِل، سَأَسحَبُه فَوراً." لا. إِذا كان Absent فَقَط (مُؤَقَّت)، vSAN يَنتَظِر 60 دَقيقة. سَحبُه يُحَوِّل المُشكِلة المُؤَقَّتة إِلى دائِمة وَيُطلِق rebuild كامِل بِلا داعٍ.

TK.4 · Capacity Planning Worksheet

قَبل أَن تَبني vSAN Cluster، أَجِب على هَذِه الأَسئِلة بِأَرقام، لا تَخمين:

Step 1: Workload Profile
  • كَم VM مُتَوَقَّع في 1 سَنَة؟ 3 سَنَوات؟
  • مُتَوَسِّط حَجم VM (Storage)؟ مَثَلاً 100 GB.
  • مُعَدَّل النُّمو السَّنَوي المُتَوَقَّع؟ (20%، 50%، 100%؟)
  • هَل هُناك VMs كَبيرة (DBs > 1 TB)؟ تَحتاج Stripe-Width مُختَلِف.
Step 2: Resilience Requirements
  • FTT المَطلوب: 1 (Standard)، 2 (Mission Critical)؟
  • FTM: RAID-1 (سُرعة)، RAID-5 (تَوازُن)، RAID-6 (مِساحة)؟
  • هَل هُناك Compliance يَتَطَلَّب Encryption؟
  • هَل هُناك Stretched Cluster بَين Sites؟
Step 3: Sizing Calculation
=== vSAN ESA Capacity Math ===

ASSUMPTIONS:
  • 100 VMs × 100 GB Provisioned = 10 TB Provisioned
  • Thin Provisioning ratio: 60% (الواقِع typically 50-70%)
  • Actual Storage = 10 TB × 0.6 = 6 TB
  • Slack Space (vSAN reserved) = 20% → 7.2 TB
  • Growth buffer (1 year) = 30% → 9.4 TB
  
FTM OVERHEAD:
  RAID-1 (FTT=1): 2x → need 18.8 TB raw
  RAID-5 (FTT=1): 1.25x → need 11.75 TB raw
  RAID-6 (FTT=2): 1.5x → need 14.1 TB raw

HOST COUNT (RAID-5, 6×NVMe per host):
  Need 11.75 TB raw → ~12 TB
  Per host: 6 × 3.84 TB = 23 TB raw
  Hosts needed: 1 → BUT minimum 5 for RAID-5!
  
  Decision: 5 hosts × 4 × 3.84 TB = 76.8 TB raw
  (مِساحة كَبيرة، لَكِنَّ host count تُحَدِّد لا Capacity)

TK.5 · vSAN Health Troubleshooting Decision Tree

عِندَما Skyline Health يَظهَر أَصفَر/أَحمَر، اِتبَع هَذا التَّسَلسُل:

DT vSAN Health Decision Tree

Layer 1: Network Health

  1. اِفحَص vSAN VMkernel ping بَين كُلِّ pair مِن hosts: vmkping -d -s 8972 -I vmk1 <target>.
  2. اِفحَص MTU: vSAN يَتَطَلَّب 9000 — أَيّ host بِـ 1500 = مَشكَلة كُبرى.
  3. اِفحَص NIOC: vSAN saturated بِسَبَب vMotion يَأكُل bandwidth؟
  4. 80% مِن مَشاكِل vSAN جَذرُها في الشَّبَكة.

Layer 2: Disk Health

  1. Skyline → Physical Disks → اِبحَث عَن أَيّ disk بِـ status غَير Healthy.
  2. اِفحَص SMART data: esxcli vsan storage list.
  3. راقِب write endurance: NVMe TBW قَريب مِن النِّهاية؟
  4. إِذا disk في Degraded → rebuild يَجري، اِنتَظِر.

Layer 3: Object Health

  1. اِفحَص: vSAN → Virtual Objects → Object Health.
  2. أَيّ object في حالة Reduced Availability أَو Inaccessible؟
  3. إِذا object inaccessible → VM لا تَستَطيع الـ I/O — فُقدان حَقيقي.
  4. راجِع vSAN → Resyncing Objects — كَم وَقت مُتَبَقّي؟

Layer 4: Cluster Configuration

  1. اِفحَص Cluster Partition: هَل كُلُّ hosts تَرى بَعضها؟
  2. اِفحَص Disk Format Version: هَل كُلُّ hosts نَفس الإِصدار؟
  3. اِفحَص Configuration Consistency.

Layer 5: Performance

  1. Aria Operations → vSAN → Performance Charts.
  2. Backend Latency > 10ms = مَشكَلة.
  3. Write Buffer fill rate مُرتَفِع = vSAN لا يَلحَق بِالكِتابات.
  4. Congestion في الـ DOM Owner؟

🧠 قاعِدة 80/20

80% مِن مَشاكِل vSAN في الشَّبَكة أَو أَخطاء تَكوين. اِبدَأ هُناك. الـ Disks الفاشِلة فِعلاً نادِرة في NVMe Enterprise.

TK.7 · Top 10 vSAN Anti-Patterns

1. The Mixed-Vendor Cluster

"سَنُضيف Dell host إِلى Cluster مِن HPE hosts." لا. vSAN يَدعَم mixing مِن نَفس Vendor فَقَط. اِلتَزِم بِـ HCL بِالضَّبط.

2. The OSA-to-ESA "Migration"

"سَنَنقُل Cluster مِن OSA إِلى ESA in-place." لا يُمكِن. ESA يَتَطَلَّب NVMe-only، policies مُختَلِفة، architecture مُختَلِف. التَّحَوُّل = Cluster جَديد + vMotion / Storage vMotion لِلـ VMs.

3. The "Free Space Forever" Assumption

"vSAN يَعرِض 80TB free، اِملأها." لا. vSAN يَحتاج 20-30% slack space لِلعَمَل. 80% الفِعلي = أَزمة.

4. The MTU Mismatch

"VDS = 9000، Switches = 1500." vSAN traffic سَيَفشَل بِصَمت. Performance terrible، رَغم أَنَّ Health قَد يَظهَر OK مَبدَئيّاً.

5. The Shared Network Catastrophe

"vSAN + Management + vMotion + VM traffic كُلُّها عَلى نَفس uplinks." Saturation تُؤَدّي إِلى host isolation events. اِفصِلها.

6. The Policy-Free Cluster

"كُلُّ VM عَلى Default Policy." لا. Mission-Critical تَحتاج FTT=2، Dev يَحتاج Force Provisioning، Archives تَحتاج RAID-6. Policies لَيست اِختِيارِيّة.

7. The Snapshot Hoarder (vSAN Edition)

Snapshots عَلى vSAN تَستَهلِك مِساحة بِسُرعة — كُلُّ snapshot يُنشِئ delta object. أَكثَر مِن 24 ساعة عُمر = خَطَر.

8. The Force Provisioning Abuser

"Force Provisioning ON لِكُلِّ شَيء — لِنَتَجَنَّب أَخطاء deployment." VMs تَعمَل بِـ FTT أَقَلّ مِن المَطلوب بِدون عِلمك. كارِثة عِند الفَشَل.

9. The "vSAN = Backup" Belief

"vSAN يَحمي بَياناتي بِـ FTT=2، لا حاجَة لِـ Backup." لا. FTT يَحمي مِن فَشَل Hardware، لَيس مِن Ransomware، خَطَأ بَشَري، أَو فَساد البَيانات المَنطِقي.

10. The Firmware Skipper

"Firmware لِلـ NVMe يَعمَل، لِماذا أُحَدِّث؟" Firmware bugs في NVMe السَّبَب الأَوَّل لِفَشَل الأَقراص الـ "غَير مُتَوَقَّع". تابِع Vendor advisories.

القَرار المِعماري المِفتاحي

المُشكِلة: Stretched Cluster بَين Primary وَ DR، أَو Standard Cluster في DR فَقَط مَع Async Replication؟

القَرار: Stretched Cluster.

  • RTO = 0 لِـ Critical VMs.
  • RPO = 0 (Synchronous).
  • Witness في Site ثالِث (Cloud).

الـ Trade-off: Network latency < 5ms بَين Sites مَطلوب. تَكلِفة dark fiber.

الـ Mitigation: Tier-1 فَقَط عَلى Stretched. Tier-2 عَلى Standard Replication.

الدُّروس المُستَفادة

  • Storage Policies قَبل VMs. Build وَ Test سياسات قَبل أَوَّل VM.
  • HCIBench قَبل Production. اِعرِف حُدودك قَبل أَن تَكتَشِفها VMs.
  • Witness في Site ثالِث. لَيس في Primary، لَيس في DR — مَكان مُحايِد.
  • Reference Architectures مَوجودة لِسَبَب. لا تُعيد اِختِراع العَجَلة.

SC.2 · SAN-to-vSAN Migration

«تَرحيل 800 VM مِن Dell EMC SAN إِلى vSAN ESA»
Scenario 2 Major Migration Campaign — Legacy SAN → vSAN ESA in 6 Months

الخَلفيّة: شَركة طَيران، 4 vSphere Clusters، 800 VM، 60 TB، عَلى Dell EMC PowerStore (SAN). End-of-Life خِلال 9 شُهور. الإِدارة قَرَّرَت التَّحَوُّل إِلى vSAN ESA لِتَخفيض Footprint وَتَوحيد Storage مَع Compute.

التَّحَدّيات

⚠ القُيود
  1. Downtime: Production لا يَتَحَمَّل أَكثَر مِن 2 ساعة لِكُلِّ VM.
  2. Mixed Workloads: Databases، Web، CAD/CAM، Analytics.
  3. Skills: Team لَم يَعمَل مَع Storage Policies سابِقاً.
  4. Network: SAN على Fibre Channel، vSAN على Ethernet — نَحتاج تَأهيل Switches.
  5. Capacity: vSAN ESA cluster الجَديد يَجِب أَن يَعمَل بِالتَّوازي مَع SAN حَتّى نِهاية Migration.

الاِستِراتيجيّة — Wave-Based

القَرار: 4 موجات تَدريجِيّة

تَجَنَّباً لِـ Big-Bang، نَنقُل VMs على 4 موجات. كُلُّ موجة تَتَعَلَّم مِن السّابِقة.

Month 1-2: Wave 1 — Dev/Test

  1. Build vSAN ESA Cluster الجَديد (مِن SC.1 above).
  2. Storage vMotion VMs (live) مِن SAN datastore إِلى vSAN datastore.
  3. كُلُّ VM يَأخُذ 10-30 دَقيقة حَسَب الحَجم.
  4. أَهَمّ شَيء: apply Storage Policy لِكُلِّ VM بَعد النَّقل.
  5. راقِب: Performance، Compliance، Skyline Health.
  6. وَثِّق: ما المُشكِلات، ما الـ Workarounds، الوَقت الفِعلي.

Month 3: Wave 2 — UAT

تَطبيق الدُّروس مِن Wave 1. اِختِبار:

  • Performance variance بَين SAN و vSAN.
  • Application compatibility — هَل تَطبيقات تَنتَبِه لِلتَّغيير؟
  • Backup vendor (Veeam) — VBR jobs تَستَمِرّ مَع vSAN؟

Month 4-5: Wave 3 — Production Tier-2

⚠ نُقطَة حَرِجة

كُلُّ Production VM تَتَطَلَّب تَنسيق مَع App Owner. اِفعَل في عُطلَة نِهاية الأُسبوع. اِترُك Rollback path (مَع snapshot قَبل النَّقل).

  1. Schedule مَع App Teams — كُلُّ VM في windows مُحَدَّدة.
  2. Pre-migration snapshot (Veeam).
  3. Storage vMotion live.
  4. Post-validation: Application health checks.
  5. إِذا أَيّ مَشكلة، Storage vMotion back إِلى SAN.

Month 6: Wave 4 — Tier-1

  1. أَكبَر VMs (DBs) — قَد تَحتاج cold migration (Power Off + Move).
  2. Database team يَتَوَلّى تَنسيق Window مَع Business.
  3. Test Performance Benchmark قَبل وَبَعد.
  4. إِذا Performance أَدنى مِن SAN، اِفحَص: Storage Policy، Compression، NVMe write-amp.

القَرار المِعماري المِفتاحي

المُشكِلة: هَل نَستَخدِم Storage vMotion (live) لِكُلِّ شَيء، أَم Cold Move (Power Off) لِـ Tier-1؟

القَرار: Storage vMotion لِـ 95% — Cold لِـ DBs الكَبيرة فَقَط.

السَّبَب: Storage vMotion على VM بِـ 5TB قَد يَستَغرِق 12 ساعات وَيُؤَثِّر عَلى Performance. Cold = 2-3 ساعات downtime + integrity check.

الدُّروس المُستَفادة

  • Wave-Based Approach يَنقُذ. Wave 1 كَشَفت Storage Policy issue كانَت سَتَكون كارِثة في Wave 4.
  • Hardware Refresh قَبل Software Project. لا تَكُن مُجبَراً عَلى Hardware في مُنتَصَف.
  • Communication مَع App Teams مُبَكِّر. 6 شُهور = خِطّة.
  • Rollback Plan لِكُلِّ Wave. SAN يَبقى UP حَتّى نِهاية المَشروع.

SC.3 · OSA-to-ESA Upgrade Decision

«Cluster vSAN OSA الحالي — نُحَدِّث إِلى ESA؟»

الخَلفيّة: Cluster vSAN OSA عَلى 6 hosts، 3 سَنَوات في الإِنتاج، 200 VM. Performance OK، لَكِنَّ Storage utilization 75%. Hardware Refresh قادِم خِلال 6 شُهور. السُّؤال: نُحَدِّث إِلى ESA أَم نَبقى عَلى OSA؟

القَرار — Framework

اِسأَل هَذِه الأَسئِلة
  1. هَل Workloads تَحتاج Performance؟ DBs، Analytics → ESA. Web، File → OSA كافٍ.
  2. هَل Hardware Refresh قادِم؟ نَعَم → ESA (لا تُضِف SSD/HDD جَديد، اِشتَرِ NVMe).
  3. هَل Budget يَسمَح؟ NVMe Enterprise أَغلى. اِحسِب TCO.
  4. هَل Team جاهِز؟ ESA يَتَطَلَّب فَهم جَديد (Policies، Object Types).
  5. هَل Compliance يَتَطَلَّب Encryption + Performance؟ → ESA حَتميّاً.

السِّيناريو — قَرار المَشروع

بَعد التَّحليل:

  • Workloads: 30% DBs + 70% Web/Apps. الـ DBs تَستَفيد مِن ESA.
  • Hardware Refresh في 6 شُهور = فُرصة ذَهَبيّة.
  • Budget: NVMe overhead = +25%، لَكِنَّ Density أَعلى = أَقَلّ hosts.
  • Team: 2 أُسبوع تَدريب كافٍ.

القَرار: اِنتَقِل إِلى ESA. لَكِن — Build cluster جَديد، Storage vMotion VMs (SC.2 approach). لا in-place upgrade — ESA لا يَدعَم migration in-place مِن OSA.

Timeline

  2. Month 2-3: Receive، Rack، Install ESA Cluster.
  3. Month 4: Build Storage Policies، Pilot VMs.
  4. Month 5: Migrate VMs بِالتَّوازي (OSA يَبقى UP).
  5. Month 6: Decommission OSA hosts.

الدُّروس

  • OSA-to-ESA لَيست upgrade — هِيَ migration. اِفهَم الفَرق.
  • NVMe Cost أَعلى، لَكِن Density أَعلى. TCO قَد يَكون مُتَقارِب.
  • ESA لِـ Performance-Sensitive Workloads. لا تَنتَقِل لِأَنَّ "ESA جَديد".

SC.4 · vSAN Capacity Crunch Crisis

«Cluster عِند 88% — VMs تَتَجَمَّد، ما العَمَل؟»
Scenario 4 Emergency Capacity Recovery — vSAN Above 85%

الخَلفيّة: Friday 14:30. PagerDuty alert: "vSAN Cluster usage = 88%". 5 دَقائق لاحِقاً، tickets بَدَأت: "VMs لا تَكتُب." بَدَأ vSAN يَرفُض I/O writes جَديدة. CTO قَلِق — كَيف وَصَلنا هُنا؟

Phase 1: Immediate Response (الـ Runbook)

  1. 14:30 — Confirm: vSAN usage = 88%. Skyline Health = أَحمَر.
  2. 14:32 — اِفصِل Snapshots نَشِطة: vCenter → Storage → اِبحَث snapshots. كَم mass storage تُحَرَّر بِحَذفها؟
  3. 14:35 — Identified: 12 snapshots قَديمة (Veeam left behind). 4.5 TB.
  4. 14:40 — اِحذِف Snapshots — vSAN يَبدَأ recovery space. حَوالي 30 دَقيقة.
  5. 15:10 — Usage = 79%. Writes تَعمَل مَرَّة أُخرى. ✓
  6. 15:30 — Update CTO + Incident Manager.

Phase 2: Root Cause Investigation

  1. لِماذا لَم نَكُن نَرى هَذا قادِماً؟
  2. Aria Operations: تَنبيهات Capacity كانَت مُكَوَّنة عِند 90%، لَيس 70%.
  3. Veeam: snapshots كانَت تُتَرَك خَلف بِسَبَب backup failures لَم تُحَلَّ.
  4. Capacity Growth: 5% / month — لَم يُحَدَّث Capacity Plan.
  5. Snapshot policy: لا hard limit عَلى عُمر snapshot.

Phase 3: Post-Mortem & Permanent Fixes

  2. Alert escalation: 70% (Email)، 75% (Slack)، 80% (PagerDuty).
  3. Capacity Plan: مُراجَعة شَهرِيّة، لَيس سَنَوِيّة.
  4. HCI Capacity dashboard في NOC — visible دائِماً.
  5. Backup Job Health → SLA tracking مَع تَنبيهات.

الدُّروس

  • Capacity هِيَ Storage Admin's #1 worry. أَيّ شَيء آخَر يُحَلّ بِالـ Time. Capacity Crunch يَحدُث الآن.
  • Snapshots هِيَ Silent Killer. راقِبها يَوميّاً، لا أُسبوعِيّاً.
  • Alerts المُبَكِّرة تُنقِذ الوَظائِف. 70% threshold = stress-free recovery. 90% = panic mode.
  • Backup Failures = Storage Risk. لا تَفصِل الـ Backups كَـ "مَشكلة backup team". هِيَ مَشكَلتُك.

SC.5 · Stretched Cluster Site Failure

«Primary Site فُقِد — هَل DR يَستَلِم؟»
Scenario 5 Stretched vSAN Failure Test — Real Site Loss Drill

الخَلفيّة: Stretched vSAN Cluster بَين Site-A (Primary) وَ Site-B (Secondary)، Witness في Cloud. Quarterly DR Drill: نَفعَل failure مُتَعَمَّد لِـ Site-A لِنَختَبِر استِجابة Site-B.

Pre-Drill Preparation

  1. Communication: أَخبِر كُلَّ Stakeholders — "DR Drill in 48h".
  2. تَأَكَّد كُلِّ VMs كَامِلة Compliance: Storage Policies تَتَطَلَّب FTT-Site=1.
  3. Witness Health = Green.
  4. Inter-Site Latency < 5ms.
  5. Backup قَبل Drill (احتِياط).

Drill Execution — Site-A Failure Simulation

  1. T-0: اِفصِل كُلَّ network links مِن Site-A (Inter-Site + Witness).
  2. T+5s: Site-B + Witness يَفقِدون اتِّصال Site-A.
  3. T+10s: Site-B + Witness تَتَواصَل — تُشَكِّل أَغلَبيّة (2 of 3).
  4. T+15s: Site-B يَستَلِم — VMs في Site-A تَفقِد الوُصول.
  5. T+30s: HA يَبدَأ — VMs تُعاد تَشغيل عَلى Site-B (مَع Read Locality).
  6. T+2min: كُلُّ VMs تَعمَل في Site-B.
  7. T+5min: Aria Operations تُؤَكِّد كُلُّها reachable.

What Worked

✅ المُلاحَظات الإِيجابِيّة
  • HA Failover خِلال 2 دَقيقة — مُطابِق لِـ RTO.
  • لا data loss — Synchronous replication حافَظَت على Consistency.
  • Application response: 95% مِن VMs response في <30 ثانية بَعد restart.
  • Backup integrity: Test restore نَجَح بِنَجاح.

What Did Not Work

⚠ المُلاحَظات السَّلبيّة
  • 5 VMs لَم تَعمَل — اِكتَشَفنا أَنَّها كانَت بِـ Storage Policy غَير مُتَوافِق مَع Stretched.
  • 2 Applications تَوَقَّفَت لِأَنَّ Database قَطَعَ connection — App لَم تَكُن configured لِـ retry.
  • Witness استَجاب بَطيئاً — Cloud network latency تَقَلَّبَ.
  • Monitoring (Aria Operations) كانَ في Site-A — فَقَدنا visibility لِـ 15 دَقيقة.

Recovery Phase — Site-A Comes Back

  1. اِعِد network connectivity إِلى Site-A.
  2. vSAN يَكشِف Site-A returning، يَبدَأ Resync.
  3. Resync كامِل لِكُلِّ Components قَد يَستَغرِق ساعات (حَسَب كَمّ data تَغَيَّر في Drill).
  4. راقِب: vSAN → Resyncing Objects.
  5. تَأَكَّد Skyline Health أَخضَر بَعد اِكتِمال.

Post-Mortem Action Items

  1. اِفحَص كُلَّ VMs لِـ Storage Policy compliance — اِجبَر FTT-Site=1.
  2. Applications: configure retry logic لِـ DB connections.
  3. Witness: اِنقُل إِلى مَوقِع ثالِث أَفضَل (latency أَقَلّ).
  4. Monitoring: Aria Operations في Site-B أَيضاً (Active/Active).
  5. Drill كُلَّ 6 شُهور لا 12.

🧠 الدُّروس مِن DR Drills

  • الـ Drill يَكشِف ما الـ Documents لا تَكشِف.
  • Monitoring في موقِع واحِد = أَنت أَعمى وَقت تَحتاج البَصَر.
  • App-level resilience أَهَمّ مِن Infrastructure-level.
  • Witness في Cloud = جَيِّد، لَكِن راقِب latency.
  • Stretched ≠ Magic. اِختَبِر، اِختَبِر، اِختَبِر.

🧠 ما يَجمَع السِّيناريوهات الخَمسة

  1. التَّخطيط أَهَمّ مِن التَّنفيذ. 60% مِن النَّجاح في الـ Pre-Work.
  2. Wave/Stage approach. Big-Bang في Storage = كارِثة.
  3. Storage Policies قَبل أَوَّل VM. لَيست optional.
  4. Capacity Monitoring يَوميّاً. ليس شَهرِيّاً.
  5. Rollback Plan لِكُلِّ Migration. SAN القَديم يَبقى UP.
  6. DR Drills الحَقيقيّة. Tabletop ≠ Real Test.
  7. Storage Admin = 25% Tech، 75% Process + Communication.

عَلاقة هَذِه السِّيناريوهات بِالامتِحان

الامتِحان لَن يَسأَلَك «كَيف تُخَطِّط لِـ SAN Migration». لَكِنَّه سَيَسأَل: «ما FTT في RAID-5؟»، «مَتى Force Provisioning مَفعَّل؟»، «ما هُوَ Witness؟». السِّيناريوهات هُنا تُعطيك السِّياق الَّذي يَجعَل الإِجابات مَنطِقيّة — لا حِفظاً بَل فَهماً.
