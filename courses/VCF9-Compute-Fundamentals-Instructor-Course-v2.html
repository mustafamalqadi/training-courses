Golden Rule for VCF Compute

كُلّ VM في VCF يَعتَمِد على ESX Host، وَكُلّ ESX Host يُدار مِن vCenter، وَكُلّ مَجموعة ESX Hosts تَتَجَمَّع في Cluster، وَكُلّ Cluster يَستَفيد مِن HA + DRS + vMotion.

إِذا فَهِمت هَذِه السِّلسِلة، تَكون قَد فَهِمت 80% مِن VCF Compute.

Module 0.5 · Beyond Configuration — Foundations for Real VCF Admins

لِماذا VCF؟ وَمَشهَد Virtualization

Why VMware Cloud Foundation & The Virtualization Landscape

قَبل أَن نَغوص في esxcli وَ vCenter وَ vMotion، اِفهَم لِماذا وُجِدَت Virtualization أَصلاً، وَكَيف وَصَلَت VMware إِلى ما هِيَ عَلَيه اليَوم، وَلِماذا اِشتَرَت Broadcom كُلَّ شَيء وَأَطلَقَت VCF كَمُنتَج وَحيد. هَذِه الوَحدة لَيست في الامتِحان — لَكِنَّها الفَرق بَين مَن يَكتُب أَوامِر وَ مَن يُصَمِّم البَيئة.

Learning Objectives — أهداف الوَحدة
  • تَفهَم المُشكِلة الجَوهَريّة الَّتي تَحُلُّها Virtualization في DataCenter حَديث.
  • تُمَيِّز بَين Type 1 وَ Type 2 Hypervisors وَلِماذا ESX يَختار Type 1.
  • تَعرِف تاريخ vSphere مِن ESX 1.0 (2001) إلى VCF 9.0.
  • تَفهَم لِماذا اِشتَرَت Broadcom شَركة VMware وَماذا تَغَيَّر في النَّموذَج التِّجاري.
  • تُمَيِّز بَين أَدوار VCF Administrator وَ vSphere Admin وَ Cloud Engineer.
  • تَعرِف مَكان VCF في مَشهَد المُنافِسة (Nutanix، OpenStack، Hyper-V، Public Cloud).

0.5.1 · The Problem Virtualization Solves

قَبل Virtualization (~عام 2000)، كانَت كُلُّ DataCenter تَحت قاعِدة ذَهَبيّة: "One Application Per Server". التَّطبيق ينَهار؟ يُؤَثِّر فَقَط على Server واحِد. لَكِن النَّتيجة:

❌ ما قَبل Virtualization
  • اِستِخدام Server المُتَوَسِّط: 5-15٪ فَقَط.
  • كُلُّ تَطبيق يَتَطَلَّب Server فيزيائي مُستَقِلّ.
  • 3-6 أَشهُر لِشِراء وَتَركيب Server جَديد.
  • تَرحيل تَطبيق = إِيقاف خِدمة لِأَيّام.
  • DR/HA يَتَطَلَّب Servers ضِعف الحَجم — مُكَلِّفة جِدّاً.
  • التَّبريد وَالكَهرَباء يَستَهلِكون مَيزانيّات ضَخمة.
✅ مَع Virtualization
  • اِستِخدام Server: 70-85٪ (بِفَضل DRS).
  • 20-40 VM على Host واحِد بِأَمان.
  • VM جَديدة في 5 دَقائق.
  • vMotion يَنقُل VM حَيّة بَين Hosts.
  • HA يَستَعيد VMs تِلقائيّاً عَن Host فاشِل.
  • تَوفير 60-80٪ في DataCenter Footprint.

Virtualization لَم تَكُن مُجَرَّد تَحسين كَفاءة — بَل غَيَّرَت اِقتِصاد الـ IT بِالكامِل. وَ VMware كانَت الشَّركة الَّتي قادَت هَذا التَّحَوُّل.

0.5.2 · Hypervisor Types — Type 1 vs Type 2

Hypervisor: طَبَقة Software تُتيح تَشغيل عِدّة Operating Systems على نَفس Hardware، عَبر إِنشاء طَبَقة Virtual لِلـ CPU، Memory، Storage، وَNetwork.

الخاصِّيّةType 1 (Bare-Metal)Type 2 (Hosted)
التَّثبيتمُباشَرة على Hardwareفَوق Host OS (Windows/Mac/Linux)
الأَداءعالٍ — وُصول مُباشَر لِلـ Hardwareأَدنى — طَبَقة OS إِضافيّة
الأَمانسَطح هُجوم أَصغَرسَطح هُجوم أَكبَر (يَشمَل Host OS)
الاِستِخدامProduction · DataCenterDevelopment · Testing · Labs
أَمثِلةVMware ESX/ESXi، Hyper-V، KVM، XenVMware Workstation، Oracle VirtualBox، Parallels
لِماذا يَختار ESX طَريق Type 1؟

في DataCenter، كُلُّ مايكروثانية في الـ I/O تَهُمّ. طَبَقة OS إِضافيّة بَين VM وَالـ Hardware تَعني Context Switches أَكثَر، اِستِخدام Memory أَكثَر، وَنُقاط فَشَل إِضافيّة. ESX تَخَلَّى تَماماً عَن مَفهوم "Host OS" — هُوَ نَفسُه الـ OS، وَ VMs تَعمَل عَلَيه مُباشَرة.

0.5.4 · Broadcom Era — What Changed

في نوفمبر 2023، أَكمَلَت Broadcom اِستِحواذها على VMware بِـ 61 مِليار دولار — أَكبَر صَفقة تِقَنيّة في تاريخ الصِّناعة. التَّأثير على VCF Admin اليَومي عَميق:

⚠ Top 5 Changes a VCF Admin Must Know
  1. Subscription-Only Licensing: اِنتَهَت الـ Perpetual Licenses. كُلُّ شَيء اشتِراك سَنَوي. الحِساب يَصبَح: عَدَد الـ Cores × الـ Tier × المُدَّة.
  2. Product Consolidation: 168+ مُنتَجاً سابِقاً → 2 خَطّ رَئيسي: VCF (full SDDC) وَ VVF (vSphere Foundation — vSphere + VSAN فَقَط).
  3. vSphere alone is gone: لا يُمكِنُك شِراء vSphere مُستَقِلّاً. إذا أَردت ESX، تَدفَع VVF كَحَدّ أَدنى.
  4. End of Free ESXi: اِنتَهَى ESXi Free الَّذي اِستَخدَمَهُ المُختَبِرون لِسَنَوات. اِلتِزام تامّ بِالاِشتِراك.
  5. Partner Program Restructure: آلاف الشُّرَكاء الصِّغار تَمّ اِستِبعادهم — التَّركيز على شُرَكاء كِبار فَقَط.
⚠ Anti-Pattern: «نَحن نَملِك License قَديم — لا نَحتاج VCF»

كَثير مِن العُمَلاء يَعتَقِدون أَنَّ اِستِمرار اِستِخدام vSphere 7 القَديم يُنقِذهم. الحَقيقة:

  • الـ Support يَنتَهي قَريباً — لا Patches أَمنيّة بَعد ذَلِك.
  • الـ Compatibility مَع Hardware الجَديد يَتَوَقَّف.
  • كُلُّ تَجديد ضَروري سَيَحتاج VCF. التَّأخير يُضاعِف التَّكلِفة لاحِقاً.

الحَلّ: اِفهَم تَكلِفَتك الحَقيقيّة، خَطِّط لِلتَّحديث، أَو ابدَأ تَقييم البَدائِل (Nutanix, Proxmox, Hyper-V) بِجِدّيّة.

0.5.5 · The VCF 9.0 Architecture — Big Picture

VCF لَيست مُنتَجاً واحِداً، بَل مَجموعة مُنتَجات مُتَكامِلة تُدار كَوِحدة واحِدة. اِفهَمها كَطَبَقات:

L1 Compute Layer — ESX + vCenter (هَذِه الدَّورة)
  • ESX: الـ Hypervisor — يُشَغِّل VMs على Hardware.
  • vCenter: الإِدارة المَركَزيّة — Inventory, Tasks, Policies.
  • vLCM: Lifecycle Manager — التَّحديث وَالـ Patching.
  • vMotion · HA · DRS · FT: خَدَمات الـ Availability وَالـ Mobility.
L2 Storage Layer — VSAN
  • vSAN ESA (Express Storage Architecture): النَّموذَج الحَديث في VCF 9 — مَبني لِـ NVMe-only.
  • vSAN OSA (Original): النَّموذَج التَّقليدي — Hybrid (SSD + HDD) أَو All-Flash.
  • Storage Policies (SPBM): تُحَدِّد FTT، RAID، Performance لِكُلِّ VM.
  • vSAN File Services: NFS/SMB من VSAN cluster.
L3 Networking Layer — NSX
  • NSX Manager: Control Plane لِلـ Software-Defined Networking.
  • Logical Switches (Segments): VLAN-free isolation.
  • Distributed Firewall: Micro-segmentation عَلى مُستَوى VM.
  • Edge Services: Routing، Load Balancing، VPN.
  • Federation: رَبط عِدّة Sites لِـ DR.
L4 Management & Automation Layer
  • SDDC Manager: الإِدارة المُوَحَّدة لِلـ VCF stack (BringUp، LCM، Domains).
  • Aria Operations (Operations Manager): Monitoring، Capacity، Anomaly Detection.
  • Aria Automation (vRealize Automation): Self-Service Portal، Blueprints.
  • Aria Logs (vRealize Log Insight): Log Aggregation.
  • HCX: Workload Migration بَين Sites أَو Clouds.

🧠 ما تَحتاج تَتَذَكَّر

هَذِه الدَّورة تُغَطّي L1 فَقَط (Compute). لاحِقاً سَتَتَعَلَّم Storage (VSAN)، Networking (NSX)، Management (Aria) في دَورات مُستَقِلّة. لَكِن كُلَّها تَجلِس فَوق ESX/vCenter — مَن يُتقِن L1، يُتقِن الباقي بِسُرعة.

0.5.8 · Study Methodology — How to Master VCF

7-Step Study Method لِـ VCF 9

ESX Installation & Configuration

Bare-Metal Hypervisor · Boot · Networking · Time · Security

اللَّبِنة الأَساسيّة لِكُلّ بيئة VCF.

1.1 لِماذا ESX؟ | Why ESX First?

قَبل الحَديث عَن vCenter أَو Clusters، يَجِب فَهم اللَّبِنة الأَساسيّة: ESX Host. كُلّ VM في VCF تَعمَل فَوقه. ESX هُوَ Bare-Metal Hypervisor — يَعني أَنَّه يُثَبَّت مُباشَرة على الـ hardware، بِدون operating system وَسيط مِثل Linux أَو Windows. هَذا يُعطيه أَداءً أَعلى وَأَماناً أَقوى.

لِماذا نَبدَأ بِـ ESX؟

كُلّ VM في VMware Cloud Foundation يَعمَل فَوق ESX Host. بِدون فَهم كَيف يَتِمّ تَثبيت ESX وَتَهيئَته، لا يُمكِنك بِناء Management Domain صَحيح وَلا تَشغيل أَيّ Workload. هَذِه الوِحدة تُؤَسِّس كُلّ ما يَأتي.

🧠 About ESX — ما هُوَ بِالضَّبط؟

  • ESX هُوَ Bare-Metal Hypervisor — يُثَبَّت مُباشَرة على الـ hardware بِدون operating system وَسيط.
  • يُوَفِّر Resources الـ hardware (CPU · Memory · Storage · Network) بِشَكل ديناميكي لِلـ VMs.
  • يَفصِل (decouples) الـ VMs عَن الـ hardware المادّي — يُعطيها قابِليّة النَّقل بَين الـ Hosts.
  • يَدمِج Resources المادّيّة في Pools يُعاد تَوزيعها ديناميكِيّاً.

1.2 ميزات الأَمان | Security Features

ESX لَيس فَقَط Hypervisor سَريع، بَل مَنصّة آمِنة بِالتَّصميم. كُلّ Host يُولَد مُحَصَّناً: Firewall مُغلَق افتِراضيّاً، Memory Hardening يَحمي الـ memory مِن الهَجَمات، Kernel Module Integrity يَمنَع تَحميل وَحَدات غَير مُوَقَّعة، وَ TPM 2.0 + UEFI Secure Boot يَضمَنان سِلسِلة إِقلاع مَوثوقة. حَتّى Core Dumps تُكتَب مُشَفَّرة.

🧠 Security Features في ESX

  • Host-based Firewall مُفَعَّل افتِراضيّاً.
  • Memory Hardening — حِماية الـ memory مِن هَجَمات الإِزاحة.
  • Kernel Module Integrity — التَّحَقُّق مِن سَلامة الوَحَدات قَبل التَّحميل.
  • TPM 2.0 — Trusted Platform Module لِتَخزين المَفاتيح.
  • UEFI Secure Boot — تَحَقُّق الإِقلاع المُوَقَّع.
  • Encrypted Core Dumps — حِفظ تَصاوير الأَعطال مُشَفَّرة.

1.4 Boot Bank — التَّقسيم وَ Rollback | Layout

ESX يَستَخدِم تَقسيماً ذَكيّاً: boot bank 0 هُوَ الإِصدار الحالي، boot bank 1 هُوَ الإِصدار السّابِق. هَذا يُعطي قُدرة Rollback فَوريّة لَو فَشِل التَّحديث. وَ ESX-OSData volume يَفصِل البَيانات الدّائِمة (configs، dumps) عَن الـ Volatile (logs، traces). فَهم هَذا التَّقسيم ضَروري لِلـ Troubleshooting.

🧠 ESX Boot Bank Layout — التَّقسيم الدّاخِلي

  • boot bank 0 → /bootbank (الإِصدار الحالي).
  • boot bank 1 → /altbootbank (الإِصدار السّابِق — لِـ Rollback).
  • ESX-OSData volume يَنقَسِم إِلى:
    • Persistent: /productLocker، /locker، /var/core، /usr/lib/vmware/isoimages، /usr/lib/vmware/floppies.
    • Non-persistent: /var/run، /var/log، /var/vmware، /var/tmp، /scratch.

الـ Persistent Data = نادِراً ما يُكتَب (configs، ISOs، dumps). الـ Non-persistent Data = كَثير الكِتابة (logs، traces).

1.5 التَّثبيت التَّفاعُلي | Interactive Installation

لِأَوَّل عَدَد قَليل مِن ESX Hosts في Management Domain، يُنصَح بِالتَّثبيت التَّفاعُلي عَبر USB أَو iDRAC/iLO. الخُطوات بَسيطة لَكِنّها مُتَسَلسِلة: EULA → اختِيار DPU (لَو مَوجود — UEFI إِجباري) → القُرص → keyboard layout → password root → F11. لِلتَّوسيع لاحِقاً، استَخدِم VCF Installer أَو Auto Deploy.

🧠 Interactive Installation — الخُطوات

  1. اِبدَأ مِن صَفحة Welcome.
  2. وافِق على EULA.
  3. إِذا كانَ النِّظام يَدعَم DPU → اختَره (UEFI إِجباري).
  4. اختَر الـ disk (Boot Disk).
  5. اختَر تَخطيط keyboard layout.
  6. أَدخِل password root.
  7. اضغَط F11 لِبَدء التَّثبيت.
  8. اضغَط Enter لِإِعادة الإِقلاع.

Validating the Installation

بَعد إِعادة الإِقلاع، تَحَقَّق مِن النَّجاح عَبر إِحدى وَسيلَتَين:

  • DCUI (Direct Console User Interface) — على شاشة الـ server.
  • VMware Host Client — مِن الـ browser بِـ https://<ESX_IP>.

سَجِّل دُخول كَـ root وَتَحَقَّق مِن أَنَّ Summary لا يُظهِر تَنبيهات.

1.6 تَهيئة Management Network

بَعد التَّثبيت، الـ Host «أَصَمّ» — لا يَعرِف vCenter. تَحتاج تَهيئة Management Network مِن DCUI: VLAN، IP، Subnet، Gateway، FQDN، DNS. هَذِه المَعلومات يَجِب أَن تَكون مُسبَقاً في DNS الفِعلي (forward + reverse). أَيّ خَطَأ هُنا يُفشِل لاحِقاً VCF Bring-up.

🧠 Configuring the Management Network (DCUI)

  • Network Adapter — اختَر الـ NIC الفِعليّ.
  • VLAN ID لِلـ Management Network.
  • IPv4 / IPv6 — IP، Subnet Mask، Default Gateway.
  • Hostname — اسم نِطاق مُؤَهَّل (FQDN).
  • DNS Servers (لا تَستَخدِم Custom DNS Suffixes).

كَرِّر هَذِه التَّهيئة لِكُلّ ESX Host في Management Domain.

1.7 الشَّهادات | Self-Signed vs CA-Signed

أَوَّل شَهادة ESX تُولَد قَبل ضَبط FQDN، فَتَكون باسم localhost.localdomain. عِند بَدء VCF Bring-up، يَفشَل التَّحَقُّق مِن HTTPS لِأَنَّ FQDN لا يُطابِق Common Name. الحَلّ: إِعادة تَوليد الشَّهادة عَبر generate-certificates ثُمَّ إِعادة تَشغيل hostd + vpxa + rhttpproxy. لِبيئات الإِنتاج الصّارِمة، استَبدِلها بِـ CA-Signed Certificate.

🧠 Self-Signed vs CA-Signed Certificates

  • عِند تَثبيت ESX، يُولَّد Self-Signed Cert قَبل ضَبط FQDN → Common Name = localhost.localdomain.
  • بَعد ضَبط Hostname، يَجِب إِعادة تَوليد الشَّهادة:
    • فَعِّل SSH مِن VMware Host Client.
    • سَجِّل دُخول وَنَفِّذ: /sbin/generate-certificates.
    • أَعِد تَشغيل: /etc/init.d/hostd restart && /etc/init.d/vpxa restart && /etc/init.d/rhttpproxy restart.
    • عَطِّل SSH مَرّة أُخرى.
  • إِذا تَطَلَّبَت السِّياسة CA-Signed → اِنسَخ الشَّهادة وَالمِفتاح إِلى /etc/vmware/ssl وَأَعِد تَشغيل hostd وَ vpxd.

1.8 مُزامَنة الوَقت | Time Sync (NTP vs PTP)

الوَقت المُتَّسِق بَين كُلّ مُكَوِّنات VCF أَمر حَرِج. بِدونه: شَهادات تَفشَل، Logs غَير قابِلة لِلتَّتَبُّع، HA يَتَصَرَّف بِشَكل غَير مُتَوَقَّع. الخِيارات: NTP (دِقّة Millisecond، الأَكثَر شُيوعاً) أَو PTP (دِقّة Microsecond، لِتَطبيقات Telco/Trading). القاعِدة الذَّهَبيّة: لا تَعمَل النِّظامَتان مَعاً، وَأَفضَل مُمارَسة 3 NTP Servers عَلى الأَقَلّ.

1.9 التَّحَكُّم بِالوُصول البَعيد | Remote Access Control

ESX Firewall مُفَعَّل افتِراضيّاً وَيَحجُب كُلّ شَيء غَير مَسموح. Lockdown Mode يَرفَع الحِماية لِأَقصى دَرَجة: لا تَسجيل دُخول مُباشِر — كُلّ شَيء يَمُرّ عَبر vCenter أَو DCUI. هَذا مَطلَب إِلزامي في كَثير مِن إِطارات Compliance (PCI-DSS، HIPAA).

🧠 Remote Access Control — حِماية ESX

  • ESX Firewall مُفَعَّل افتِراضيّاً وَيَحجُب أَيّ traffic غَير مَسموح.
  • Lockdown Mode يَمنَع تَسجيل الدُّخول المُباشِر — الوُصول فَقَط عَبر DCUI أَو vCenter.
  • الخَدَمات (NTP Client، SSH Client) تُدار بِواسِطة مُستَخدِمين بِصَلاحيّات Admin.
  • كُلّ خِدمة تَستَطيع تَحديد IP/subnet مَسموح لَها بِالاتِّصال.

1.10 Memory Tiering — جَديد VCF 9.0 | New in 9.0

إِحدى أَبرَز ميزات VCF 9.0: استِخدام NVMe PCIe Devices كَطَبَقة ثانِية مِن الـ memory. النِّظام يُوَزِّع تِلقائيّاً: الـ Hot Pages على DRAM السَّريعة، وَالـ Cold Pages على NVMe الأَرخَص. النَّتيجة: تَشغيل VMs أَكثَر بِنَفس hardware. الـ Guest OS لا يَعرِف وَلا يَكتَرِث — كُلّ شَيء شَفّاف.

Memory Tiering with NVMe — مُمَيِّز VCF 9.0

تَقنيّة جَديدة: استِخدام NVMe PCIe Devices كَطَبَقة ثانِية مِن الـ memory. النَّتيجة:

  • زِيادة سَعة الـ memory الإِجماليّة في الـ Host.
  • تَوزيع ذَكي: الـ VM الحَرِجة على DRAM، الأَقَلّ نَشاطاً على NVMe.
  • لا يَعرِف الـ Guest OS ولا الـ Workload أَيّ شَيء عَن هَذا التَّوزيع.
  • أَكثَر Workloads على نَفس الـ Host = تَكلِفة أَقَلّ.

1.X · Deep Dive Concept Cards — مَا يَجِب أَن يَعرِفه مُهَندِس ESX

هَذِه البِطاقات تُلَخِّص أَهَمّ المَفاهيم في ESX بِعَمَق المُمارِس — كَيف يَفكُر مُهَندِس ESX حِين يُصَمِّم، وَيُحَدِّث، وَيُصلِح.

ARCH ESX Architecture — مِن VMkernel إلى UserWorld
🎯 المَفهوم الأَساسي

ESX لَيس Linux، وَلا Unix. هُوَ OS مُتَخَصِّص طُوِّر مِن الصِّفر لِيَكون Hypervisor. يَحتَوي على VMkernel (النَّواة) وَ UserWorlds (العَمَليّات).

🏗 المُكَوِّنات الرَّئيسيّة
  • VMkernel: النَّواة — يُدير CPU، Memory، Storage، Network لِلـ VMs.
  • VMM (Virtual Machine Monitor): Process لِكُلِّ VM — يُتَرجِم instructions مِن GuestOS إلى Hardware.
  • hostd: agent يُتيح اِتِّصال vCenter وَ esxcli.
  • vpxa: vCenter agent — يُنَفِّذ الأَوامِر مِن vCenter.
  • DCUI: Direct Console User Interface — الشّاشة الصَّفراء الَّتي تَراها عَبر iDRAC.
⚙ كَيف تَعمَل المُكَوِّنات مَعاً
  1. VM تَطلُب CPU instruction.
  2. VMM يَلتَقِطها (binary translation أَو direct execution مَع VT-x).
  3. VMkernel يُجَدوِل CPU time لِلـ VMM.
  4. Hardware يُنَفِّذ.
  5. النَّتيجة تَعود لِـ VM.

كُلُّ هَذا في نانوثوانٍ. الـ Overhead أَقَلّ مِن 5% في VCF 9.

⚠ أَخطاء شائِعة
  • الاعتِقاد أَنَّ ESX = Linux: أَوامِر top، ps تَعمَل لَكِنَّها لِـ UserWorlds فَقَط. لِلـ VMs، اِستَخدِم esxtop.
  • تَعديل /etc/ يَدَوياً: ESX يُعيد كِتابَة كَثير مِن الملَفّات عِند Reboot. اِستَخدِم esxcli دائِماً.
  • تَثبيت Tools خارِجيّة: ESX لَيس مَكاناً لِتَثبيت Monitoring Agents. اِستَخدِم Aria Operations.
BOOT ESX Boot Bank Layout — التَّخزين الدّاخِلي
🎯 لِماذا تَهتَمّ؟

عِندَما تُحَدِّث ESX، الإِصدار القَديم لا يُحذَف — يَبقى في altbootbank. هَذا يُتيح Rollback إذا فَشَل التَّحديث. لَكِن لِمَعرِفَة أَين ذَهَب كُلُّ بايت، يَجِب فَهم Boot Layout.

📝 Knowledge Check

Q1. What is ESX?

Correct: B

ESX is a bare-metal (Type-1) hypervisor — runs directly on the server hardware.

Q2. Minimum RAM required for an ESX 9.0 host (production)?

Correct: C

8 GB minimum, 12 GB for production environments.

Q3. Which time-sync method provides microsecond accuracy?

Correct: C

PTP = Precision Time Protocol — microsecond accuracy. NTP = millisecond.

Q4. Which port does NTP use?

Correct: B

NTP uses UDP port 123. PTP uses UDP 319/320.

Q5. Why must the self-signed certificate be regenerated after setting the FQDN?

Correct: B

Initial cert is generated before hostname → CN = localhost.localdomain. HTTPS validation against FQDN will fail.

Q6. What does Lockdown Mode do?

Correct: B

Lockdown Mode restricts direct host access — administration must go through vCenter.

Q7. Which is NOT one of the four Enabler-equivalent ESX security features?

Correct: D

ESX firewall is closed by default, not open.

Q8. Memory Tiering in ESX 9.0 uses what device type as a second memory tier?

Correct: B

Memory Tiering uses direct-attached NVMe PCIe devices to extend host memory.

VCSA · Photon · vpxd · SSO · VIDB · Linked Groups

المَنصّة الَّتي تُحَوِّل ESX Hosts إِلى Cloud Foundation.

2.1 لِماذا vCenter؟ | Why Central Management?

تَخَيَّل بيئة فيها 50 ESX Host وَ 1500 VM. بِدون vCenter: كُلّ Host جَزيرة، لا يُمكِنك نَقل VM بَين Hosts، لا يُمكِنك تَطبيق سياسة مُوَحَّدة، لا يُمكِنك Monitoring الكُلّ مِن مَكان واحِد. vCenter هُوَ ما يُحَوِّل Hardware مُتَفَرِّق إِلى Cloud Foundation.

ما هُوَ vCenter؟

vCenter هُوَ مَنصّة الإِدارة المَركَزيّة لِكُلّ ESX Hosts وَ VMs. بِدونه، كُلّ Host جَزيرة مَعزولة. مَعَه: تُديرها كَمَجموعة واحِدة، تُفَعِّل vMotion، HA، DRS، FT، وَ vLCM.

2.2 بِنية vCenter Server | Architecture

vCenter الآن يَأتي حَصرِيّاً كَـ Virtual Appliance (VCSA). هَذا يَعني: Photon OS + PostgreSQL + كُلّ خَدَمات vCenter، مُعَبَّأة كَـ VM واحِدة. لا يُوجَد إِصدار Windows مُنذ سَنَوات. الفائِدة: نَشر بِنَقرَة، تَحديث مُوَحَّد، وَدَعم تَقَني أَبسَط.

🧠 vCenter Server Architecture

  • Operating System: VMware Photon OS (Linux مُحَسَّن).
  • Database: مَدمَج PostgreSQL (vPostgres).
  • Form Factor: Virtual Appliance (VCSA) — يُنشَر كَـ VM على ESX.
  • Services: vpxd (المُخّ)، vAPI Endpoint، Lookup Service، Identity Management، License Service، Tagging، ESX Agent Manager، Update Manager.
  • كُلّ الخَدَمات مُعَبَّأة في appliance واحِد — نَشر مُبَسَّط، تَحديث مُوَحَّد.

2.3 الاتِّصال بَين ESX و vCenter | Communication

عِند تَسجيل ESX Host في vCenter، يَتِمّ تَنزيل وَكيل vpxa عَلى الـ Host. مِن لَحظَتها: كُلّ أَمر يَأتي مِن vSphere Client يَمُرّ في سِلسِلة vpxd → vpxa → hostd → Hardware. أَمّا الـ VMware Host Client أَو SSH فَيَتَحَدَّثان مَع hostd مُباشَرة، مُتَجاوِزَين vCenter — هَذا مُفيد لِلتَّشخيص عِند فَشل vCenter.

🧠 ESX ↔ vCenter Communication

  • vpxd — المُخّ المَركَزي عَلى vCenter.
  • vpxa — vCenter Agent عَلى كُلّ ESX Host.
  • hostd — Host Daemon عَلى كُلّ ESX Host (يُديرالـ hardware فِعليّاً).
  • التَّسَلسُل: vSphere Client ⇄ vpxd ⇄ vpxa ⇄ hostd ⇄ Hardware.
  • الـ VMware Host Client أَو SSH يَتَجاوَزان vCenter وَيَتَكَلَّمان مَع hostd مُباشَرة.

ماذا يَحدُث عِند فَصل ESX Host عَن vCenter؟

إِذا قُمت بِـ Disconnect Host أَو فُقِدَ الاتِّصال:

  • الـ VMs تَستَمِرّ في العَمَل بِشَكل طَبيعي (hostd ما زالَ يُديرها).
  • لَكِن vCenter لا يَستَطيع إِدارَتها (لا vMotion، لا Power ops، لا Snapshot).
  • الحَلّ: اتَّصِل عَبر VMware Host Client مُباشَرة لِإِدارة طارِئة.
  • لا تُعِد بَدء VMs يَدَويّاً — HA قَد يَفعَل ذَلِك إِذا كانَت العُزلة فِعليّة.

2.4 vCenter Single Sign-On | SSO

SSO هُوَ القَلب الأَمني لِكُلّ vSphere. يُعطي تَجرِبة دُخول واحِدة لِكُلّ الخَدَمات. التَّقنيّة المُستَخدَمة: SAML 2.0 — Token مُوَقَّع رَقمِيّاً يُمَرَّر بَين الخَدَمات بَدَلاً مِن إِعادة إِرسال كَلِمة المُرور. تَدعَم vsphere.local الافتِراضيّة لَكِن يَجِب تَغييرها عِند النَّشر لِأَسباب أَمنيّة.

🧠 vCenter Single Sign-On (SSO) — البِنية

  • SSO = آلِيّة المُصادَقة المَركَزيّة لِكُلّ خَدَمات vSphere.
  • المُستَخدِم يُسَجِّل دُخولاً مَرّة واحِدة → يَتَلَقّى SAML Token → يَستَخدِمه مَع كُلّ خَدمة.
  • النِّطاق الافتِراضي: vsphere.local (يَجِب تَغييره عِند النَّشر).
  • Identity Sources: Active Directory، LDAP/LDAPS، OpenLDAP، Local OS، vsphere.local.
  • Federated Identity Providers الجَدد:
    • Microsoft ADFS
    • Microsoft Entra ID (سابِقاً Azure AD)
    • Okta
    • PingFederate

SAML Token Flow — تَدَفُّق المُصادَقة

  1. المُستَخدِم يَفتَح vSphere Client.
  2. الـ Client يُحَوِّل لِـ VIDB.
  3. VIDB يُحَوِّل لِـ Identity Provider (مَثَل: Entra ID).
  4. المُستَخدِم يُدخِل بَيانات الدُّخول هُناك.
  5. الـ IdP يُصدِر SAML Token مُوَقَّعاً.
  6. الـ Token يَعود إِلى vSphere Client.
  7. الـ Client يَستَخدِم الـ Token مَع كُلّ خَدمة (vpxd، vAPI...) دون إِعادة دُخول.

2.5 VMware Identity Broker — جَديد VCF

في VCF 9.0، تَمَّ تَوحيد المُصادَقة عَبر طَبَقة جَديدة اسمها VMware Identity Broker (VIDB). هِيَ المُتَرجِم بَين vCenter (وَكُلّ مُكَوِّنات VCF) وَالـ Identity Providers الخارِجيّة. يَدعَم البُروتوكولات الحَديثة: SAML 2.0 + OIDC. هَذا يَفتَح الباب لِـ Microsoft Entra ID، Okta، PingFederate، وَ ADFS كَ-IdPs مُوَحَّدة لِكامِل بيئة VCF.

🧠 VMware Identity Broker (VIDB)

  • VIDB = الوَسيط الجَديد بَين vCenter وَالـ Identity Providers الخارِجيّة.
  • يَدعَم بُروتوكولات حَديثة: SAML 2.0 + OpenID Connect (OIDC).
  • السَّبَب: تَوحيد المُصادَقة عَبر كُلّ مَنصّات VCF (Operations, Automation, NSX, Identity).
  • يَستَخدِم Token-based authentication بَدَلاً مِن إِرسال كَلِمات المُرور.

2.6 Enhanced Linked Mode — مَهجور | Deprecated

ELM كانَ الحَلّ التَّقليدي لِإِدارة عِدّة vCenter instances كَوَحدة واحِدة (حَتّى 15). في VCF 9.0، الفِكرة تَطَوَّرت: بَدَلاً مِن إِدارة vCenters كَ-Peers، يُدار كُلّ شَيء مِن VCF Operations Console عَبر vCenter Linked Groups. النَّتيجة: نَفس القُدرة لَكِن بِطَريقة أَنظَف وَأَكثَر تَكامُلاً مَع باقي VCF.

Enhanced Linked Mode — مَهجور في 9.0!

Enhanced Linked Mode (ELM) سَمَح تاريخيّاً بِإِدارة حَتّى 15 vCenter instances مِن جَلسة واحِدة، مَع مَزامَنة Tags وَ Roles وَ Licenses بَينَهم.

  • في VCF 9.0: ELM مَهجور (Deprecated) لَكِن لا يَزال مَدعوماً مُؤَقَّتاً لِلانتِقال.
  • البَديل الجَديد: vCenter Linked Groups عَبر VCF Operations Console.
  • هَذا يَتَوافَق مَع نَهج VCF المُوَحَّد: كُلّ شَيء يُدار مِن مَكان واحِد.

2.7 المَخزون | Inventory Views

عِند فَتح vSphere Client، تَجِد أَربَع طُرُق لِعَرض نَفس البَيانات. كُلّ مُشاهَدة تُرَكِّز على بُعد مُختَلِف: الـ hardware، التَّنظيم المَنطِقي لِلـ VMs، الـ storage، أَو الـ network. اِختَر المُشاهَدة المُناسِبة حَسَب المَهَمّة. مَثَلاً: لِنَقل VM إِلى Cluster آخَر، استَخدِم Hosts and Clusters. لِتَنظيم VMs حَسَب الفَريق المالِك، استَخدِم VMs and Templates.

🧠 vCenter Inventory — أَربَع مُشاهَدات

  • Hosts and Clusters — التَّنظيم بِالـ hardware المادّي وَالـ Clusters.
  • VMs and Templates — التَّنظيم المَنطِقي لِلـ VMs وَالقَوالِب.
  • Storage — Datastores وَ Storage Pods.
  • Networks — Standard / Distributed Switches، Port Groups، Distributed Port Groups.

🧠 Inventory Objects — الكيانات

  • Datacenter — الحاوية الرَّئيسيّة (يَجِب أَن تَكون أَوَّل ما تُنشِئ).
  • Folder — تَنظيم مَنطِقي (يَدعَم Permissions Inheritance).
  • Cluster — مَجموعة ESX Hosts تَتَشارَك Resources.
  • Host — ESX Server فِعلي.
  • VM / Template / vApp — الـ Workloads.
  • Resource Pool — تَجزِئة CPU/Memory هَرَميّاً.
  • Datastore — مَخزون لِلـ Files.
  • Network — Port Group أَو Distributed Port Group.

في VCF 9.0، إِدارة Datacenters، Folders، وَ Custom Tags تُمَركَز في VCF Operations بَدَلاً مِن كُلّ vCenter على حِدة.

SSO vCenter Single Sign-On — الهَويّة المَركَزيّة
🎯 الهَدَف

إِدارة هَويّة مُوَحَّدة لِكُلِّ خَدَمات VMware (vCenter، NSX، Aria…). تَسجيل دُخول مَرَّة واحِدة، صَلاحيّات في كُلِّ مَكان.

🏗 المُكَوِّنات
  • SSO Domain: الافتِراضي vsphere.local (تَستَطيع تَغييرُه عِند Deploy).
  • STS (Security Token Service): يُصدِر SAML tokens.
  • VMware Directory Service (VMDir): LDAP داخِلي لِـ users المَحَلّيّين.
  • Identity Sources: مَصادِر مُمكِنة — vsphere.local (default)، AD، LDAP، OpenLDAP، AD over LDAPS، Identity Federation (OIDC).
⚙ كَيف يَعمَل تَسجيل الدُّخول
  1. User يُدخِل username/password في vSphere Client.
  2. SSO يَتَحَقَّق مَع Identity Source المُناسِب (AD مَثَلاً).
  3. إذا صَحيح، STS يُصدِر SAML Token.
  4. Token يُستَخدَم لِكُلِّ تَفاعُل مَع vCenter وَ NSX وَ Aria.
  5. Token صالِح لِمُدَّة مُحَدَّدة (افتِراضي 8 ساعات).
⚠ أَخطاء شائِعة
  • الاِعتِماد على administrator@vsphere.local: لا تَستَخدِمه لِلعَمَل اليَومي. اِربِط AD، اِعطِ مُستَخدِمين شَخصيّين صَلاحيّات.
  • عَدَم تَكوين Time Sync: SSO حَسّاس لِلوَقت. Time skew > 5 دَقائق يَكسِر الـ Authentication.
  • تَجاهُل Password Policy: vCenter Server Appliance لَدَيها password expiry. إذا اِنتَهَت كَلِمة سِرّ root، يُمكِن أَن تُحبَس خارِج VAMI.
PERM Permissions Model — Tags · Roles · Privileges
🎯 المَفهوم

vCenter يَستَخدِم نَموذَج (Object، Role، Principal):

  • Object: Datacenter، Cluster، Host، VM، Folder، Network…
  • Role: مَجموعة مِن Privileges (مَثَلاً "VM Power User" يَسمَح بِـ Power On/Off).
  • Principal: user أَو group مِن Identity Source.
⚙ كَيف تُحَدِّد صَلاحيّات
  1. اِنتَقِل إلى Object (مَثَلاً Folder VMs/Production).
  2. Permissions tab → Add Permission.
  3. اِختَر Principal (مَثَلاً DOMAIN\App-Owners-Group).
  4. اِختَر Role (مَثَلاً "VM Operator").
  5. اِختَر "Propagate to children" إذا تُريد التَّوريث.
💡 مَثال عَمَلي

السِّياسة: «فَريق DevOps يَستَطيع إِنشاء VMs في Folder Dev، لَكِن لا يَستَطيع لَمس Production.»

  1. Create Role: "DevOps Operator" مَع privileges (VM Create، Power، Snapshot).
  2. Folder /Dev → Add Permission: DOMAIN\DevOps-Team مَع Role "DevOps Operator"، Propagate ✓.
  3. Folder /Production → لا permission لِـ DevOps-Team.
  4. النَّتيجة: مُستَخدِم DevOps لا يَرى /Production أَصلاً.
⚠ أَخطاء شائِعة
  • Admin-by-Default: «أَعطِيه Administrator، أَسهَل». لا. اِستَخدِم Least Privilege.
  • Permission على Datacenter بَدَلاً مِن Folder: يُعطي وُصول لِكُلِّ شَيء تَحت Datacenter. اِفصِل بِـ Folders.
  • عَدَم Audit: راجِع Permissions كُلَّ رُبع سَنَة. مُغادِرون يَجِب إِزالتهم.

  • لا Documentation لِـ root password: Password lost = VCSA rebuild. اِحفَظه في Vault.

Module 03 · vSphere Networking

Standard & Distributed Switches · VMkernel

vSS · vDS · Port Groups · Uplinks · VLAN · Teaming

الشَّبَكة الَّتي تَربِط VMs بِالعالَم.

3.1 لِماذا Virtual Networking؟ | Why Virtual Networking?

الـ VM تَحتاج NIC. لَكِن لا يُمكِنك تَوصيل كُلّ VM بِـ NIC فِعلي مُنفَصِل — هَذا غَير عَمَلي. الحَلّ: Virtual Switch داخِل ESX. هَذا الـ Switch يَعمَل في VMkernel وَيُقَدِّم خَدَمات L2 (Forwarding، VLAN، Filtering) كَأَنَّه Switch فِعلي. الـ VMs تَتَّصِل بِه بِشَكل Virtual، وَهُوَ يَتَّصِل بِالخارِج عَبر Uplinks (vmnics فِعليّة).

لِماذا تَفهَم vSphere Networking؟

الشَّبَكة هِيَ ما يَربِط VMs بِالعالَم. أَيّ خَطَأ في الشَّبَكة يَعني VMs مَعزولة، vMotion فاشِل، HA غَير مُتَوَقَّع، وَ vSAN مَكسور. هَذِه الوِحدة تُعَلِّمك كَيف تُشَكِّل الـ Virtual Switches بِشَكل صَحيح.

3.2 Standard vs Distributed Switch

هُناك نَوعان مِن Virtual Switches: Standard (vSS) الَّذي يَعمَل عَلى Host واحِد فَقَط، وَ Distributed (vDS) الَّذي يَمتَدّ عَبر كُلّ Hosts الـ Datacenter (حَتّى 2000 Host). الـ vSS مَجاني وَبَسيط، لَكِنَّه يُجبِرك على تَكرار التَّهيئة عَلى كُلّ Host. الـ vDS مَركَزي، يُعطي ميزات مُتَقَدِّمة، وَهُوَ إِلزامي لِـ NSX وَ VCF.

في VCF: NSX إِلزامي، وَ NSX يَتَطَلَّب vDS. بِالإِضافة لِذَلِك، vSAN وَ vMotion يَستَفيدان مِن Network I/O Control لِضَمان جَودة الخِدمة. لِذَلِك في كُلّ بيئة VCF: Distributed Switch هُوَ الأَساس، وَ Standard Switch فَقَط لِسيناريوهات اِستِثنائيّة.

3.3 بِنية Distributed Switch | Architecture

الـ vDS مَبني على فِكرة الفَصل بَين Control Plane (في vCenter) وَ I/O Plane (في VMkernel عَلى كُلّ Host). فائِدة هَذا الفَصل: حَتّى لَو سَقَط vCenter، الشَّبَكة تَستَمِرّ في العَمَل لِأَنَّ كُلّ Host يَحمِل نُسخة مِن السِّياسة في VMkernel. التَّعديلات فَقَط تَتَطَلَّب vCenter.

🧠 Distributed Switch Architecture

  • Control Plane — يَعمَل في vCenter، يَحتَفِظ بِالتَّهيئة المُوَحَّدة.
  • I/O Plane — يَعمَل في VMkernel عَلى كُلّ Host (vSwitch مَخفي).
  • عِند فَقد vCenter: الـ I/O Plane يَستَمِرّ في تَمرير الحَزَم — لا تَنقَطِع الشَّبَكة.
  • التَّعديلات تَأتي مِن vCenter فَقَط، ثُمَّ تُنشَر إِلى كُلّ Host.
  • Uplinks = أَسماء مَنطِقيّة (Uplink1, Uplink2…) تُرَبَط بِـ vmnic فِعليّة عَلى كُلّ Host.

3.4 أَنواع الاتِّصال | Connection Types

كُلّ Switch (سَواء vSS أَو vDS) يُقَدِّم ثَلاثة أَنواع مِن المَنافِذ:

  1. VM Port Groups — لِلـ VMs ذاتها.
  2. VMkernel Ports — لِخَدَمات ESX الدّاخِليّة.
  3. Uplinks — لِرَبط الـ Switch بِـ vmnic فِعليّة على Host.

🧠 Connection Types — أَنواع الاتِّصال

  • Virtual Machine Port Groups — لِاتِّصال VMs بِالشَّبَكة.
  • VMkernel Ports — لِخَدَمات ESX نَفسه (Management, vMotion, vSAN…).
  • Uplink Ports / Adapters — لِرَبط الـ Switch بِـ vmnic فِعليّة.

3.5 خَدَمات VMkernel | VMkernel Services

الـ VMkernel Port لا يَحمِل حَركة VM — يَحمِل حَركة ESX نَفسه. كُلّ خِدمة تَحتاج VMkernel Port مُخَصَّص (أَو مُشتَرَك إِذا كانَ التَّخطيط يَسمَح). الفَصل المَنطِقي بَين الخَدَمات أَفضَل مُمارَسة: Management على VLAN، vMotion على VLAN آخَر، vSAN على VLAN ثالِث.

3.6 VLAN Tagging — VST

كَيف يَنفَصِل VM A عَلى VLAN 10 عَن VM B عَلى VLAN 20 وَهُما عَلى نَفس الـ vmnic؟ الجَواب: vSwitch يَضَع Tag (802.1Q) على كُلّ إِطار خارِج، وَيُزيله عِند الدُّخول. هَذا اسمه Virtual Switch Tagging (VST). شَرطه: المَنفَذ الفِعلي على Switch خارِجي يَجِب أَن يَكون Trunk Port يَسمَح لِكُلّ الـ VLANs المَطلوبة بِالمُرور.

🧠 VLAN Tagging — Virtual Switch Tagging (VST)

  • الـ Physical Port عَلى الـ Switch الفِعلي يَجِب أَن يَكون Trunk Port.
  • الـ Port Group في vSphere يُحَدَّد لَه VLAN ID.
  • الـ vSwitch يَقوم بِالـ Tagging/Untagging — الـ VM ولا تَدري شَيئاً.
  • هَذا الأُسلوب اسمه Virtual Switch Tagging (VST) — الأَكثَر شُيوعاً.
  • أَنماط أُخرى أَقَلّ شُيوعاً: EST (External Switch Tagging) وَ VGT (Virtual Guest Tagging).

MTU — حَجم الإِطار

  • الافتِراضي: 1500 Bytes.
  • لِـ vMotion وَ vSAN: يُنصَح بِـ Jumbo Frames (MTU 9000).
  • القاعِدة: MTU يَجِب أَن يَتَطابَق بَين VM ↔ Port Group ↔ vSwitch ↔ Uplink ↔ Physical Switch. أَيّ عَدَم تَطابُق = Packet Drop.

3.7 NIC Teaming & Failover | تَجميع NICs

لِلتَّوافُريّة وَالأَداء، تُجَمَّع عِدّة vmnics كَـ Uplinks لِنَفس الـ vSwitch. السِّياسة تُحَدِّد كَيف تُوَزَّع الحَركة بَيْنَها. الأَكثَر استِخداماً: Originating Virtual Port — كُلّ VM Port يُرتَبَط بِـ Uplink مُحَدَّد. لِأَداء أَحسَن: Physical NIC Load (vDS فَقَط) — يُوازِن حَسَب الحُمل الفِعلي. لِـ LACP: استَخدِم IP Hash مَع EtherChannel على Switch الفِعلي.

🧠 NIC Teaming — تَجميع البِطاقات

  • Route based on Originating Virtual Port — الافتِراضي، بَسيط وَفَعّال.
  • Route based on Source MAC Hash
  • Route based on IP Hash — يَتَطَلَّب EtherChannel/LAG عَلى Switch الفِعلي.
  • Route based on Physical NIC Load — Distributed Switch فَقَط، يُوازِن حَسَب التَّحميل الفِعلي.
  • Use Explicit Failover Order — Active/Standby مُحَدَّد يَدَويّاً.

Failback: عِند رُجوع uplink فَعّال، هَل يَستَردّ الحَركة؟ (Yes/No).

📝 Knowledge Check

Q1. What is a key advantage of a Distributed Switch over a Standard Switch?

Correct: B

vDS is configured in vCenter and pushed to all member hosts — central management.

Q2. Maximum number of hosts a single Distributed Switch can span?

Correct: C

vDS supports up to 2000 hosts.

Q3. Which feature is available ONLY on Distributed Switch?

Correct: C

LACP, NIOC, NetFlow, port mirroring, inbound shaping — all vDS-only.

Q4. Which port type connects VMs to the network?

Correct: C

VM port groups carry VM traffic. VMkernel = host services. Uplinks = physical NICs.

Q5. Which VMkernel service is used for live migration of VMs?

Correct: C

vMotion service handles live migration of running VMs between hosts.

Q6. What is the default MTU?

Correct: B

Default MTU is 1500. Jumbo Frames (9000) recommended for vMotion/vSAN.

Q7. In Virtual Switch Tagging (VST), the physical switch port must be:

Correct: B

VST requires trunk on the physical switch so multiple VLAN tags can flow to the vSwitch.

Q8. What happens to the network if vCenter is offline (with Distributed Switch)?

Correct: B

vDS I/O plane runs in VMkernel locally; vCenter only manages the control plane.

🧠 Virtual NICs — أَنواع البِطاقات

  • E1000 / E1000E — مُحاكاة Intel Gigabit، الافتِراضي لِـ Windows.
  • VMXNET3 — Paravirtualized، أَفضَل أَداء، يَتَطَلَّب VMware Tools.
  • PVRDMA — Virtual RDMA لِتَطبيقات HPC.
  • SR-IOV Passthrough — تَجاوُز الـ Hypervisor، أَعلى أَداء، أَقَلّ مَرونة.

مَهجور في ESX 9.0: Flexible، VLANCE، VMXNET2. الافتِراضي اليَوم: VMXNET3 لِكُلّ ما يَدعَم VMware Tools.

4.6 VMware Tools — أَهَمّيّتها

VMware Tools لَيس اختياريّاً. هِيَ المَجموعة الَّتي تُحَوِّل VM «بَدائيّة» إِلى VM ذَكِيّة: تُقَدِّم Drivers مُحَسَّنة، تُزامِن الوَقت، تُرسِل Heartbeat لِـ HA، وَتَدعَم Quiesce لِلـ Snapshots المُتَّسِقة. تَحديثها بِشَكل مُنتَظِم أَفضَل مُمارَسة.

🧠 VMware Tools — لِماذا إِلزامي

  • Time Synchronization مَع الـ Host.
  • Heartbeat — يُخبِر vCenter أَنَّ الـ Guest حَيّ (لِـ HA VM Monitoring).
  • Quiescing — تَجميد نِظام المَلَفّات لِـ Snapshots/Backups مُتَّسِقة.
  • Customization — استِقبال Customization Specs.

4.7 القَوالِب | Templates

الـ Template هُوَ VM «مُتَجَمِّد»: لا يُمكِن تَشغيله، لَكِنَّه أَساس لِنَشر VMs مُتَطابِقة. تُحَوِّل VM ذَهَبي إِلى Template، ثُمَّ تَنشُر مِنه أَيّ عَدَد. عِند الحاجة لِتَحديث الـ Template: حَوِّله إِلى VM مَرّة أُخرى، طَبِّق التَّحديثات، ثُمَّ أَعِده Template.

🧠 Templates — القَوالِب

  • الـ Template = VM مُجَمَّد، لا يُمكِن تَشغيله، يُستَخدَم لِنَشر VMs مُتَطابِقة.
  • الـ .vmx يَتَحَوَّل إِلى .vmtx.
  • طُرُق الإِنشاء: Convert to Template أَو Clone to Template.
  • عِند Clone-to-Template، تَختار: Eager / Lazy / Thin / Same as Source.
  • لِلتَّعديل: حَوِّل إِلى VM، عَدِّل، ثُمَّ أَعِده Template.

🧠 Customization Specifications

  • تُطَبَّق عِند Deploy from Template.
  • تَهيئ:
    • Computer Name (Hostname)
    • Network (DHCP أَو Static + DNS)
    • License Info
    • Time Zone
    • Administrator Password
    • Windows-only: SID جَديد + Domain Join (Workgroup أَو OU)
    • Linux: يَستَخدِم cloud-init الحَديث.

🧠 Content Libraries — مَكتَبات المُحتَوى

  • Published Library — Local Library تُتاح لِلاستِهلاك مِن خارِج.
  • Subscribed Library — تَستَهلِك مِن Published Library في vCenter آخَر.

قُيود مُهِمّة:

  • الـ Subscribed Library لا تُمكِن نَشرها مَرّة أُخرى (لا تَتَحَوَّل لِـ Published).
  • الـ Subscribed Library لا يُمكِن تَعديلها — تَحديث تِلقائي مِن المَصدَر فَقَط.

4.9 Snapshots — المَفهوم وَالاستِخدام

الـ Snapshot يَلتَقِط حالة VM في لَحظة. مُفيد قَبل تَطبيق Patch، Upgrade، أَو تَغيير حَرِج: لَو فَشِل، تَرجِع إِلى الحالة السّابِقة بِنَقرة. لَكِن — وَنُكَرِّر — لَيس Backup. الـ Snapshots تَعيش مَع الـ VM عَلى نَفس الـ Datastore.

تَنبيه — Snapshots لَيست Backup!

الـ Snapshot تَلتَقِط الحالة الحاليّة لِـ VM (Disk + اختِياريّاً Memory) لِغَرَض اختِبار تَغيير، Patch، أَو Upgrade. لَيست بَديلاً عَن الـ Backup. لِأَنَّها:

  • تُخَزَّن مَع الـ VM ذاتها — فَقد الـ disk = فَقد كُلّ شَيء.
  • تُؤَثِّر سَلباً عَلى الأَداء بِمُرور الوَقت (سِلسِلة Delta Disks).
  • غَير مُصَمَّمة لِلاحتِفاظ الطَّويل (لا تَزيد على 72 ساعة كَقاعِدة عامّة).

  • SEsparse — VMFS6 الافتِراضي + VMFS5 ≥ 2TB، Block 4KB، يَدعَم unmap (Space Reclamation).
  • vsanSparse — vSAN Original Storage Architecture (OSA)، Block 4MB.
  • vSAN ESA: الـ Snapshot يُخَزَّن في نَفس الـ Object — لَيس مَلَفّاً مُنفَصِلاً.

🧠 Snapshot Operations

  • Take Snapshot — مَع/بِدون الـ memory، مَع/بِدون Quiesce (Quiesce فَقَط بِدون memory).
  • Edit / Rename — تَغيير الاسم وَالوَصف.
  • Delete — دَمج الـ Delta مَع Parent (تَحرير مِساحة).
  • Delete All — دَمج كُلّ الـ Snapshots.
  • Revert — إِرجاع الـ VM إِلى حالة Snapshot مُحَدَّد.

4.10 Backup | Backup with VADP

الـ vSphere Storage APIs - Data Protection (VADP) تُقَدِّم نَموذج Backup الحَديث: مَنتَج Backup يَستَخدِم Backup Proxy لِيَقرَأ VMDK مُباشَرة (LAN-free) عَبر SAN Transport أَو SCSI HotAdd. الـ Changed Block Tracking (CBT) يُسَرِّع Incremental Backups بِشَكل دِراماتيكي. أَيّ مَنتَج Backup حَديث (Veeam، Avamar، Commvault، Rubrik...) يَستَخدِم VADP.

🧠 vSphere Backup — VADP

  • vSphere Storage APIs - Data Protection (VADP) — مَجموعة APIs مُقَدَّمة مُنذ vSphere 4.0.
  • السَّبَب: نَقل عِبء النَّسخ عَن الـ ESX Host إِلى Backup Proxy Server.
  • الفائِدة:
    • LAN-free backups (مَع SAN transport).
    • Off-host processing — لا تَأثير على الـ Production VM.
    • Changed Block Tracking (CBT) — incremental backups سَريعة.

🧠 VADP Transport Methods

  • SAN Transport — مُباشَرة عَبر Fibre Channel/iSCSI، الأَسرَع.
  • SCSI HotAdd — يُلحَق الـ disk بِـ Backup Proxy كَـ VM، سَريع وَلا يَحتاج SAN.
  • NBD over LAN — Network Block Device عَبر الشَّبَكة (بَطيء نِسبيّاً).
  • NBDSSL — NBD مَع SSL (آمَن لَكِن أَبطَأ).
  • Local File Access — لِـ Backup proxy على نَفس الـ Host.

إِدارة VMs الجَيِّدة لَيست مُجَرَّد Power On/Off — بَل فَهم كَيف تُخَزَّن، تُنسَخ، تُعاد إِنشاءها بِكَفاءة.

SNAP VM Snapshots — صَديق أَم عَدُوّ؟
🎯 المَفهوم الحَقيقي

Snapshot لَيس Backup. هُوَ Point-in-Time Marker + Delta Disk. كُلُّ كِتابَة بَعد Snapshot تَذهَب إلى Delta File، لَيس إلى Base Disk.

⚙ كَيف يَعمَل
  1. قَبل Snapshot: VM يَكتُب إلى vm.vmdk.
  2. عِند Snapshot: VM يَتَوَقَّف لِجُزء ثانية. vm-000001.vmdk (delta) يُنشَأ.
  3. بَعد Snapshot: كُلُّ كِتابَة تَذهَب إلى delta. Base يَبقى مُجَمَّداً.
  4. Snapshot ثانٍ: vm-000002.vmdk. كُلُّ كِتابَة تَذهَب إلَيه.
  5. عِند Read: VM يَقرَأ مِن أَحدَث delta، إذا لَم يَجِد، يَنزِل إلى أَقدَم، إلى Base.
  6. عِند Consolidate: تُدمَج deltas في Base.
⚠ المَخاطِر
  • Snapshot قَديم على VM نَشِط: Delta يَكبُر بِسُرعة. VM 100 GB قَد تُصبِح 500 GB في يَوم.
  • Snapshot Chain طَويلة: أَكثَر مِن 3 deltas = أَداء سَيِّء، Consolidation طَويلة.
  • Forgotten Snapshots: Backup tools تَأخُذ snapshot، تَنسى حَذفه. اِفحَص أُسبوعِيّاً.
  • Snapshot كَـ Backup: لا. Snapshot يَعتَمِد على Base disk. Datastore فَشَل = كُلُّ شَيء يَضيع.
💡 السِّياسة المُوصَى بِها
  • أَقصى عُمر snapshot: 24 ساعة.
  • أَقصى عَدَد snapshots نَشِطة: 2.
  • Snapshots لِـ Upgrade/Patch فَقَط، وَتُحذَف فَوراً بَعد التَّأكيد.
  • Quiesce (مَع VMware Tools) لِـ Application Consistency. بِدونها = Crash-consistent فَقَط.
TMPL Templates & Content Library — التَّوحيد
🎯 المَفهوم

Template هُوَ VM Image جاهِزة لِلنَّسخ. Content Library هُوَ مَستودَع لِـ Templates وَ ISOs، يُتاح عَبر عِدّة vCenters.

⚙ Workflow Recommended
  1. اِبنِ Master VM مَع OS + Tools + Hardening.
  2. Generalize (sysprep لِـ Windows، cloud-init لِـ Linux).
  3. Convert to Template (تَفقُد القُدرة على Power On).
  4. اِنشُر إلى Content Library — يَتَزامَن لِكُلِّ Site.
  5. Customization Specification لِتَخصيص hostname، IP، domain join تِلقائيّاً.
  6. Deploy from Template + Customization = VM جاهِزة في 5 دَقائق.
📤 الفَوائِد
  • Consistency كُلُّ VM تُبدَأ بِنَفس النُّقطة.
  • Speed VM في 5 دَقائق بَدَلاً مِن 2 ساعة.
  • Compliance Hardening مَركَزي.
  • Patching Update template مَرَّة، VMs جَديدة patched.
⚠ أَخطاء شائِعة
  • Template قَديم: Update شَهرِيّاً عَلى الأَقَلّ. اِبنِ Pipeline لِتَوليد templates جَديدة.
  • VM Owner-Specific في Template: Hostname، SSH keys، application-specific — كُلُّها تَجِب أَن تُحذَف.
  • Skipping sysprep: Windows VMs بِنَفس SID = AD issues.

5.1 Cluster — اللَّبِنة الأَساسيّة | The Foundation

الـ Cluster هُوَ مَجموعة Hosts تُدار كَوَحدة واحِدة. عِند انضِمام Host لِـ Cluster، يَحصُل عَلى ميزات لا يَستَطيع تَوفيرها وَحده: HA لِلتَّوافُريّة، DRS لِلتَّوازُن، vMotion لِلنَّقل الحَيّ، vSAN لِلتَّخزين الموزَّع، FT لِصِفر انقِطاع. في vSphere 9.0، فَلسَفة Cluster تَطَوَّرت: كُلّ Cluster يُدار بِـ Image واحِد (لا Baselines)، وَهَذا إِلزامي عِند الإِنشاء.

لِماذا Clusters؟

ESX Host واحِد = نُقطة فَشَل واحِدة. Cluster = مَجموعة Hosts تَتَشارَك Resources وَتُوَفِّر تَوافُريّة عالِية. عِند انضِمام Hosts لِـ Cluster، تُصبِح ميزات قَويّة مُتاحة: vMotion، DRS، HA، FT، وَ vSAN. هَذِه الوِحدة تُغَطِّيها كُلَّها.

🧠 Cluster Basics — VCF 9.0

  • الحَدّ الأَقصى: 96 ESX hosts لِكُلّ Cluster.
  • مَع vSAN: الحَدّ 64 hosts.
  • الميزات المُتاحة: DRS · HA · vSAN · vMotion · FT · Cluster Image.
  • في vSphere 9.0: تَعيين Cluster Image إِلزامي عِند إِنشاء Cluster.
  • الـ Baselines القَديمة لَم تَعُد مَدعومة — كُلّ Cluster يَجِب أَن يُدار بِـ Image.

5.2 vMotion — نَقل VMs الحَيّ | Live Migration

vMotion هُوَ التَّقنيّة الأَيقونيّة في VMware: نَقل VM شَغّالة مِن Host إِلى آخَر بِدون انقِطاع لِلخِدمة، وَ User لا يَلاحِظ شَيئاً. هَذِه القُدرة هِيَ ما يَجعَل Maintenance الـ hardware، تَوازُن Resources، وَإِيقاف Hosts لِتَوفير الطّاقة، عَمَلِيّة بِلا تَأثير.

🧠 vMotion Types — أَنواع النَّقل

  • Cold Migration — الـ VM مُطفَأة. أَبسَط نَوع.
  • Hot Migration (vMotion) — الـ VM شَغّالة. الأَنواع:
    • Compute-only vMotion — تَغيير الـ Host، نَفس الـ Datastore.
    • Storage vMotion — تَغيير الـ Datastore، نَفس الـ Host.
    • Both — تَغيير Host وَ Datastore مَعاً.
    • Cross-vCenter vMotion — نَقل بَين vCenters مُختَلِفة (نَفس SSO).
    • vGPU Live Migration — جَديد، يَدعَم نَقل VMs بِـ vGPU بِدون انقِطاع.

vMotion Performance — VCF 9.0

تَحسينات في vCenter 9.0:

  • Parallel TCP streams — استِخدام عِدّة اتِّصالات بِالتَّوازي.
  • Zero-copy techniques — تَقليل نَسخ البَيانات في الـ memory.
  • Autoscaling precopy — تَكييف ديناميكي مَع التَّحميل.
  • النَّتيجة: نَقل أَسرَع مِن GPU-attached VMs.

5.3 vMotion CPU Constraints & EVC

أَكبَر سَبَب لِفَشَل vMotion هُوَ عَدَم تَوافُق CPUs. الـ Guest OS داخِل الـ VM يَعتَمِد عَلى تَعليمات CPU مُحَدَّدة (SSE، AVX، إِلخ). إِذا انتَقَلَت لِـ Host بِـ CPU أَقدَم لا يَدعَم تِلك التَّعليمات، تَتَعَطَّل. الحَلّ التاريخي: EVC الَّذي يَكشِف فَقَط التَّعليمات المُشتَرَكة. VCF 9.0 أَضافَ Custom EVC لِتَحَكُّم أَدَقّ، وَ EVC for vSGA لِتَوحيد ميزات GPUs.

🧠 vMotion CPU Constraints

  • الشَّرط الأَهَمّ: الـ Source وَ Destination Hosts يَجِب أَن يَكون لَهُما:
    • نَفس Vendor الـ processor (Intel أَو AMD — لا يُمكِن المَزج).
    • نَفس عائِلة الـ processor.
  • تَعليمات SSE3 / SSSE3 / SSE4.1 يَجِب أَن تَتَطابَق.
  • إِذا لَم تَتَطابَق، يَفشَل vMotion أَو يُمنَع.

🧠 EVC — Enhanced vMotion Compatibility

  • المُشكِلة: Hosts بِأَجيال مُختَلِفة مِن نَفس الـ processor لَدَيها مَجموعات تَعليمات مُختَلِفة.
  • الحَلّ: EVC يُنشِئ Baseline CPU mask يَكشِف فَقَط التَّعليمات المُشتَرَكة لِكُلّ VMs.
  • هَذا يَسمَح بِـ vMotion بَين أَجيال مُختَلِفة.
  • Custom EVC — تَحَكُّم دَقيق في تَفعيل/تَعطيل ميزات بِعَينها.
  • EVC for vSGA — جَديد لِتَوحيد عَرض ميزات GPUs بَين Hosts.

5.4 DRS — تَوزيع Resources | Distributed Resource Scheduler

الـ DRS هُوَ «المُدير الذَّكي» لِلـ Cluster: يَتَّخِذ قَرارات Initial Placement عِند تَشغيل VM، وَ Load Balancing بَين Hosts كُلّ دَقيقة، وَيُسَهِّل Maintenance Mode بِنَقل VMs بِشَكل تِلقائي. المُستَوى الافتِراضي: Fully Automated + Migration Threshold L3.

🧠 DRS — Distributed Resource Scheduler

  • الوَظيفة:
    1. Initial Placement — اختِيار Host عِند تَشغيل VM.
    2. Load Balancing — نَقل VMs لِتَحقيق تَوازُن.
    3. Maintenance Mode — إِخلاء Host عِند Maintenance.
  • VM-focused: يَعمَل عَلى تَسجيل (Score) لِكُلّ VM كُلّ دَقيقة.
  • المُتَطَلَّبات: شَبَكة vMotion + Shared Storage.

🧠 DRS Automation Levels

Migration Threshold (1–5):

  • L1 Conservative — فَقَط الضَّروري (مَثَل Maintenance Mode).
  • L3 (الافتِراضي) — مُتَوازِن.
  • L5 Aggressive — تَوازُن مُكَثَّف، تَوصيات حَتّى لِلتَّحسينات الطَّفيفة.

🧠 Affinity & Anti-Affinity Rules

  • VM-VM Affinity — VMs مُحَدَّدة عَلى نَفس الـ Host (مَثَل: App + DB).
  • VM-VM Anti-Affinity — VMs لا تَكون عَلى نَفس الـ Host (مَثَل: Domain Controllers).
  • VM-Host Affinity — VMs على Hosts مُحَدَّدة (Should/Must).
  • VM-Host Anti-Affinity — VMs لا تَكون عَلى Hosts مُحَدَّدة (مَثَل: Licensing).

5.5 vSphere HA — التَّوافُريّة العالِية

HA يَحمي مِن فَشَل Host: عِند سُقوط Host، VMs الَّتي كانَت عَلَيه يُعاد تَشغيلها على Hosts أُخرى في الـ Cluster. الفِكرة بَسيطة لَكِن التَّنفيذ مُعَقَّد: كَيف نَعرِف أَنَّ Host سَقَط فِعلاً وَلَيس مَعزولاً عَن الشَّبَكة؟ الإِجابة: Network Heartbeats + Datastore Heartbeats مَعاً.

🧠 vSphere HA — High Availability

  • الفِكرة: عِند فَشَل Host، إِعادة تَشغيل VMs الَّتي عَلَيه عَلى Hosts أُخرى تِلقائيّاً.
  • الوَكيل: FDM (Fault Domain Manager).
  • VCF 9.0: FDM أَصبَح جُزءاً مِن ESX Default Image (سابِقاً كانَ يُدفَع).
  • المُكَوِّنات: Primary + Secondary FDM nodes.

🧠 FDM Election & Heartbeats

  • الانتِخاب يَستَغرِق ~ 15 ثانِية عِند تَفعيل HA أَو سُقوط Primary.
  • Primary Selection: الـ Host الَّذي يَصِل إِلى أَكبَر عَدَد مِن Datastores.
  • عِند التَّعادُل: أَصغَر MOID (Managed Object ID).
  • الـ Heartbeats:
    • UDP أَثناء الانتِخاب (سَريع، خَفيف).
    • TCP آمَن بَعد الانتِخاب (مَوثوق، مُشَفَّر).
  • Network + Datastore Heartbeats مَعاً يُمَيِّزان بَين:
    • Failure (الـ Host سَقَط).
    • Partition (الشَّبَكة مُجَزَّأة).
    • Isolation (الـ Host مَعزول لَكِن شَغّال).

🧠 VM Component Protection (VMCP)

  • تَحمي مِن أَعطال Storage Path (Permanent Device Loss أَو All Paths Down).
  • عِند الكَشف، تَنقُل VMs المُتَأَثِّرة لِـ Host آخَر يَصِل لِلـ Datastore.
  • إِعدادات لِكُلّ نَوع PDL/APD: Disabled, Issue Events, Power Off & Restart.

5.6 Admission Control — حَجز Resources لِلـ Failover

إِذا فَشِل Host، أَين تَذهَب VMs؟ يَجِب أَن تَكون هُناك Resources كافِية في باقي Hosts. Admission Control هُوَ الآلِيّة الَّتي تَضمَن ذَلِك: تَمنَع تَشغيل VMs جَديدة إِذا كانَ ذَلِك سَيُخِلّ بِقُدرة الـ Cluster عَلى احتِواء فَشَل Host. الافتِراضي: Cluster Resource Percentage.

🧠 Admission Control — 4 Settings

  • Disabled — لا قُيود (غَير مُوصى).
  • Slot Policy — حِساب Slots بِناءً على أَكبَر VM Reservation.
  • Cluster Resource Percentage (الافتِراضي) — حَجز نِسبة % مِن CPU/Memory لِلـ Failover.
  • Dedicated Failover Hosts — Hosts مُخَصَّصة لا تَستَقبِل VMs إِلّا عِند الفَشَل.

HA + DRS Integration

عِند فَشَل Host:

  1. HA يَتَحَقَّق مِن Resources المُتَوَفِّرة في باقي Hosts.
  2. إِذا كانَت Resources كافِية → يَبدَأ Restart.
  3. إِذا لَم تَكُن كافِية → يَطلُب مِن DRS إِعادة التَّوزيع.
  4. بَعد إِعادة تَشغيل VMs، الـ Hosts المُستَقبِلة تَكون مُحَمَّلة → DRS يُوازِن لاحِقاً.

5.7 Fault Tolerance — صِفر انقِطاع | Zero Downtime

FT يَأخُذ التَّوافُريّة لِمُستَوى أَعلى مِن HA. بَدَلاً مِن إِعادة تَشغيل الـ VM بَعد فَشَل (الَّذي يَعني انقِطاعاً قَصيراً)، FT يَحتَفِظ بِنُسخة حَيّة (Secondary VM) جاهِزة لِلتَّوَلّي فَوراً. النَّتيجة: لا يَلاحِظ المُستَخدِم أَيّ شَيء. السِّعر: Resources مُضاعَفة (CPU + RAM + شَبَكة FT Logging) وَ Hosts مُختَلِفة لِلـ Primary وَ Secondary.

🧠 vSphere Fault Tolerance — FT

  • Zero downtime + Zero data loss + No network connectivity loss.
  • يُنشِئ Secondary VM مُطابِقة لِـ Primary VM عَلى Host آخَر.
  • الـ Secondary تُنَفِّذ نَفس التَّعليمات بِشَكل مُتَزامِن.
  • عِند فَشَل Primary → Secondary تَتَوَلّى فَوراً، تُنشَأ Secondary جَديدة تِلقائيّاً.
  • المُتَطَلَّبات:
    • HA مُفَعَّل.
    • الـ Primary وَ Secondary على Hosts مُختَلِفَة.
    • FT Logging Network مُخَصَّصة.
  • DRS يَختار Initial Placement لَكِن لا يُهاجِر FT VMs تِلقائيّاً.

5.8 · Deep Dive Concept Cards — مَا يَحتاج المُمارِس مَعرِفَتُه

هَذِه البِطاقات تُلَخِّص كُلَّ خَدمة Cluster بِعَمَق المُمارِس — لَيس فَقَط ما هِيَ، بَل مَتى تَستَخدِمها، وَمَتى تَتَجَنَّبها، وَمَا الأَخطاء الشّائِعة.

DRS Distributed Resource Scheduler — التَّوزيع الذَّكي
🎯 الهَدَف

تَوزيع VMs تِلقائيّاً عَلى Hosts لِتَحقيق التَّوازُن في الـ CPU وَالـ RAM، وَتَقليل Contention، وَتَحسين الأَداء.

📥 المُدخَلات
  • VM resource demand (CPU، RAM).
  • Host capacity (مُتاح، مُحتَجَز).
  • VM-Host Affinity Rules.
  • VM-VM Anti-Affinity Rules.
  • Resource Pools (Shares، Reservations، Limits).
⚙ كَيف يَعمَل (Internal Algorithm)
  1. DRS يَجمَع statistics كُلَّ 5 دَقائق.
  2. يَحسُب Cluster Imbalance Metric.
  3. إذا تَجاوَز Threshold، يَقتَرِح Migrations.
  4. بِناءً عَلى Automation Level: Manual (suggest only)، Partially Automated (placement فَقَط)، Fully Automated (يَنقُل بِدون مُوافَقة).
  5. كُلُّ vMotion يَستَهلِك Resources، لِذَلِك DRS يَوازِن بَين Migration Cost وَ Benefit.
📤 المَخرَجات
  • Initial Placement أَين تَبدَأ VM جَديدة.
  • Load Balancing نَقل VMs لِتَوازُن Cluster.
  • DRS Score 0-100% — كَم Cluster مُتَوازِن.
⚠ أَخطاء شائِعة
  • Manual Mode في Production: يَعني VMs غَير مُتَوازِنة دائِماً. اِستَخدِم Fully Automated إِلّا إذا كانَ هُناك سَبَب قَوي.
  • Migration Threshold Conservative جِدّاً: DRS لا يَنقُل شَيء، Cluster يَظَلّ غَير مُتَوازِن. الافتِراضي (Level 3) عادَة جَيِّد.
  • Affinity Rules مُتَناقِضة: «VM-A يَجِب أن يَكون مَع VM-B» + «VM-A يَجِب أن يَكون عَلى Host-1». DRS يُحاوِل، يَفشَل، يُؤَدّي إلى alerts.
  • Resource Pools كَـ Folders: Resource Pool لَيس مُجَلَّد. اِستَخدِم Folders لِلتَّنظيم، Resource Pools لِلتَّحَكُّم في Resources.
💡 مَثال حَقيقي

Cluster مِن 8 hosts. Host-3 مُحَمَّل بِنِسبَة 85% بَينَما Host-7 بِنِسبَة 35%. DRS يُلاحِظ:

  1. يَحسُب: نَقل VM-X (4 vCPU, 16 GB) مِن Host-3 إلى Host-7 يُحَسِّن DRS Score مِن 65 إلى 87.
  2. Cost: vMotion يَستَغرِق ~30 ثانية، يَستَهلِك ~2 Gbps في الـ vMotion Network.
  3. Benefit: VM-X كانَت تَعاني CPU Ready 8%، سَتَنخَفِض إلى 2%.
  4. القَرار: نَفِّذ — Benefit > Cost.
HA vSphere HA — High Availability
🎯 الهَدَف

إِعادة تَشغيل VMs تِلقائيّاً عَلى Hosts أُخرى عِند فَشَل Host. الـ HA لا يَمنَع الانقِطاع — يُقَلِّل مُدَّتَه. VMs سَتُعيد التَّشغيل = downtime 1-5 دَقائق.

📥 المُدخَلات
  • Cluster Hosts (الأَقَلّ مِنها 2).
  • Shared Storage (أَو vSAN).
  • Management Network (لِـ Heartbeat).
  • Admission Control Policy.
⚙ كَيف يَعمَل (FDM Agents)
  1. كُلُّ host لَدَيه FDM agent. أَحَدُهم Master، البَقيّة Slaves.
  2. Master يُراقِب Slaves عَبر Network Heartbeat (كُلَّ 1 ثانية).
  3. إذا فَقَدَ Heartbeat لِـ 12-15 ثانية: يُجَرِّب Datastore Heartbeat كَـ Backup.
  4. إذا فَقَدَ كِلَيهما: يَعتَبِر Host مَيِّتاً.
  5. VMs الَّتي كانَت عَلى Host المَيِّت تُعاد تَشغيلها عَلى Hosts أُخرى.
  6. إذا Master يَفشَل: Election Process بَين Slaves لِاختِيار Master جَديد.
📤 الخَدَمات المُتاحة
  • VM Restart أَلسَبَب الأَشهَر.
  • VM Monitoring يَكشِف VM hang، يُعيد تَشغيل (VMware Tools heartbeat).
  • Application Monitoring اِختِياري — يَكشِف App hang.
  • Datastore Monitoring Hosts تَفقِد Storage تُعتَبَر isolated.
  • Proactive HA (مَع Hardware Vendor): قَبل أن يَفشَل Host، اِنقُل VMs.
⚠ أَخطاء شائِعة
  • Single Management Network: إذا فَشَل، كُلُّ Cluster يَبدو down. اِستَخدِم 2 VMK Management عَلى shared switches مُختَلِفة.
  • Admission Control = None: «اِسمَح بِكُلِّ شَيء». ثُمَّ Host يَفشَل، VMs لا تَستَطيع البَدء لِأَنَّ لا سَعة. كارِثَة.
  • VM Reservations مُبالَغ فيها: يَأكُل سَعة HA. اِحفَظها لِـ Critical فَقَط.
  • HA Isolation Response = Power Off: Aggressive جِدّاً. اِستَخدِم Shutdown أَو Leave Powered On.
💡 مَثال حَقيقي

Cluster مِن 6 hosts، 200 VMs. Host #4 يَفشَل بِسَبَب PSU failure:

  1. T+0s: Host #4 يَتَوَقَّف.
  2. T+15s: Master يَفقِد Heartbeat.
  3. T+30s: Datastore Heartbeat يَفشَل أَيضاً → Host مَيِّت.
  4. T+45s: VMs Restart يَبدَأ عَلى Hosts 1-3, 5-6.
  5. T+2min: 35 VM (الَّتي كانَت عَلى Host #4) كُلُّها تَعمَل.
  6. T+5min: Aria Operations يُؤَكِّد كُلُّها reachable.
  7. إِجمالي downtime: 2-5 دَقائق لِكُلِّ VM.
FT Fault Tolerance — Zero Downtime
🎯 الهَدَف

VM تَستَمِرّ في العَمَل بِدون انقِطاع حَتّى لَو فَشَل Host بِالكامِل. صِفر downtime، عَكس HA الَّذي يُعيد التَّشغيل.

⚙ كَيف يَعمَل
  1. VM Primary على Host A.
  2. VM Secondary (نُسخَة مُتَطابِقة) عَلى Host B.
  3. كُلُّ instruction CPU، كُلُّ I/O يُكَرَّر بِالتَّوازي عَلى Secondary.
  4. Primary وَ Secondary في حالة Lockstep.
  5. إذا فَشَل Host A: Secondary يُصبِح Primary تَلقائيّاً.
  6. FT يَبدَأ Secondary جَديد عَلى Host آخَر فَوراً.
⚠ القُيود
  • VM ≤ 8 vCPU فَقَط. (تَوسيع لِأَكثَر في إِصدارات حَديثة).
  • VM ≤ 128 GB RAM.
  • يَستَهلِك ضِعف الـ Resources (نُسخَتان مِن كُلِّ شَيء).
  • FT Logging Network: مُنفَصِل، 10 GbE الأَقَلّ، Latency < 1ms.
  • لا Snapshots عَلى FT VM.
  • لا Hot-Add (CPU، Memory، Disk).
💡 مَتى تَستَخدِم FT؟
  • ✅ VM لا يَتَحَمَّل 1 دَقيقة انقِطاع: SCADA control، Telecom signaling.
  • ✅ VM Legacy لا تَدعَم Clustering عَلى مُستَوى التَّطبيق.
  • ❌ Web servers: اِستَخدِم Load Balancing بَدَلاً.
  • ❌ Databases: اِستَخدِم Database Native Clustering (Always-On، Oracle RAC).
  • ❌ VMs تَتَطَلَّب أَكثَر مِن 8 vCPU.

القاعِدة: 95% مِن VMs لا تَحتاج FT. HA كافٍ. اِستَخدِم FT بِحَذَر — التَّكلِفة عالية.

vMot vMotion — Live VM Migration
🎯 الهَدَف

نَقل VM عامِلة بَين Hosts بِدون إِيقافها — لا انقِطاع لِلخِدمة. الأَساس لِـ DRS، HA، Maintenance Mode، Patching.

⚙ كَيف يَعمَل
  1. VM يُختار لِلنَّقل مِن Host A إلى Host B.
  2. Pre-copy memory مِن A إلى B عَبر vMotion Network.
  3. أَثناء النَّقل، VM يَستَمِرّ في الكِتابَة في الذّاكِرة — تُتَتَبَّع كَـ dirty pages.
  4. عِدّة جَولات pre-copy حَتّى يَقِلّ عَدَد dirty pages.
  5. Stun: VM يَتَوَقَّف لِجُزء مِن الثّانية (typically < 1s).
  6. تُنقَل آخِر dirty pages.
  7. VM يَستَأنِف عَلى Host B.
  8. Host A يَتَخَلَّى عَن Resources الـ VM.
📋 المُتَطَلَّبات
  • Shared Storage بَين Source وَ Destination (أَو vSphere Storage vMotion).
  • vMotion VMkernel على كُلِّ Host (شَبَكة مُنفَصِلة، 10 GbE+).
  • CPU Compatibility (نَفس Vendor أَو EVC enabled).
  • VM portgroup مُتاح على Destination Host.
⚠ أَخطاء شائِعة
  • vMotion Network مُشتَرَك مَع Management: Saturation تُؤَدّي إلى HA isolation. اِفصِلهُما.
  • MTU mismatch: Switches مَع MTU 1500، vMK MTU 9000 — Migrations تَفشَل.
  • Wrong NIC Teaming: Active-Active بِدون LACP يُؤَدّي إلى packet loss.
  • EVC غَير مُفَعَّل: CPU mismatch يَمنَع vMotion. اِفعَل EVC عِند بِناء Cluster.

6.1 vLCM — لِماذا وَكَيف | Why & How

إِدارة دَورة حَياة Software الـ Hypervisor كانَت تاريخيّاً مَأساة: Patches مُتَفَرِّقة، Drivers مِن Vendors مُختَلِفين، VIBs يَدَويّة. vLCM حَلَّ ذَلِك بِنَموذج Image-based: Image واحِدة مُعَرَّفة، تُطَبَّق عَلى كُلّ Hosts في Cluster، وَكُلّ Host يَكون compliant أَو لا. لا حالات وَسَط.

إِدارة دَورة الحَياة في VCF 9.0

كَوْنك VCF Administrator، أَنت مَسؤول عَن إِبقاء كُلّ شَيء حَديثاً: vCenter، ESX، الـ VMware Tools، وَ Virtual Hardware. vSphere Lifecycle Manager (vLCM) هُوَ الأَداة المَركَزيّة لِذَلِك — وَفي vSphere 9.0، أَصبَحَت الأُسلوب الإِلزامي (Baselines القَديمة لَم تَعُد مَدعومة).

🧠 ما يَفعَله vLCM | What vLCM Does

  • إِدارة VMware Tools وَ VM Hardware upgrades.
  • Upgrade وَتَصحيح ESX hosts.
  • تَثبيت وَتَحديث Third-party software على ESX.
  • تَثبيت وَتَحديث ESX Drivers وَ Firmware.
  • تَوحيد ESX Images عَبر كُلّ Hosts في Cluster.

6.2 مُكَوِّنات الصُّورة | Image Components

الصُّورة تَتَكَوَّن مِن طَبَقات: Base Image هُوَ ESX الأَساسي مِن VMware (إِلزامي). Components هِيَ تَجميعات VIBs لِوَظائِف مُحَدَّدة. Vendor Add-ons هِيَ تَخصيصات OEM (Dell EMC، HPE، Lenovo). Firmware/Drivers Add-on يَتَطَلَّب Hardware Support Manager Plug-in وَيَجلِب التَّحديث الكامِل لِلـ hardware.

🧠 ESX Image Components — مُكَوِّنات الصُّورة

  • ESX Base Image — الإِصدار الأَساسي مِن VMware (إِلزامي، يُختار أَوَّلاً).
  • Components — تَجميع لِـ VIBs (vSphere Installation Bundles).
  • Vendor Add-ons — مَجموعات OEM (Dell، HPE، Lenovo، Cisco).
  • Firmware & Drivers Add-on — يَتَطَلَّب Hardware Support Manager (HSM) plug-in.

كُلّ Cluster يُطَبِّق Image واحِد عَلى كُلّ Hosts فيه — هَكَذا يَتَوَحَّد Software.

6.3 Heterogeneous Clusters & Image Assignment

الواقِع: مُعظَم بيئات الإِنتاج لَدَيها Hosts مِن Vendors مُتَعَدِّدين أَو أَجيال مُختَلِفة. vLCM يَدعَم 4 Images كَحَدّ أَقصى لِكُلّ Cluster، وَالـ Assignment يَتِمّ تِلقائيّاً بِناءً على بَيانات BIOS (Vendor, Model, OEM string, Family) أَو يَدَويّاً.

Heterogeneous Clusters — Cluster مُختَلِط

ماذا لَو كانَ Cluster يَحوي Hosts مِن أَجيال أَو Vendors مُختَلِفين؟

  • تَستَطيع إِرفاق حَتّى 4 Images لِنَفس الـ Cluster.
  • كُلّ Image لِمَجموعة Hosts مُحَدَّدة.
  • يَدعَم: عِدّة OEMs، أَجيال hardware مُختَلِفة، Drivers مُتَخَصِّصة.

🧠 Host Image Assignment

  • Manual — تَختار Image لِكُلّ Host بِنَفسك.
  • Automatic — يَستَخدِم بَيانات BIOS:
    • Vendor
    • Model
    • OEM string
    • Family
  • الأُسلوب التِلقائي يَضمَن أَنَّ كُلّ Host يَحصُل عَلى Image مُناسِب لَه.

6.6 Live Patch — تَحديث بِدون Reboot

أَكبَر ابتِكار في إِدارة دَورة الحَياة الحَديثة: Live Patch يُمَكِّنك مِن تَطبيق تَحديثات أَمنيّة حَرِجة بِدون دُخول Maintenance Mode، بِدون تَهجير VMs، بِدون Reboot. الفِكرة: استِخدام Mount Revisions + FSR لِإِعادة تَحميل العَمَلِيّات الجاريّة.

🧠 Live Patch — مُنذ vSphere 8U3

  • الفِكرة: تَطبيق تَحديثات Security/Urgent بِدون Maintenance Mode أَو Reboot.
  • الفَوائِد:
    • تَطبيق فَوري لِلـ Critical Patches.
    • نَوافِذ Maintenance أَقصَر بِكَثير.
    • Partial Maintenance Mode — لا تَهجير VMs أَو vSAN data.
    • تَكامُل مَع vLCM.
  • غَير مُفَعَّل افتِراضيّاً — يَجِب تَفعيله صَراحة.

🧠 Live Patch Requirements

  • vCenter ≥ 8.0 Update 3.
  • ESX ≥ 8.0 Update 3.
  • Enforce Live Patch setting مُفَعَّل في vLCM.
  • DRS مُفَعَّل وَعَلى Fully Automated.
  • الـ Patch مُوسَّم كَـ Eligible.

عَدَم تَوافُق:

  • Hosts بِـ TPM.
  • Hosts بِـ DPU.
  • VMs تَستَخدِم Direct Path I/O (يَجِب تَهجيرها أَو إِيقافها).

🧠 Live Patch Process

  1. الـ Patch يُحَمَّل في vLCM Depot.
  2. عِند التَّطبيق، يُنشَأ Mount Revision جَديد.
  3. الـ Patch يُثَبَّت عَلى الـ Mount الجَديد.
  4. Switch over مِن الـ Mount الأَصلي إِلى المُحَدَّث.
  5. الـ FSR (Fast Suspend Resume) يُعيد تَحميل vmx process لِلـ VMs الجاريّة.

FSR — Fast Suspend Resume

الـ FSR هِيَ التَّقنيّة الَّتي تَجعَل Live Patch مُمكِناً:

  • عَمَليّة غَير مُعَطِّلة — الـ VM تَستَمِرّ في العَمَل.
  • تُعيد تَحميل vmx process مِن الخَدَمات المُحَدَّثة.
  • تَستَخدَم أَيضاً عِند إِضافة/إِزالة Virtual Hardware لِـ VM شَغّالة.
  • غَير مُتَوافِقة مَع VMs تَستَخدِم Direct Path I/O.

🧠 Live Patch Extension في ESX 9.0

  • في ESX 8.x: Live Patch يُغَطّي User-space وَ vmx فَقَط.
  • في ESX 9.0: تَوَسَّع لِيَشمَل:
    • VMkernel
    • NSX components (المُدمَجة الآن في ESX Image)
  • أَكثَر Patches الآن مُؤَهَّلة لِـ Live Patch — تَقليل أَكبَر لِلصِّيانة.
  • vLCM يُبَلِّغ عَن أَيّ خَدَمات تَحتاج Restart.

🧠 Enforce Live Patch Policy

  • عِند التَّفعيل: vLCM يَسمَح فَقَط بِالـ Patches المُؤَهَّلة لِـ Live Patch.
  • أَيّ Patch يَتَطَلَّب Maintenance Mode أَو Reboot يُمنَع.
  • هَذا يُجبِر فَريق العَمَلِيّات على الالتِزام بِالنَّمط الجَديد.

6.7 Upgrade VMware Tools وَ VM Hardware

كُلّ إِصدار جَديد مِن ESX يَجلِب VMware Tools مُحَدَّثة (Bug fixes، Security patches، Drivers جَديدة) وَ VM Hardware جَديدة (vGPU، vNVMe، vSGX، vTPM). القاعِدة الذَّهَبيّة: Tools أَوَّلاً، Hardware ثانياً، وَفَقَط عِند الحاجة لِميزة جَديدة.

Tools قَبل Hardware — قاعِدة ذَهَبيّة

كُلّ إِصدار جَديد مِن ESX يَأتي مَع:

  • VMware Tools جَديدة (Bug fixes، Security، Drivers).
  • VM Hardware جَديد (مَزايا جَديدة: vGPU، vNVMe، vSGX، vTPM).

القاعِدة: دائِماً قُم بِـ Upgrade VMware Tools أَوَّلاً، ثُمَّ VM Hardware (فَقَط إِذا احتَجت ميزات جَديدة). Upgrade Hardware قَبل Tools = مُشاكِل Drivers.

Hardware Compatibility — التَّوافُق

قَبل أَيّ Upgrade أَو Deploy:

  • VMware Compatibility Guide: https://compatibilityguide.broadcom.com/
  • vSAN HCL: قائِمة مُنفَصِلة لِـ I/O وَ Network controllers + Disk drives.
  • تَأَكَّد مِن: Server Models، CPU، Storage Adapters، NICs، Disks، Firmware versions.

6.8 vCenter Reduced Downtime Upgrade — RDU

Upgrade vCenter كانَت تاريخيّاً تَعني انقِطاع طَويل. RDU غَيَّر اللُّعبة: الأُسلوب الجَديد Migration-based. يُنشَر appliance جَديد، تُنسَخ البَيانات بَينَما القَديم يَعمَل، ثُمَّ Switchover سَريع (تِلقائي أَو يَدَوي). الانقِطاع الفِعلي = دَقائِق بَدَلاً مِن ساعات.

🧠 vCenter Reduced Downtime Upgrade — RDU

  • الفِكرة: Upgrade vCenter بِأَقَلّ انقِطاع مُمكِن.
  • الأُسلوب: Migration-based — لَيس in-place.
  • الخُطوات:
    1. نَشر vCenter Appliance جَديد (الإِصدار الجَديد).
    2. نَسخ كُلّ البَيانات وَالتَّهيئة مِن القَديم لِلجَديد.
    3. أَثناء النَّسخ، vCenter القَديم يَظَلّ شَغّالاً!
    4. Switchover إِمّا تِلقائي أَو يَدَوي.
  • مَدعوم لِـ:
    • Self-managed vCenter.
    • Non-self-managed.
    • vCenter HA mode.

6.X · Deep Dive Concept Cards — مَا يَجِب أَن يَعرِفه مُهَندِس الـ Lifecycle

vLCM في VCF 9 هُوَ المُختَلَف الأَكبَر مُقارَنة بِالإِصدارات السّابِقة. اِفهَم Images، Live Patch، وَ RDU.

IMG vLCM Images vs Baselines — لِماذا تَحَوَّلنا؟
🎯 الفَرق الأَساسي
  • Baselines (Legacy، VUM): طَبَقات patches. Host = ESX base + patch 1 + patch 2 + ... — Drift مَع الوَقت.
  • Images (Modern، vLCM): Single desired state. Host = exactly this image (ESX + drivers + components). لا Drift.
⚙ مُكَوِّنات الـ Image
  1. ESX Base Image: e.g. ESX 9.0.0-12345678.
  2. Vendor Add-On: Dell، HPE، Lenovo customizations.
  3. Firmware Add-On: BIOS/Firmware updates (مَع Hardware Support Manager).
  4. Components: Drivers، VIBs مُحَدَّدة (e.g. NSX vibs).
  5. Solutions: NSX، vSAN، HCX integration.
📤 الفَوائِد
  • Drift-Free كُلُّ Host في Cluster = نَفس Image.
  • Atomic Apply يَنجَح كُلُّ شَيء أَو يَفشَل كُلُّ شَيء.
  • Firmware Integration Update ESX + BIOS في عَمَلِيّة واحِدة.
  • Predictability Image مُختَبَر في Test، يُطَبَّق على Prod حَرفيّاً.
⚠ في VCF 9

Baselines اِنتَهَت. VCF 9 يَدعَم Images فَقَط. إذا قادِم مِن vSphere 7 مَع Baselines، يَجِب التَّحَوُّل إلى Images قَبل Upgrade.

LP Live Patch — Update بِدون Reboot
🎯 الميزة الكُبرى في VCF 9

تَحديث ESX بِدون reboot أَو even Maintenance Mode. الـ Host يَستَمِرّ في تَشغيل VMs أَثناء التَّحديث. مُمكِن فَقَط لِنَوع مُحَدَّد مِن Patches (security، minor updates).

⚙ كَيف يَعمَل
  1. Patch يَحتَوي عَلى علامة "Live-Patchable".
  2. vLCM يُحَدِّد إذا Patch مُطابِق لِـ Live Patch criteria.
  3. Code Hot-Patching داخِل VMkernel مُباشَرة.
  4. VMs لا تَتَأَثَّر — لا Stun، لا Migration.
  5. Validation بَعد التَّحديث.
⚠ القُيود
  • Live Patch لا يَعمَل لِكُلِّ Update. Driver updates، Kernel changes كَبيرة تَحتاج Reboot.
  • VMware يُحَدِّد ما هُوَ Live-Patchable — لَيس قَرارَك.
  • Live Patch لا يُغَيِّر الـ "version" مِن نِظام التَّتَبُّع — يُضيف Patch ID.
  • عَدَد مَحدود مِن Live Patches قَبل Required Reboot.
💡 مَتى تَستَخدِم

Security advisory critical — تَحتاج Patch خِلال 24 ساعة. Maintenance window لَن يَتَوَفَّر إلّا بَعد أُسبوع. Live Patch = Apply الآن، Reboot لاحِقاً مَع Standard Update.

RDU Rapid Deployment Utility — Imaging السَّريع
🎯 المَفهوم

RDU يُتيح Reimage Host في دَقائق — لَيس ساعات. مُهِمّ لِـ:

  • إِضافَة Hosts كَثيرة في Cluster جَديد (Greenfield).
  • Re-imaging Host بَعد Hardware failure.
  • Standardizing Cluster بَعد drift طَويل.
⚙ Workflow
  1. Host يُسَجَّل في Cluster كَـ "RDU-Ready".
  2. Image مُحَدَّد (مِن vLCM).
  3. Trigger Re-image: Host → RDU → Apply Image.
  4. vLCM يَنشُر Image عَبر iDRAC/iLO Virtual Media أَو PXE.
  5. Host يُعيد Boot مَع Image الجَديد.
  6. Auto-configure Network، join Cluster، Apply Settings.
  7. Total Time: ~15-30 دَقيقة لِـ Host واحِد.
💡 مَثال إِنتاجي

Cluster جَديد بِـ 16 hosts. بِدون RDU: 16 × 3 ساعات (TK.1 manual) = 48 ساعة. مَع RDU: 16 × 30 دَقيقة (parallel) = 8 ساعات. وَفر 5 أَيّام عَمَل.

⚠ مُتَطَلَّبات
  • Hardware Vendor Integration (Dell OpenManage Integration، HPE OneView).
  • BMC/iDRAC reachable مِن SDDC Manager.
  • Image جاهِز في vLCM مَع Vendor Add-On.
  • Network مُعَدّ مُسبَقاً (Switches، VLANs، DHCP إذا PXE).

TK.1 · ESX Host Onboarding Checklist

كُلَّما أَضَفت Host جَديد إلى Cluster، اِتبَع هَذِه القائِمة. تَجاهُل خُطوة واحِدة يُؤَدّي إلى ساعات Troubleshooting لاحِقاً.

Runbook 1 Adding a New ESX Host to a vSphere Cluster

Pre-Installation (قَبل إِدخال Hardware)

  1. تَأكَّد مِن HCL: Hardware مَدعوم لِـ ESX 9.0 (broadcom.com/vmware/hcl).
  2. سَجِّل Serial Number, Service Tag, MAC Addresses في CMDB.
  3. اِحجِز IP Addresses: Management, vMotion, vSAN, Backup, NSX TEP (5+ IPs).
  4. اِحجِز Hostname في DNS (Forward + Reverse).
  5. تَأكَّد مِن iDRAC/iLO/IPMI: شَبَكة مُنفَصِلة، كَلِمة سِرّ قَويّة، Firmware حَديث.

BIOS / Firmware Configuration

  1. Update BIOS/Firmware إلى آخِر إِصدار مُعتَمَد مِن HCL.
  2. Enable: VT-x, VT-d (Intel) أَو AMD-V, AMD-Vi (AMD).
  3. Enable: NX/XD bit (Execute Disable).
  4. Disable: Hyperthreading فَقَط إذا كانَت Compliance تَتَطَلَّب ذَلِك (L1TF mitigation).
  5. Set: Power Profile = "OS Control" (لِيُتيح ESX إِدارة الطّاقة).
  6. Boot Mode: UEFI (لَيس Legacy BIOS).
  7. Secure Boot: Enabled (مَع TPM 2.0 إذا مُتَوَفِّر).

ESX Installation

  1. Boot مِن USB / iDRAC Virtual Media مَع VCF-Installer-9.0.x.
  2. اِختَر Boot Device: USB internal أَو M.2 (لَيس HDD/SSD يُخَصَّص لِـ VMs).
  3. Set root password (مُتَوافِق مَع Password Policy).
  4. تَأَكَّد مِن الـ Reboot الكامِل.

Post-Install Configuration

  1. Configure Management Network: IP, Subnet, Gateway, DNS, NTP.
  2. Set Hostname (FQDN). تَأكَّد عَبر esxcli system hostname get.
  3. Test DNS: nslookup hostname مِن ESX shell.
  4. Test NTP: esxcli system time get — يَجِب أن تَتَطابَق مَع vCenter.
  5. Enable SSH (مُؤَقَّتاً لِلتَّكوين، أَوقِفه لاحِقاً).
  6. Apply Lockdown Mode = "Normal" بَعد الانضِمام لِـ vCenter.

Add to vCenter

  1. اِنتَقِل إلى Cluster → Add Host.
  2. أَدخِل FQDN (لَيس IP — لِأَنَّ Certificates تَستَخدِم FQDN).
  3. اِقبَل Thumbprint بَعد التَّحَقُّق مِنه.
  4. اِنتَظِر Configure HA task — يَجِب أن يَنتَهي بِنَجاح.

Networking

  1. اِنقُل Host إلى VDS المُناسِب.
  2. Configure VMkernels: vMotion, vSAN, NSX TEP.
  3. Test كُلَّ VMkernel: vmkping -I vmk1 <ip-of-another-host>.
  4. تَأَكَّد مِن MTU (1500 لِـ Management، 9000 لِـ vMotion/vSAN).

Storage (إذا VSAN)

  1. Claim disks لِـ VSAN (Cache + Capacity).
  2. تَأَكَّد مِن انضِمام Host لِـ VSAN Cluster.
  3. Test: esxcli vsan health cluster get.

Final Validation

  1. vSphere HA: Configured & Healthy.
  2. DRS: Connected, accepting recommendations.
  3. vMotion: Test مَع VM صَغير.
  4. Update Compliance: Image-based (vLCM) — يَتطابَق مَع Cluster Image.
  5. Add tags: Environment, Rack, Owner, MaintenanceWindow.
  6. أَوقِف SSH عِند الانتِهاء.

🧠 وَقت تَقديري

إِجمالي: 3-5 ساعات لِـ Host الواحِد (مَع الـ Hardware ready). إذا تَطَلَّب أَكثَر، شَيء خاطِئ في العَمَليّة — وَثِّق وَحَسِّن.

TK.2 · Patch / Update Playbook

Runbook 2 Monthly Patching Workflow — vLCM Image-Based

T-14 days · التَّخطيط

  1. راجِع VMware Security Advisories (VMSA) لِلشَّهر.
  2. اِفحَص Release Notes — أَيُّ KB warnings؟
  3. اِفتَح Change Request في ITSM.
  4. أَخبِر فَريق التَّطبيقات بِنَوافِذ المَوافَقة.

T-7 days · الاختِبار في Dev/Test

  1. Apply Patch على Test Cluster (نُسخَة مُصَغَّرة مِن Production).
  2. اِترُك 5-7 أَيّام مُراقَبة.
  3. راقِب: Aria Operations alerts، vCenter events، أَداء VMs.

T-1 day · الاِستِعداد

  1. تَأَكَّد مِن Backup ناجِح لِـ vCenter (VCSA snapshot أَو File-Based Backup).
  2. تَأَكَّد مِن Health: cluster health، vsan health، HA admission control.
  3. تَأَكَّد مِن DRS = Fully Automated.
  4. تَأَكَّد مِن Capacity: على الأَقَلّ N+1 Host احتِياط (يُمكِنُك خَسارة host واحِد بِدون ضَغط).
  5. Communication: أَرسِل Reminder قَبل 24 ساعة.

T-0 · التَّنفيذ

  1. Pre-check: Cluster → Updates → Run pre-check.
  2. اِبدَأ Remediation. اِترُك vLCM يَختار التَّسَلسُل (Rolling).
  3. راقِب كُلَّ Host: Maintenance Mode → Update → Reboot → Exit MM.
  4. إذا Host واحِد فَشَل، أَوقِف الـ Remediation وَحَلِّل قَبل الاستِمرار.

T+1 · ما بَعد التَّنفيذ

  1. تَأَكَّد مِن Compliance: 100٪ Compliant.
  2. اِفحَص VMs الحَسّاسة (Critical applications) — هَل تَعمَل؟
  3. اِفحَص Aria Operations: لا alerts جَديدة.
  4. أَغلِق Change Request بِنَجاح.
  5. وَثِّق: ما الَّذي تَمَّ، ما المُشكِلات، ما الدُّروس.
⚠ Anti-Pattern: "Patch on Friday"

لا تُحَدِّث Production في Friday. إذا فَشَل شَيء، Team Support أَقَلّ في عُطلة الأُسبوع. الأَفضَل: Tuesday/Wednesday morning، Team كامِل مُتاح.

TK.4 · Cluster Design Worksheet

قَبل أن تَبنِي Cluster، أَجِب على هَذِه الأَسئِلة. الإِجابات تُحَدِّد كُلَّ القَرارات اللاحِقة:

Step 1: Workload Profile
  • كَم VM مُتَوَقَّع في 1 سَنَة؟ 3 سَنَوات؟
  • ما مَتَوَسِّط VM: vCPU؟ vRAM؟ Storage؟
  • ما Peak VM (الأَكبَر)؟ — يُحَدِّد الحَدّ الأَدنى لِـ Host RAM.
  • هَل هُناك Mission-Critical workloads تَتَطَلَّب FT أَو HA-Priority؟
  • هَل التَّطبيقات Latency-Sensitive؟ (تَحتاج Reservation أَو CPU Affinity؟)
Step 2: Availability Requirements
  • RTO = ? (Recovery Time Objective) → يُحَدِّد HA configuration.
  • RPO = ? (Recovery Point Objective) → يُحَدِّد Backup/Replication frequency.
  • هَل DR site مَطلوب؟ → يُحَدِّد vCenter Linked Mode، NSX Federation.
  • ما الـ Maintenance Window المُتاح؟ → يُحَدِّد Live Patch vs Standard Reboot.
Step 3: Sizing Calculation

الحِساب الأَساسي:

Hosts Required = ROUNDUP[ (Total vCPU + Total RAM + Total Storage) / 
                       (Host capacity × Target utilization) ] + N (HA reserve)

مَثَلاً: 100 VM × (2 vCPU, 4 GB) = 200 vCPU, 400 GB RAM
Host: Dell R760 (32 cores × 2 = 64 cores, 512 GB RAM)
Target utilization: 70% (لِترك مَجال نُمو وَ peaks)

Cores needed: 200 / (64 × 0.7) = 4.5 → 5 hosts
RAM needed:   400 / (512 × 0.7) = 1.1 → 2 hosts
الحَدّ الحاكِم: CPU → 5 hosts
+ N=1 (HA): 5 + 1 = 6 hosts minimum
+ Growth (20%): 7 hosts recommended
Step 4: Network Design
  • عَدَد VMNICs لِكُلِّ Host: 4 × 25 GbE (مِعياري) أَو 2 × 100 GbE (high-end).
  • Traffic Separation: Management + vMotion + vSAN + VM Traffic + NSX TEP.
  • MTU: 1500 (Mgmt) / 9000 (vMotion, vSAN, NSX).
  • LAG/LACP: مَع switches داعِمة فَقَط.

TK.5 · Troubleshooting Decision Tree

عِندَما يَشتَكي مُستَخدِم مِن «VM بَطيئة»، اِتبَع هَذا التَّسَلسُل:

DT VM Performance Troubleshooting Workflow

Layer 1: Guest OS (داخِل VM)

  1. هَل GuestOS يُظهِر High CPU/Memory؟ — Task Manager، top، htop.
  2. هَل هُناك runaway process؟
  3. هَل antivirus scan قَيد التَّشغيل؟
  4. هَل OS swap نَشِط؟ — يَعني RAM غَير كافٍ.
  5. إذا نَعَم لِأَيٍّ مِنها: المُشكِلة في GuestOS، لَيس VCF.

Layer 2: VM Configuration

  1. هَل vCPU كافٍ؟ — اِفحَص %RDY في esxtop (> 5% مُشكِلة).
  2. هَل vRAM كافٍ؟ — اِفحَص Memory Balloon، Swap (يَعني VM يَستَخدِم Swap).
  3. هَل VM يَعمَل على vNUMA صَحيح؟ — VMs > 8 vCPU، تَحَقَّق.
  4. هَل VMware Tools حَديث؟ — قَديم = drivers قَديمة.
  5. هَل هُناك Snapshots نَشِطة؟ — قَديمة = Performance ↓.

Layer 3: ESX Host

  1. هَل Host مُحَمَّل بِشِدّة؟ — vCenter → Performance → CPU Ready.
  2. هَل DRS مُتَوازِن؟ — إذا VM واحِدة على Host مَشغول، DRS يَنقُلها.
  3. هَل هُناك Memory Pressure؟ — High Ballooning, Compression, Swap.
  4. هَل Host في Maintenance Mode تَدريجيّاً؟

Layer 4: Storage

  1. اِفحَص Datastore Latency (vCenter → Performance → Datastore).
  2. Read Latency أَكبَر مِن 20ms = مُشكِلة.
  3. Write Latency أَكبَر مِن 30ms = مُشكِلة.
  4. اِفحَص vSAN Health (إذا VSAN).
  5. اِفحَص Queue Depth في HBA.

Layer 5: Network

  1. Network Latency بَين VM وَالـ Client؟
  2. Packet Drops على vNIC أَو pNIC؟
  3. NIOC saturation؟
  4. NSX DFW rule يُبطِئ شَيء؟

TK.7 · Top 10 VCF Admin Anti-Patterns

1. The Snapshot Hoarder

«سَأَترُك Snapshot كَـ Backup». Snapshots لَيست backups — هِيَ Delta files تَستَهلِك Storage بِسُرعة وَتُبطِئ VM. قاعِدة: أَقصى عُمر snapshot = 24 ساعة، أَقصى عَدَد = 2.

2. The Reservation Junkie

«سأَحجِز كُلَّ الـ RAM لِكُلِّ VM لِضَمان الأَداء». Reservations تَقتُل DRS وَتَمنَع overcommit. اِستَخدِمها فَقَط لِـ Critical VMs، وَ توثيق السَّبَب.

3. The Mega-VM Builder

«اِبنِ لِي VM بِـ 32 vCPU». الـ Wider VM، الأَصعَب على DRS تَوزيعُها، وَالأَعلى احتِمال CPU Ready. ابدَأ صَغير، اِنمُ تَدريجيّاً.

4. The "I'll Document Later"

«مَعروف، سأكتُبه لاحِقاً». لا. وَثِّق فَوراً عَبر Tags، Annotations، Wiki. الأَسوَأ: مِعماري يَترُك الشَّركة بِدون وَثائِق.

5. The Lone Wolf Admin

«لا أَحتاج Change Process، أَعرِف ما أَفعَل». حَتّى الخُبَراء يُخطِئون. كُلُّ تَغيير في Production يَمُرّ بِـ Change Approval، حَتّى لَو بَدا تافِهاً.

6. The Patch Avoider

«ESX يَعمَل، لِماذا أُحَدِّث؟». لِأَنَّ Security Patches تُغلَق Vulnerabilities. لَم تُحَدِّث = بَيئَتك مَكشوفة.

7. The Single Datastore Hero

«كُلُّ شَيء على Datastore واحِد كَبير». = Single Point of Failure. اِفصِل: OS، Data، Logs، Swap على Datastores مُختَلِفة.

8. The Permissions Sprawler

«أَعطِيه Admin، أَسهَل». لا. اِستَخدِم Role-Based Access. كُلُّ مُستَخدِم يَحصُل عَلى أَقَلّ صَلاحيّة يَحتاجها (Least Privilege).

9. The Monitoring Skipper

«vCenter يُريني كُلَّ شَيء». لا. اِستَخدِم Aria Operations لِلـ Trends، Skyline لِلـ Proactive Health، Log Insight لِلـ Logs. كُلٌّ مِنها يَكشِف ما الآخَر يَخفي.

10. The Backup Trust Faller

«Backup يَعمَل، لا حاجَة لِلاختِبار». لا. Test Restore شَهرِيّاً عَلى الأَقَلّ. Backup لَم يُختَبَر = Backup لَم يَنجَح.

TK.8 · VCF Admin Interview Prep — Real Questions

Technical (Concepts)

  1. اِشرَح الفَرق بَين HA وَ FT وَ DRS. مَتى تَستَخدِم كُلَّ واحِد؟
  2. ما الَّذي يَحدُث تَقَنيّاً عِند تَنفيذ vMotion؟
  3. اِشرَح SPBM — Storage Policy-Based Management.
  4. ما الفَرق بَين vDS وَ vSS؟ مَتى تَختار كُلّاً مِنهُما؟
  5. اِشرَح vLCM Image vs vLCM Baseline.
  6. كَيف يَعمَل HA Admission Control؟ ما خِياراتُه؟

Operational (Real Work)

  1. VM يُعطي High CPU Ready. كَيف تُحَلِّل؟
  2. Cluster بَدَأ يَتَلَقّى تَنبيهات «Datastore Latency Above 30ms». ما الخُطُوات؟
  3. Host لا يَخرُج مِن Maintenance Mode. ما الأَسباب المُحتَمَلة؟
  4. VM Boot loop بَعد Snapshot Revert. ما العَمَل؟
  5. vCenter لا يَتَّصِل. كَيف تَستَعيد؟

Architectural (Design Thinking)

  1. عَميل عِندَه 500 VM وَيُريد VCF — كَيف تُصَمِّم Cluster Topology؟
  2. كَيف تَختار بَين vSAN OSA vs ESA؟
  3. عَميل يُريد Stretched Cluster بَين Datacenters على بُعد 50 كم — ما المُتَطَلَّبات؟
  4. كَيف تَخطُط لِـ DR Site؟ ما الـ RPO/RTO المُمكِنة؟

Behavioral

  1. اِخبِرني عَن أَكبَر Outage تَعامَلت مَعَه — ماذا تَعَلَّمت؟
  2. اِخبِرني عَن خِلاف مَع فَريق التَّطبيقات — كَيف حَلَلتَه؟
  3. كَيف تُقَرِّر بَين إِيقاف خِدمة لِـ Patch vs الانتِظار؟
  4. كَيف تَبقى مُحَدَّثاً مَع VCF roadmap؟

🧠 Interview Wisdom

أَفضَل إِجابة لِأَيّ سُؤال تِقَني هِيَ: «دَعنا نَعتَبِر السِّياق… [اِفتَرِض جَيِّداً] … ثُمَّ أَقتَرِح [حَلّ] لِأَنَّ [سَبَب] مَع التَّحَفُّظ على [مَخاطِر]». الـ Junior يُجيب بِلا سِياق. الـ Senior يَسأَل قَبل أن يُجيب.

اِتبَع TK.1 Onboarding Checklist لِكُلِّ Host. Hosts الأَربَعة في يَوم واحِد إذا كانَ كُلُّ شَيء جاهِزاً.

Test Plan
  • vMotion اِنقُل test VM بَين كُلِّ pair مِن hosts (6 tests for 4 hosts).
  • HA أَوقِف Host فَجأة — تَأَكَّد VMs تَنتَعِش.
  • DRS اِسبِق Host واحِد، راقِب DRS rebalancing.
  • Storage بَنشمارك Latency تَحت حِمل.

القَرار المِعماري المِفتاحي

المُشكِلة: هَل نَستَخدِم Enhanced Linked Mode (ELM) مَع vCenter في HQ؟

القَرار: نَعَم، لِلأَسباب التّالية:

  • إِدارة مُوَحَّدة — مَنظَر واحِد لِجَميع البَيئات.
  • Roles وَ Permissions مَركَزيّة.
  • Cross-vCenter vMotion مُمكِن لِـ DR.

الـ Trade-off: Network dependency بَين الفَرع وَ HQ يَجِب أن تَكون stable.

الـ Mitigation: Each vCenter مُستَقِل في حالات الانقِطاع — VMs تَستَمِرّ.

الدُّروس المُستَفادة

  • التَّخطيط = 60٪ مِن النَّجاح. Week 1-2 يُنقِذ Week 5-9.
  • اِبدَأ صَغير، اِنمُ تَدريجيّاً. 4 hosts كافٍ لِلبَدء.
  • Reference Architectures مَوجودة لِسَبَب. لا تُعيد اختِراع العَجَلة.
  • Test كُلَّ شَيء قَبل Production. أَوَّل VM في Production لَيست أَوَّل اختِبار.

SC.2 · Major Upgrade Campaign

«Upgrade مِن vSphere 7 إلى VCF 9 خِلال 6 أَشهُر»
Scenario 2 Enterprise Upgrade Campaign — 30 Hosts, 500 VMs, 4 Clusters

الخَلفيّة: شَركة تَأمين كَبيرة، 4 Clusters (Production, DR, Test, Dev) عَلى vSphere 7.0 U3. Broadcom أَعلَنَت end-of-support لِـ vSphere 7 خِلال 12 شَهراً. الإِدارة طَلَبَت Upgrade إلى VCF 9.0 خِلال 6 أَشهُر.

التَّحَدّيات

⚠ القُيود
  1. Hardware Compatibility: 8 hosts قَديمة مِن 30 لا تَدعَم VCF 9.
  2. Licensing: التَّحَوُّل مِن Perpetual إلى Subscription — تَفاوُض مَع Broadcom.
  3. Downtime: Production لا يَتَحَمَّل أَكثَر مِن 2 ساعات في الشَّهر.
  4. Skills: الفَريق لَم يَعمَل مَع vLCM Images.
  5. Backup Vendor: يَحتاج certifying لِـ VCF 9 — قَيد التَّحَقُّق.

الاِستِراتيجيّة — Wave-Based Approach

القَرار: التَّحديث على 4 أَمواج

بَدَلاً مِن Upgrade كُلِّ شَيء مَرَّة، نُنَفِّذ على 4 موجات، تَتَعَلَّم كُلُّ موجة مِن السّابِقة.

Month 1: Wave 1 — Dev Cluster

  1. Replace 2 hosts قَديمة بِجَديدة مَدعومة.
  2. Backup vCenter (VCSA snapshot + File-Based Backup).
  3. Upgrade vCenter أَوَّلاً (lower version)، vCenter دائِماً أَوَّل.
  4. Build vLCM Image (ESX 9 + Driver Bundle + Vendor Add-ons).
  5. Apply Image إلى Cluster.
  6. Rolling host upgrade — يَأخُذ كُلّ host ~90 دَقيقة.

  8. Documentation: ما المُشكِلات، الـ Workarounds، الـ Timing الفِعلي.

Month 2-3: Wave 2 — Test Cluster

تَطبيق الدُّروس مِن Wave 1. اِختِبار:

  • NSX compatibility (إذا مُستَخدَم).
  • Backup vendor (Veeam أَو CommVault) certification on VCF 9.
  • Monitoring tools (Aria Operations) compatibility.
  • VM Tools auto-upgrade.

Month 4: Wave 3 — DR Site

⚠ نُقطَة حَرِجة

DR Site تَجِب أن تَكون عَلى نَفس Version أَو أَحدَث مِن Production. لِذَلِك DR أَوَّلاً، Production ثانياً. إذا حَدَث Disaster خِلال الـ Window، DR تَستَطيع اِستِقبال Production 7 (older version) عَلى VCF 9 — backward compatibility مَدعومة.

  1. Production Upgrade Plan مَع كُلِّ Application Team — تَواريخ مُعتَمَدة.
  2. Each weekend: 2 hosts (rolling).
  3. كُلُّ host: vMotion VMs out → Maintenance Mode → Upgrade → Reboot → Exit MM → VMs back.
  4. اِفحَص بَين كُلِّ host: alerts، performance.
  5. إذا أَيُّ مُشكِلة، اِستَخدِم vLCM Live Patch (إذا مُتاح) أَو Rollback.

القَرار المِعماري المِفتاحي

المُشكِلة: هَل نُحَدِّث vCenter أَوَّلاً مَرَّة واحِدة، أَم نَحتَفِظ بِـ Multiple vCenters لِكُلِّ Wave؟

القَرار: vCenter واحِد، Upgrade مَرَّة واحِدة في الأَوَّل.

السَّبَب: vCenter 9 يُدير ESX 7 وَ ESX 9 في نَفس البَيئة (N-2 compatibility). هَذا يُتيح Migration تَدريجي.

الـ Trade-off: إذا فَشَل Upgrade vCenter، كُلُّ شَيء يَتَوَقَّف. لِذَلِك Backup كامِل قَبل، وَ Maintenance Window 4 ساعات لِـ Rollback إذا لَزِم.

الدُّروس المُستَفادة

  • Hardware Refresh دائِماً قَبل Software Upgrade. لا تَكُن مُجبَراً عَلى Replace في مُنتَصَف الـ Project.
  • Wave-Based ينَقذ. Wave 1 كَشَفَت مَشاكِل سَتَكون كارِثة في Wave 4.
  • Communication مَع App Teams مُبَكِّر. 6 أَشهُر يَعني خِطّة، لَيس Surprise.
  • Rollback Plan ضَروري لِكُلِّ Step. «نَأمَل أَن يَعمَل» لَيست خِطَّة.

SC.3 · Network Migration to VDS

«اِنتَهَينا مِن vSS — كُلُّ شَيء على VDS»
Scenario 3 Standard Switch → Distributed Switch Migration (Live)

الخَلفيّة: Cluster قَديم اِستَخدَم vSphere Standard Switches (vSS) لِسَنَوات. الـ Architect الجَديد طَلَب التَّحَوُّل إلى VDS لِـ Consistent Configuration، NIOC، وَ NSX Readiness. التَّحَدّي: لا Downtime مَسموح بِه.

لِماذا VDS؟

الميزةvSSVDS

Migration Strategy — Per-Host Approach

Per-Host Migration Steps
  1. Build VDS Empty: Create VDS مَع نَفس Port Groups كَـ vSS (نَفس VLAN IDs، نَفس MTU).
  2. Add Host to VDS: Cluster → VDS → Add Host. لا تَنقُل VMNICs بَعد!
  3. Migrate Unused VMNIC First: إذا host لَدَيه 4 VMNICs (2 used على vSS، 2 idle)، اِنقُل 1 idle VMNIC إلى VDS.
  4. Migrate VMkernels One-by-One: vMotion VMK أَوَّلاً (less risk). تَأَكَّد بَين كُلِّ خُطوة: vmkping يَعمَل.
  5. Migrate VMs Port Groups: Network Migration Wizard — اِختَر VMs، اِختَر VDS Port Group destination. هَذا live، بِدون VM downtime.
  6. Migrate Remaining VMNIC: عِندَما لا VM على vSS، اِنقُل آخِر VMNIC.
  7. Cleanup: اِحذِف vSS مِن Host.
  8. Repeat لِكُلِّ Host في Cluster.

الخَطَأ الَّذي يَجِب تَجَنُّبه

⚠ Anti-Pattern: «اِنقُل كُلَّ VMNICs مَرَّة واحِدة»

إذا اِنقَلَبَ شَيء، تَفقِد كامِل Host network. الـ Host يُصبِح orphaned — vCenter لا يَراه، لا تَستَطيع Login عَبر vSphere Client. الحَلّ الوَحيد: DCUI أَو KVM. دائِماً اِنقُل VMNIC واحِدة في كُلِّ مَرَّة، اِترِك واحِدة على vSS كَـ failover.

Validation بَعد كُلِّ Host

  • VMK Ping Test — كُلُّ vmkping بَين Hosts يَنجَح.
  • VM Connectivity — اِختَبِر VM واحِدة على كُلِّ Port Group.
  • vMotion Test — اِنقُل VM إلى/مِن Host المَنقول.
  • HA Heartbeat — تَأَكَّد HA Status = Connected.
  • Health Check — اِفعَل VDS Health Check (MTU + VLAN) لِـ 10 دَقائق.

Time Estimation

  • Per Host: 30-45 دَقيقة (مَع Validation).
  • Cluster 10 Hosts: ~6-8 ساعات (يَوم كامِل، مَع Buffer).
  • اِخطُط Window مِن 12 ساعة لِلسَّلامة.

القَرار المِعماري المِفتاحي

المُشكِلة: كَم VDS نَحتاج؟ واحِد لِكُلِّ Cluster، أَم واحِد كَبير لِجَميع Clusters؟

القَرار: VDS واحِد لِكُلِّ Cluster.

السَّبَب: Blast Radius — إذا فَشَل VDS، يُؤَثِّر فَقَط على Cluster واحِد، لَيس كُلَّ البَيئة.

الاِستِثناء: Cross-vCenter NSX يَتَطَلَّب VDS مُشتَرَك — هَذا حالة خاصَّة.

SC.4 · VM Sprawl Cleanup

«لَدَينا 2000 VM، نَعرِف ماذا يَفعَل 1200 مِنها»
Scenario 4 Identifying & Right-Sizing Orphaned Workloads

الخَلفيّة: شَركة كَبيرة، 5 سَنَوات مِن نُمو VCF بِدون governance قَوي. 2000 VM، لَكِنَّ كَثيراً مِنها orphaned (مالِك غَير مَعروف)، over-provisioned، أَو ميتة. الـ CFO يَسأَل: «لِماذا التَّكلِفة تَرتَفِع كُلَّ سَنَة؟»

Discovery Phase — Identify the Problem

  1. Inventory Audit: Export كُلَّ VMs مَع Tags، Owner، Power State، Storage، Last Activity.
  2. اِفصِل VMs إلى Buckets:
Bucketالمِعيارالمُتَوَقَّع %
🔴 ZombiesPowered Off > 90 يَوم5-10%
🔴 GhostPowered On، لَكِن لا Network Traffic > 30 يَوم10-15%
🟡 Over-SizedCPU < 10% avg، RAM < 30% avg30-40%
🟡 Orphanedلا Owner Tag20%
🟢 HealthyDocumented + utilized30-40%

Right-Sizing Methodology

Right-Size Logic
  1. Aria Operations يُولِّد Right-Sizing Reports تِلقائيّاً.
  2. قاعِدة Right-Size: 95th percentile usage في آخِر 30 يَوم + 30% buffer.
  3. مَثَلاً: VM 8 vCPU، 95th percentile = 1.5 vCPU. → Right-size إلى 2 vCPU.
  4. مَثَلاً: VM 16 GB RAM، 95th percentile = 4 GB. → Right-size إلى 6 GB.
  5. التَّحقيق: اِسأَل مالِك VM قَبل التَّنفيذ. ما زال يُريد هَذِه السَّعة لِلـ Peak؟

Cleanup Campaign — 90 Days

  1. Week 1-2: Discovery Report مَع كُلِّ Application Owner. اِبعَث Email: «هَذِه VMs مَنسوبة لِك. ما زالَت مَطلوبة؟»
  2. Week 3-4: VMs بِلا response → Power Off (لا حَذف فَوري). إذا أَحَد اشتَكى خِلال 7 أَيّام، أَعِد التَّشغيل.
  3. Week 5-8: VMs powered off لِأَكثَر مِن 14 يَوم → Move to "ToBeDeleted" Folder.
  4. Week 9-10: Snapshot لِجَميع "ToBeDeleted" VMs (لِلطَّوارئ)، ثُمَّ Delete.
  5. Week 11-12: Right-Size Campaign لِـ Over-sized VMs. Schedule downtime مَع owners.

Governance Going Forward

قَواعِد جَديدة (مُعتَمَدة مِن IT Steering Committee)
  • Mandatory Tags: Owner, CostCenter, Environment, AppName عَلى كُلِّ VM جَديدة.
  • Auto-Detection: Aria Operations يُنبِّه عَلى idle VMs لِأَكثَر مِن 30 يَوم.
  • Quarterly Review: كُلُّ App Owner يُؤَكِّد VMs الخاصَّة بِه.
  • Chargeback: تَكلِفَة VM تَظهَر شَهرِيّاً لِـ Cost Centers.
  • 90-Day Rule: VM Powered Off > 90 يَوم تُحذَف تِلقائيّاً (بَعد إِشعار).

النَّتيجة الفِعليّة (مَثال)

  • 2000 VMs → 1450 VMs.
  • تَخفيض اِستِخدام Storage: 35%.
  • تَخفيض اِستِخدام CPU: 28%.
  • تَأجيل شِراء 4 hosts جَديدة → تَوفير ~$200K.
  • التَّكلِفة الإِدارية لِلـ Campaign: 3 FTE-months.
  • ROI: أَكثَر مِن 8x.

SC.5 · HA Event Post-Mortem

«Host تَعَطَّل في 03:00 صَباحاً — ماذا حَدَث؟»
Scenario 5 Root Cause Analysis — Unplanned Host Failure

الخَلفيّة: ساعة 03:14 صَباح الثُّلاثاء، PagerDuty أَيقَظ On-Call: «Host esx-prd-07 unreachable». 23 VMs كانَت على هَذا Host. HA Failed Over them خِلال 5 دَقائق. الـ CTO يُريد Post-Mortem كامِل خِلال 48 ساعة.

Phase 1: Immediate Response (الـ Run Book)

  1. 03:14 — PagerDuty alert received. On-Call eng يَتَّصِل بِـ secondary لِلتَّأكيد.
  2. 03:17 — Login إلى vCenter. تَأكيد: Host disconnected، 23 VMs تَنتَقِل (HA event).
  3. 03:20 — تَأَكَّد VMs إعادة تَشغيل: Aria Operations dashboard. كُلُّها UP.
  4. 03:25 — اِفحَص باقي Cluster: Capacity OK، No further alerts. HA reserve عاد لِـ N+1.
  5. 03:35 — Try iDRAC: لا يَستَجيب. مُحتَمَل Hardware failure.
  6. 03:40 — اِفصِل Host في vCenter (Remove from Inventory) لِمَنع HA من إِعادة المُحاوَلة.
  7. 03:50 — Open Vendor Ticket مَع Dell. اِبعَث logs مِن نَظام Monitoring.
  8. 04:00 — Update Stakeholders (Email): «Service Restored. Host hardware under investigation.»
  9. 04:15 — العَودة لِلنَّوم. التَّحقيق التَّفصيلي صَباحاً.

Phase 2: Deep Investigation (08:00 next day)

  1. iDRAC SEL Log: «Critical: CPU 1 Voltage Out of Range» في 03:13:52.
  2. Aria Operations: قَبل الفَشَل بِـ 4 ساعات، CPU 1 Temperature بَدَأ يَرتَفِع تَدريجيّاً (لَم يَتَجاوَز Threshold).
  3. Hardware History: هَذا Host مِن نَفس batch هَدَفَ ل firmware advisory في الشَّهر الماضي — لَم يُحَدَّث.
  4. Conclusion: VRM (Voltage Regulator Module) failure على motherboard. Dell سَيُرسِل replacement خِلال 4 ساعات.

  2. Wave/Stage approach. لا تُغَيِّر كُلَّ شَيء مَرَّة. اِبدَأ صَغير.

Cluster Resource Percentage.

Different hosts so a single host failure doesn't kill both.

Up to 4 images per cluster.

Correct: B

Always Tools BEFORE Hardware.

Correct: B

Migration-based reduces downtime — old vCenter stays online while data copies to new appliance.
